# MQ 使用场景：异步、削峰、解耦、广播

## 面试定位

MQ 是 Java 后端高频基础设施题。面试不会只问“为什么用 MQ”，通常会继续追问：

- 异步、削峰、解耦分别解决什么问题。
- MQ 引入后带来哪些新问题。
- 什么场景不应该用 MQ。
- 如何保证消息不丢、不重复、最终一致。
- 线上消息堆积、重复消费、消费失败怎么处理。

回答时不要只背四个词，要把 MQ 放在系统链路里讲清楚：MQ 本质上是在生产者和消费者之间加入一个可持久化、可缓冲、可重试的中间层，用同步调用的确定性换取吞吐、隔离和扩展性，同时引入最终一致、重复消费、顺序性和可观测性成本。

## 核心概念

MQ 解决的是“消息传递”和“流量缓冲”问题，不是直接替代 RPC、数据库事务或缓存。

典型角色：

- Producer：生产者，负责发送消息。
- Broker：消息服务端，负责存储、路由、复制和投递消息。
- Topic：消息主题，按业务语义分类。
- Consumer：消费者，负责拉取或接收消息并执行业务逻辑。
- Consumer Group：消费组，同组内通常负载均衡，不同组之间独立消费。

MQ 常见收益：

- 时间解耦：生产者不必等待消费者执行完成。
- 空间解耦：生产者不直接依赖消费者地址和数量。
- 流量削峰：生产速度和消费速度可以不完全一致。
- 事件分发：一个事件可以被多个系统分别处理。

MQ 常见成本：

- 一致性从强一致变成最终一致。
- 失败路径变多：发送失败、存储失败、消费失败、重试失败。
- 消费可能重复，必须做幂等。
- 可能出现消息堆积和延迟。
- 需要监控 Topic、消费组、堆积、失败率、重试和死信。

## 使用场景一：异步

异步适合把主流程中非关键、耗时、可延迟完成的动作拆出去。

下单场景中，主链路可能包括：

```text
创建订单
  -> 扣库存
  -> 支付
  -> 返回用户结果
  -> 发短信
  -> 发优惠券
  -> 更新画像
  -> 同步搜索索引
```

其中短信、优惠券、画像、索引通常不应该阻塞用户请求，可以通过 MQ 异步处理：

```text
创建订单
  -> 扣库存
  -> 支付
  -> 发送 OrderPaidEvent
  -> 返回用户结果

OrderPaidEvent
  -> 短信消费者
  -> 优惠券消费者
  -> 画像消费者
  -> 搜索索引消费者
```

异步的核心收益：

- 降低接口 RT。
- 降低主链路故障面。
- 非核心动作失败后可以重试。
- 便于新增订阅方，不侵入主流程。

但要注意：

- 用户看到成功时，异步任务可能还没完成。
- 异步任务失败需要补偿。
- 不能把强一致要求的动作随意异步化。

不适合异步的场景：

- 需要立即影响返回结果的校验，例如支付前风控强校验。
- 失败后无法补偿的核心动作。
- 需要同一个数据库事务强约束的步骤。

## 使用场景二：削峰

削峰适合生产流量短时间远大于消费能力的场景，例如秒杀、抢券、活动报名、日志采集。

没有 MQ 时：

```text
大量请求 -> 订单服务 -> 库存服务 -> 数据库
```

峰值会直接打到下游，数据库或库存服务容易被瞬时流量压垮。

引入 MQ 后：

```text
大量请求 -> 轻量校验 -> 写入 MQ -> 快速返回排队中
                         -> 消费者按能力匀速处理
```

削峰不是让系统无限承载，而是把突发流量变成队列积压，由消费者按可控速率处理。

关键设计点：

- 入口要先做限流、资格校验和库存预扣，避免无效消息冲垮 MQ。
- MQ 队列容量要有上限意识，堆积不是无限免费。
- 消费速度要和下游数据库、缓存、RPC 能力匹配。
- 堆积时间要有业务可接受范围，例如订单 30 分钟内处理完成。

削峰常见风险：

- 队列积压过久，用户体验恶化。
- 消费者扩容受 Partition 或 Queue 数限制。
- 下游一直慢，MQ 只是在延后故障。
- 消息过期或被清理导致业务丢单。

## 使用场景三：解耦

解耦适合一个业务事件要通知多个系统，但生产者不想直接依赖所有消费者。

同步调用模式：

```text
订单服务
  -> 调用积分服务
  -> 调用优惠券服务
  -> 调用风控服务
  -> 调用推荐服务
```

问题：

- 订单服务要知道所有下游。
- 下游新增、下线、变更都会影响订单服务。
- 任一下游慢或失败都可能拖慢主流程。
- 调用链变长，整体可用性下降。

事件模式：

```text
订单服务 -> OrderCreatedEvent

积分服务订阅 OrderCreatedEvent
优惠券服务订阅 OrderCreatedEvent
风控服务订阅 OrderCreatedEvent
推荐服务订阅 OrderCreatedEvent
```

订单服务只负责发布事实事件：订单已创建。谁消费、消费多少次、如何处理，由订阅方自己负责。

面试中要强调：

- 事件应该表达已经发生的业务事实，不应该表达“请某服务执行某命令”的强耦合意图。
- 消息体要保持稳定，字段新增要兼容旧消费者。
- Topic 粒度不能过粗，否则消费者都要解析无关消息；也不能过细，否则治理成本高。

## 使用场景四：广播

广播指同一条消息被多个独立消费者都处理一遍。

在 Kafka / RocketMQ 中，广播通常通过不同 Consumer Group 实现：

```text
Topic: order-event

consumer-group-a: 积分系统
consumer-group-b: 搜索系统
consumer-group-c: 数据仓库同步
```

每个消费组都有独立 offset。同一组内多个实例负载均衡；不同组之间相互独立，都会收到消息。

适合广播的场景：

- 订单事件分发给多个业务系统。
- 配置变更通知多个服务刷新本地缓存。
- 用户行为事件同时进入实时推荐、画像、数仓。
- 审计、监控、告警系统旁路消费业务事件。

广播要注意：

- 不同订阅方的处理速度互不影响，但同一个 Broker 资源会共享。
- 一个消费者组堆积不会影响另一个消费者组的 offset，但会占用服务端存储保留时间。
- RocketMQ 还有广播消费模式，但失败消息通常不会像集群消费那样进入服务端重试，业务上要谨慎。

## MQ 与 RPC 的区别

| 维度 | RPC | MQ |
| --- | --- | --- |
| 调用关系 | 点对点直接调用 | 生产者和消费者通过 Broker 间接交互 |
| 时间关系 | 通常同步等待结果 | 通常异步最终完成 |
| 一致性 | 更适合强依赖结果 | 更适合最终一致 |
| 可用性 | 下游失败直接影响调用方 | 下游失败可重试、可堆积 |
| 吞吐 | 受下游实时能力限制 | 可缓冲削峰 |
| 复杂度 | 链路直观 | 需要处理重复、丢失、堆积和补偿 |

判断标准：

- 需要立即拿结果，用 RPC。
- 只需要通知事件，用 MQ。
- 需要强事务一致，不要只靠 MQ。
- 需要削峰和重试，MQ 更合适。

## 代码示例

### Spring Kafka 发送业务事件

```java
@Service
public class OrderEventPublisher {
    private final KafkaTemplate<String, OrderPaidEvent> kafkaTemplate;

    public OrderEventPublisher(KafkaTemplate<String, OrderPaidEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void publish(OrderPaidEvent event) {
        kafkaTemplate.send("order-paid-topic", event.getOrderId(), event)
                .whenComplete((result, ex) -> {
                    if (ex != null) {
                        // 生产环境应记录失败并触发告警或落库补偿
                        throw new IllegalStateException("send order paid event failed", ex);
                    }
                });
    }
}
```

### RocketMQ 发送业务事件

```java
@Service
public class OrderEventPublisher {
    private final RocketMQTemplate rocketMQTemplate;

    public OrderEventPublisher(RocketMQTemplate rocketMQTemplate) {
        this.rocketMQTemplate = rocketMQTemplate;
    }

    public void publish(OrderPaidEvent event) {
        rocketMQTemplate.syncSend("order-paid-topic", event);
    }
}
```

### 消费者处理异步任务

```java
@KafkaListener(topics = "order-paid-topic", groupId = "coupon-service")
public void onMessage(OrderPaidEvent event) {
    couponService.sendCoupon(event.getUserId(), event.getOrderId());
}
```

真实工程中消费者必须补上幂等、异常处理、监控和死信处理，不能只写业务调用。

## 线上风险

### 最终一致风险

主流程成功不代表异步动作成功。比如订单支付成功后，发券消费者失败，用户短时间内看不到优惠券。

治理方式：

- 消费失败自动重试。
- 超过次数进入死信队列。
- 提供补偿任务按业务状态重新扫描。
- 用户侧展示“处理中”或可重试状态。

### 重复消费风险

MQ 通常只能保证至少一次投递。消费者可能因为重试、超时、重平衡、提交 offset 失败而重复处理。

治理方式：

- 消费逻辑幂等。
- 使用业务唯一键，例如 `orderId + eventType`。
- 数据库唯一索引兜底。
- 状态机只允许合法状态流转。

### 消息丢失风险

可能发生在：

- 生产者发送失败但没有检查结果。
- Broker 未刷盘宕机。
- 消费者先提交 offset 后执行业务，业务失败后消息被跳过。
- 消息过期被清理。

治理方式：

- 生产者同步发送或正确处理异步回调。
- 关键 Topic 使用可靠 ack 和副本策略。
- 消费成功后再提交 offset。
- 业务侧保留补偿表或 outbox。

### 消息堆积风险

消费者慢、下游慢、消息量突增、分区热点都会导致堆积。

治理方式：

- 监控 consumer lag。
- 增加消费者实例，但不能超过分区或队列并行度上限。
- 优化消费逻辑，批量处理，降低下游 RT。
- 对热点 key 拆分。
- 必要时临时扩容 Topic 分区或 Queue。

## 排查优化

排查 MQ 问题时按链路分层：

```text
生产端
  -> 是否发送成功
  -> 是否有重试、超时、限流
  -> 消息 key、topic、tag 是否正确

Broker
  -> Topic 是否存在
  -> 分区/队列是否均衡
  -> 磁盘、网络、复制、刷盘是否异常
  -> 消息是否过期

消费端
  -> 消费组是否正确
  -> offset 是否提交
  -> 是否持续异常重试
  -> 下游依赖是否慢
  -> 是否发生重平衡
```

优化优先级：

- 先确认消息是否真的发送到 Broker。
- 再确认消息是否被正确消费组看到。
- 再看消费者处理是否报错或变慢。
- 最后再考虑扩容和参数调整。

不要一上来就盲目加消费者。如果分区只有 4 个，加到 20 个实例也只有 4 个实例能并行消费。

## 常见追问

### MQ 为什么能削峰？

因为 MQ 把生产速率和消费速率解耦。请求高峰时消息先进入 Broker 持久化队列，消费者按照自身和下游能承受的速度处理。但 MQ 不是无限容量，长期堆积仍然会转化为延迟、磁盘和过期风险。

### MQ 会不会让系统更稳定？

不一定。MQ 可以隔离下游故障和削峰，但会引入重复消费、消息丢失、最终一致和堆积。只有配套幂等、重试、死信、补偿和监控，系统稳定性才会提升。

### 什么时候不用 MQ？

- 调用方必须立即知道处理结果。
- 步骤之间需要同一个本地事务强一致。
- 消息量很小且链路简单，引入 MQ 反而增加复杂度。
- 消费失败无法补偿。
- 业务不能接受最终一致延迟。

### MQ 异步后怎么保证业务一致？

常见方案是本地事务 + outbox。业务数据和待发送消息写入同一个数据库事务，后台任务或 CDC 再可靠投递到 MQ。消费者侧再通过幂等表或业务唯一约束保证重复消息不会产生副作用。

## 易错点

- 认为 MQ 可以保证消息只消费一次。
- 认为发送成功就代表业务最终一定成功。
- 只做生产端可靠，不做消费端幂等。
- 用 MQ 包装强依赖 RPC，结果只是把失败延后。
- Topic 设计过粗，所有消费者解析大而全消息。
- 消费失败只打日志不重试、不告警。
- 堆积时盲目扩实例，忽略分区数、热点 key 和下游瓶颈。

## 自检清单

- 是否能区分异步、削峰、解耦、广播各自解决的问题。
- 是否能说清 MQ 引入的副作用。
- 是否知道 MQ 和 RPC 的选择标准。
- 是否能举出订单、秒杀、日志、缓存刷新等真实场景。
- 是否能说明最终一致如何补偿。
- 是否能说明为什么消费者必须幂等。
- 是否能从生产端、Broker、消费端三层排查 MQ 问题。
