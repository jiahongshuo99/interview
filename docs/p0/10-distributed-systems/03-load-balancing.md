# 负载均衡策略

## 面试定位

负载均衡是服务调用治理的核心能力。面试官通常希望你能从“怎么选一个实例”扩展到“如何在动态实例、不同权重、故障节点、热点请求、机房路由和重试场景下稳定分配流量”。

常见考点：

- 轮询、随机、权重轮询、最少连接、一致性哈希的原理和适用场景。
- 负载均衡在 RPC 调用链路中的位置。
- 负载均衡如何结合服务发现、熔断、重试和灰度。
- 为什么简单轮询也可能打爆某个实例。
- 如何处理慢实例、故障实例和热点 key。

回答时不要只背策略名称，要结合业务场景说取舍。

## 负载均衡位置

典型客户端侧负载均衡：

```text
Consumer 发起 RPC
  -> 获取本地服务实例列表
  -> 路由过滤：版本、灰度、机房、标签
  -> 剔除不可用实例：熔断、连接失败、健康状态
  -> 负载均衡选择一个实例
  -> 发起调用
  -> 失败后按策略决定是否重试并重新选择实例
```

服务端侧负载均衡则由网关、Nginx、LVS、云负载均衡等组件负责选择后端实例。

## 核心指标

评价负载均衡不能只看请求数是否平均，还要看：

- 实例 CPU、内存、线程池、连接数。
- 实例处理能力差异。
- 请求耗时差异。
- 错误率。
- P95/P99 延迟。
- 是否跨机房。
- 是否出现热点 key。
- 重试是否集中打到少数实例。

真正目标不是“绝对平均”，而是在容量、延迟和可用性之间取得稳定分布。

## 常见策略

### 随机

从可用实例中随机选择一个。

优点：

- 实现简单。
- 在请求量足够大时分布相对均匀。
- 不需要维护复杂状态。

缺点：

- 小流量下可能不均匀。
- 不考虑实例性能差异。
- 不感知当前负载。

适合：

- 实例能力接近。
- 请求量较大。
- 对均匀性要求不极端的场景。

### 轮询

按顺序依次选择实例。

```text
A -> B -> C -> A -> B -> C
```

优点：

- 简单直观。
- 请求数分布均匀。

缺点：

- 不考虑请求耗时差异。
- 不考虑实例处理能力差异。
- 慢实例仍会被持续分配流量。

适合：

- 实例规格一致。
- 请求耗时相近。
- 没有明显热点。

### 加权轮询

给不同实例配置不同权重，权重高的实例分到更多请求。

示例：

| 实例 | 权重 |
| --- | --- |
| A | 5 |
| B | 3 |
| C | 2 |

理论流量比例是 5:3:2。

适合：

- 实例规格不同。
- 新实例预热低权重。
- 灰度发布逐步放量。
- 临时降低某个实例流量。

风险：

- 权重配置错误会造成倾斜。
- 静态权重无法反映实时负载。
- 发布系统和注册中心权重要一致。

### 平滑加权轮询

普通加权轮询可能出现短时间请求集中。平滑加权轮询会让权重分布更均匀。

例如权重 A=5、B=1、C=1，不希望前 5 次都打到 A，而是尽量穿插分配。

适合对短时间尖峰更敏感的 RPC 场景。

### 最少连接

选择当前活跃连接数或在途请求数最少的实例。

优点：

- 能一定程度避开慢实例。
- 比轮询更接近实时负载。

缺点：

- 需要维护连接或请求计数。
- 计数在分布式客户端之间不全局准确。
- 短请求和长请求混合时仍可能误判。

适合：

- 请求耗时差异较大。
- 长连接或长请求较多。
- 网关层能观察较完整连接状态。

### 最短响应时间

根据实例最近响应时间选择更快实例。

优点：

- 能动态避开慢实例。
- 对延迟敏感场景有价值。

缺点：

- 需要统计滑动窗口。
- 容易因短期抖动频繁切换。
- 快实例可能被打成慢实例。

常见改进：

- 使用 EWMA 平滑响应时间。
- 加入随机因子。
- 设置最低采样量。
- 配合熔断和限流。

### 一致性哈希

根据请求 key 选择实例，同一个 key 尽量路由到同一个实例。

```text
hash(userId) -> 哈希环 -> Provider
```

优点：

- 实例增减时只迁移部分 key。
- 适合本地缓存命中。
- 适合有会话粘性或状态亲和的场景。

缺点：

- key 分布不均会造成热点。
- 某些实例可能负载高。
- 实例数少时需要虚拟节点改善均匀性。

适合：

- 缓存服务。
- 会话粘性。
- 同一用户请求希望打到同一实例。
- 有本地状态或预热成本高的服务。

### 随机两次选择

随机选两个实例，再从中选负载更低的一个。

优点：

- 成本低。
- 能显著减少极端不均。
- 不需要全量比较所有实例。

适合大规模实例列表场景。

## 路由和负载均衡的区别

路由先过滤实例，负载均衡再选择实例。

示例：

```text
全部实例：
  A stable shanghai
  B stable shanghai
  C gray shanghai
  D stable beijing

请求标签：gray=true, zone=shanghai

路由过滤后：
  C

负载均衡：
  从过滤后的实例中选择
```

如果路由过滤后为空，要有兜底规则：

- 降级到 stable。
- 降级到同地域任意实例。
- 快速失败并告警。

## 负载均衡和重试

重试时通常不应该继续打同一个失败实例。

策略：

- 记录本次调用已经尝试过的实例。
- 重试时从剩余实例中选择。
- 如果失败原因是业务异常，不重试。
- 如果是连接失败或短暂超时，可换实例重试。

风险：

- 重试会放大流量。
- 如果所有实例都慢，换实例没有意义。
- 重试可能突破幂等边界。

## 负载均衡和熔断

熔断器打开的实例应从候选列表中排除或降低权重。

常见做法：

```text
实例错误率升高
  -> 客户端实例级熔断
  -> 负载均衡不再选择该实例
  -> 半开状态放少量探测请求
  -> 成功后恢复
```

注意区分：

- 服务级熔断：整个下游服务不可用。
- 实例级熔断：某个 Provider 实例不可用。

实例级熔断能避免一台坏机器持续拖累调用方。

## 负载均衡和预热

新实例刚启动时可能存在：

- JIT 未充分优化。
- 连接池未建立。
- 本地缓存为空。
- 类加载和初始化成本高。

如果直接分配满权重流量，容易造成新实例抖动。

预热方案：

- 初始低权重。
- 按时间逐步提升权重。
- 预先加载热点数据。
- 建立下游连接池。
- 观察指标后自动提升。

## 工程方案

### 普通内部 RPC

推荐：

- 默认随机或轮询。
- 支持权重。
- 支持实例级熔断。
- 支持同机房优先。
- 支持重试避开已失败实例。

### 缓存型服务

推荐：

- 一致性哈希。
- 虚拟节点。
- 热点 key 监控。
- 大 key 拆分。
- 必要时热点 key 多副本。

### 网关层流量

推荐：

- 加权轮询或最少连接。
- 健康检查。
- 慢启动。
- 主动摘除异常后端。
- 限流和熔断。

### 灰度发布

推荐：

- 基于标签先路由。
- 对灰度实例设置低权重。
- 支持用户、城市、版本、租户等维度。
- 灰度失败时权重归零。

## 典型场景

### 某台机器变慢

现象：

- 总体成功率下降不明显。
- P99 升高。
- 某个实例耗时明显高。

处理：

- 实例维度监控定位。
- 降低权重或摘除。
- 客户端实例级熔断。
- 排查该机器 CPU、GC、线程池、下游依赖。

### 热点用户打爆某实例

如果使用一致性哈希，同一个热点 key 会集中到一台实例。

解决：

- 热点 key 识别。
- key 加随机后缀拆分。
- 热点 key 多副本。
- 对热点请求单独限流。

### 新实例发布后大量超时

可能原因：

- 没有预热。
- 权重过高。
- 缓存未加载。
- 下游连接池冷启动。

解决：

- 慢启动。
- 预热请求。
- 降低初始权重。
- 发布平台分批放量。

## 线上风险

### 平均请求数不等于平均负载

同样 100 个请求，如果某个实例处理的都是慢请求，它的真实负载更高。

所以要看：

- 在途请求数。
- 响应时间。
- CPU 和线程池。
- 请求类型。

### 客户端视角不全局

客户端侧负载均衡只知道自己观察到的状态，不知道其他客户端也在如何选择实例。

影响：

- 某个实例可能被多个客户端同时选中。
- 局部最优不等于全局最优。

解决：

- 加随机性。
- 使用平滑统计。
- 服务端限流兜底。

### 路由规则过严

灰度、版本、机房多层过滤后可能无实例。

必须考虑：

- 过滤后为空的处理。
- 是否允许跨机房兜底。
- 是否快速失败。
- 是否告警。

### 权重配置错误

错误权重可能导致：

- 新实例被打满。
- 老实例无流量，缓存失效。
- 灰度流量过大。

权重变更需要审计、监控和回滚。

## 常见追问

### 轮询和随机怎么选

可以这样答：

```text
如果实例规格相近、请求量较大，随机和轮询都可以。轮询请求数更均匀，但不考虑耗时；随机实现简单且能避免所有客户端同步按同一顺序选择实例。实际 RPC 框架通常还会结合权重、路由、熔断和重试，不会只靠单一策略。
```

### 一致性哈希解决什么问题

它解决实例增减时 key 大规模迁移的问题，并提升本地缓存命中率。它不解决热点 key 问题，热点 key 仍可能集中到一台实例。

### 最少连接一定最好吗

不一定。它依赖连接数或在途请求数统计，不能完全反映 CPU、DB 等真实压力。客户端侧还存在视角不全局的问题。它适合长连接或请求耗时差异大的场景，但仍要配合限流和熔断。

### 重试时为什么要换实例

如果失败是实例级问题，继续打同一实例会浪费重试机会。换实例能提高短暂故障下的成功率。但如果失败是全局下游过载，重试只会放大流量。

## 易错点

- 只背策略，不讲适用场景。
- 认为轮询一定能均衡真实负载。
- 使用一致性哈希却忽略热点 key。
- 新实例没有预热，直接满权重。
- 重试仍打同一个失败实例。
- 路由过滤和负载均衡概念混淆。
- 没有实例维度监控，只看服务总体指标。
- 权重调整没有审计和回滚。

## 自检清单

- 能否说明路由和负载均衡的先后关系。
- 能否讲清随机、轮询、加权轮询、最少连接、一致性哈希。
- 能否解释一致性哈希为什么需要虚拟节点。
- 能否说明重试时为什么要避开失败实例。
- 能否识别慢实例、热点 key、新实例冷启动风险。
- 能否结合灰度、机房和权重讲工程方案。
- 能否说明客户端侧负载均衡的局限。
- 能否给出实例维度监控指标。
