# 服务注册与发现

## 面试定位

服务注册与发现是微服务体系的基础。面试官通常关注你是否理解服务实例如何上线、下线、健康检查、缓存、订阅变更，以及注册中心故障时调用方如何继续工作。

常见考点：

- 为什么需要注册中心。
- Provider 如何注册，Consumer 如何发现。
- 心跳、租约、健康检查的区别。
- 客户端发现和服务端发现的区别。
- 注册中心 AP/CP 取舍。
- 服务上下线、灰度、机房路由如何实现。
- 注册中心不可用时系统是否还能调用。

回答时要强调：注册中心不在每次请求的强同步链路上，客户端通常依赖本地缓存和变更订阅。

## 核心概念

### 服务实例

一个服务实例通常包含：

| 字段 | 含义 |
| --- | --- |
| serviceName | 服务名 |
| ip/port | 实例地址 |
| protocol | 协议，例如 dubbo、grpc、http |
| weight | 权重 |
| version/group | 版本、分组、环境 |
| zone/region | 机房、地域 |
| metadata | 标签、灰度标识、能力信息 |
| healthStatus | 健康状态 |
| lastHeartbeat | 最近心跳时间 |

### 注册中心职责

注册中心负责：

- 接收服务实例注册。
- 维护服务名到实例列表的映射。
- 处理实例续约和摘除。
- 向消费者推送服务列表变更。
- 存储服务元数据。
- 支持路由、权重、版本和分组能力。

注册中心通常不负责真实业务流量转发。

### 本地缓存

Consumer 一般会缓存服务列表：

- 启动时拉取一次。
- 长连接订阅变更。
- 注册中心推送变更后更新本地缓存。
- 注册中心不可用时继续使用旧缓存。

没有本地缓存的服务发现会把注册中心变成每次调用的强依赖，风险很高。

## 基本流程

### 服务注册流程

```text
Provider 启动
  -> 初始化 RPC 服务
  -> 暴露端口
  -> 构造实例元数据
  -> 向注册中心注册
  -> 周期性心跳或续约
  -> 监听关闭信号并主动注销
```

关键点：

- 端口真正可用后再注册。
- 健康检查通过后再对外暴露。
- 关闭时先摘流，再停止接收请求。

### 服务发现流程

```text
Consumer 启动
  -> 根据依赖服务名拉取实例列表
  -> 建立本地缓存
  -> 订阅实例变更
  -> 调用时从本地缓存读取实例
  -> 路由过滤
  -> 负载均衡选择实例
```

调用时一般不直接访问注册中心。

### 变更推送流程

```text
Provider 上线或下线
  -> 注册中心更新服务列表
  -> 通知订阅该服务的 Consumer
  -> Consumer 更新本地缓存
  -> 新请求按新列表路由
```

推送可能延迟或丢失，所以客户端也常配合定期全量拉取做兜底。

## 健康检查机制

### 心跳

Provider 周期性向注册中心发送心跳，表示自己还活着。

特点：

- 实现简单。
- 能发现进程崩溃、网络隔离等问题。
- 不一定能发现业务线程池满、DB 不可用等深层故障。

### 租约

注册中心为实例维护租约。超过一定时间没有续约，就认为实例失效。

关键参数：

- 心跳间隔。
- 租约过期时间。
- 剔除周期。

参数太短会导致网络抖动时误摘除；参数太长会导致故障实例残留时间过长。

### 主动健康检查

注册中心或独立健康检查组件主动访问实例健康接口。

可以检查：

- 进程是否存活。
- 端口是否可连。
- 关键依赖是否可用。
- 线程池和队列是否过载。

风险：

- 健康检查太重会反过来压垮服务。
- 深度检查依赖下游，可能导致故障传播。
- 健康接口要有超时和限流。

### 客户端侧健康判断

Consumer 可以根据调用结果临时降低或摘除异常实例。

例如：

- 连续连接失败。
- 连续超时。
- 错误率过高。
- 熔断器打开。

这种判断更接近真实调用质量，但也要避免因为少量波动误伤实例。

## 客户端发现和服务端发现

### 客户端发现

Consumer 自己从注册中心拿实例列表并做负载均衡。

优点：

- 少一层转发，性能好。
- 客户端可做精细路由、灰度和重试。
- 常见于 Dubbo、gRPC 客户端侧负载均衡。

缺点：

- 客户端逻辑复杂。
- 多语言 SDK 需要重复实现。
- 治理能力升级要推动客户端版本。

### 服务端发现

客户端请求统一入口，例如网关或负载均衡器，由入口查找后端实例。

优点：

- 客户端简单。
- 多语言接入成本低。
- 治理能力集中在网关层。

缺点：

- 多一跳网络转发。
- 网关可能成为瓶颈或单点。
- 某些端到端治理能力不如客户端精细。

### 面试回答

可以这样说：

```text
客户端发现适合内部高性能 RPC，调用方可以直接做路由、负载均衡和重试；服务端发现适合 HTTP 网关、跨语言或外部接入，客户端只需要访问统一入口。两者不是绝对对立，很多系统内部 RPC 用客户端发现，对外流量用网关做服务端发现。
```

## 注册中心一致性取舍

### AP 型注册中心

偏可用性和分区容忍，允许短时间服务列表不一致。

适合服务发现的原因：

- 服务发现可以容忍秒级延迟。
- Consumer 有本地缓存。
- 短时旧列表通常比完全不可用更好。

风险：

- 故障实例可能残留。
- 不同 Consumer 看到的实例列表可能短时间不一致。

### CP 型注册中心

偏一致性，分区时可能拒绝写入或不可用。

适合：

- 配置一致性要求高。
- 主节点选举。
- 分布式协调。

但用于服务发现时要评估注册中心不可用对服务上下线的影响。

### 工程观点

服务发现一般更重视可用性。因为每次调用真正依赖的是客户端本地服务列表，注册中心短时不可用时应该不影响已有服务之间的调用。

## 工程方案

### 优雅上线

上线流程建议：

```text
实例启动
  -> 初始化 Spring 容器
  -> 初始化连接池和缓存
  -> 启动 RPC 端口
  -> 预热关键资源
  -> 健康检查通过
  -> 注册到注册中心
  -> 小流量预热
  -> 逐步放大权重
```

避免刚启动就承接满流量。

### 优雅下线

下线流程建议：

```text
收到停止信号
  -> 从注册中心摘除或权重置 0
  -> 等待客户端感知变更
  -> 停止接收新请求
  -> 等待在途请求完成
  -> 关闭线程池和连接池
  -> 退出进程
```

如果直接 kill：

- Consumer 可能仍把请求打到旧实例。
- 在途请求中断。
- 连接异常导致大量重试。

### 灰度和路由

服务实例可以带元数据：

- env=test/prod。
- version=v1/v2。
- zone=shanghai/beijing。
- gray=true。
- tenant。
- appVersion。

Consumer 根据请求标签过滤实例：

```text
请求携带 grayUser=true
  -> 只路由到 gray=true 的 Provider
普通请求
  -> 路由到 stable Provider
```

灰度要有兜底规则，避免过滤后无实例。

### 多机房路由

常见策略：

- 同机房优先。
- 同地域优先。
- 跨机房兜底。
- 故障时自动切流。

多机房要关注：

- 跨机房延迟。
- 数据一致性。
- 容量是否足够承接切流。
- 注册中心数据是否按地域隔离。

### 本地缓存兜底

客户端本地缓存要考虑：

- 内存缓存。
- 磁盘快照。
- 启动时注册中心不可用能否使用上次快照。
- 缓存过期策略。
- 缓存为空时是否快速失败。

不要在服务列表为空时盲目打到随机地址。

## 典型场景

### Provider 崩溃

```text
Provider 崩溃
  -> 心跳停止
  -> 注册中心租约过期后摘除
  -> Consumer 收到变更推送
  -> 后续请求不再路由到该实例
```

在摘除前，Consumer 可能仍访问失败，所以客户端重试、熔断和异常实例隔离仍然需要。

### 发布新版本

安全发布：

- 新版本先注册小权重。
- 观察错误率、耗时和业务指标。
- 逐步提升权重。
- 异常时权重置 0 或注销实例。

### 注册中心短时不可用

已有 Consumer 应继续使用本地缓存调用。

影响：

- 新服务可能无法注册。
- 新 Consumer 可能启动失败。
- 服务列表变更无法及时推送。
- 已有服务之间调用不应立即中断。

## 线上风险

### 注册太早

服务端还没初始化完成就注册，Consumer 会打到不可用实例。

解决：

- 端口监听和资源初始化完成后再注册。
- 健康检查通过后再注册。
- 启动后低权重预热。

### 下线不优雅

直接停止实例会导致：

- 连接断开。
- 在途请求失败。
- Consumer 未及时感知。
- 上游重试放大流量。

解决：

- 先摘流。
- 等待传播延迟。
- 再关闭服务。

### 服务列表为空

可能原因：

- 注册中心异常。
- 路由规则过严。
- 版本或分组配置错。
- 灰度标签不匹配。

处理：

- 使用上一次可用列表兜底。
- 过滤后为空时退回默认路由或快速失败。
- 告警服务列表为空。

### 脑裂和网络分区

网络分区时，不同 Consumer 可能看到不同实例列表。服务发现通常接受短暂不一致，但要通过心跳、客户端失败隔离和定期全量同步来收敛。

## 常见追问

### 注册中心挂了，服务还能调用吗

可以这样答：

```text
如果 Consumer 已经拿到服务列表，并且本地缓存还在，已有服务之间通常还能继续调用。注册中心挂掉主要影响新实例注册、下线变更推送和新 Consumer 启动。为了保证可用性，客户端要有本地缓存和失败兜底，不能每次调用都依赖注册中心。
```

### 心跳正常是否代表服务可用

不一定。心跳只能证明进程和网络大概率可达，不代表业务线程池、数据库、缓存、下游依赖都正常。更准确的可用性要结合健康检查和真实调用指标。

### 如何避免发布时流量打到未预热实例

做法：

- 启动完成后再注册。
- 注册初期低权重。
- 预热连接池、JIT、缓存和热点数据。
- 观察指标后逐步提升权重。

### 服务发现为什么更偏 AP

服务发现数据允许短时间不一致，调用方还有本地缓存。分区时继续提供旧列表，通常比整个系统无法发现服务更可接受。

## 易错点

- 认为每次 RPC 都实时查询注册中心。
- 认为注册中心故障会立即导致所有调用失败。
- 只做服务端心跳，不看客户端真实失败率。
- 注册成功但服务端端口还不可用。
- 下线时先停进程再注销。
- 灰度路由过滤后没有空列表兜底。
- 多机房路由忽略容量和数据一致性。
- 把注册中心当配置中心或数据库滥用。

## 自检清单

- 能否画出 Provider 注册和 Consumer 发现流程。
- 能否解释本地缓存为什么重要。
- 能否说明心跳、租约和健康检查的区别。
- 能否回答注册中心挂了已有调用是否受影响。
- 能否讲清优雅上线和优雅下线。
- 能否说明客户端发现和服务端发现的优缺点。
- 能否解释服务发现为什么通常更看重可用性。
- 能否识别服务列表为空、注册太早、下线不优雅等风险。
