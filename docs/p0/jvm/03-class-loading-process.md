# 类加载过程：加载、验证、准备、解析、初始化

## 面试定位

类加载过程是 JVM 和框架原理之间的桥梁。Spring、SPI、JDBC、Tomcat、热部署、动态代理、插件化、隔离加载都离不开类加载。面试中要能把加载、验证、准备、解析、初始化五个阶段说清楚，并能解释什么时候触发类初始化。

推荐回答：

1. 类加载包括加载、验证、准备、解析、初始化，验证、准备、解析合称连接。
2. 加载阶段把类的二进制字节流变成方法区中的类元数据，并创建 `Class` 对象。
3. 验证保证字节码安全。
4. 准备为类变量分配内存并设置零值。
5. 解析把符号引用转成直接引用。
6. 初始化执行类构造器 `<clinit>`，为静态变量赋真实值，执行静态代码块。

## 生命周期总览

```text
加载 -> 验证 -> 准备 -> 解析 -> 初始化 -> 使用 -> 卸载
          \_________ 连接 _________/
```

注意：

- 解析阶段在某些情况下可以发生在初始化之后，这是为了支持 Java 的动态绑定。
- 类加载是按需进行的，不是 JVM 启动时把所有类一次性加载。
- 类加载器不仅负责加载类，也参与类身份判断。同一个类名由不同类加载器加载，在 JVM 中可以被视为不同类型。

## 加载

加载阶段主要做三件事：

1. 通过类的全限定名获取定义该类的二进制字节流。
2. 将字节流代表的静态存储结构转换为方法区的运行时数据结构。
3. 在堆中生成一个代表该类的 `java.lang.Class` 对象，作为访问方法区类元数据的入口。

字节流来源可以很多：

- 本地 `.class` 文件。
- Jar 包。
- 网络。
- 动态代理运行时生成。
- JSP 编译生成。
- 加密 class 解密后加载。

面试中可以补充：数组类不是通过类加载器直接加载，而是由 JVM 直接创建；数组元素类型仍需要由类加载器加载。

## 验证

验证阶段保证 class 文件字节流符合 JVM 规范，不会危害虚拟机安全。

常见验证内容：

- 文件格式验证：魔数、版本号、常量池结构是否合法。
- 元数据验证：类继承关系、是否继承 final 类、抽象方法实现是否合法。
- 字节码验证：操作数栈和局部变量表类型是否匹配、跳转是否合法。
- 符号引用验证：访问权限、类和字段方法是否存在等。

验证的意义：

- Java 语言编译器生成的字节码通常安全，但 JVM 不能假设所有 class 都来自可信编译器。
- 动态生成字节码、手写字节码、被篡改 class 都需要防护。

## 准备

准备阶段为类变量分配内存并设置初始零值。这里的类变量指 `static` 变量，不包括实例变量。

示例：

```java
public class Demo {
    static int a = 10;
    static final int B = 20;
    static Object obj = new Object();
}
```

准备阶段：

- `a` 分配内存并设为 `0`，不是 `10`。
- `obj` 分配引用槽并设为 `null`，不是创建对象。
- 对于编译期常量 `static final int B = 20`，可能在准备阶段就设为常量值。

初始化阶段才会执行赋值语句，把 `a` 设为 `10`，执行 `new Object()`。

面试易错点：准备阶段不是执行静态代码块，也不是给静态变量赋源码中的真实值。

## 解析

解析阶段将常量池中的符号引用替换为直接引用。

符号引用：

- 用一组符号描述目标，可以与虚拟机内存布局无关。
- 例如类名、字段名、方法名、描述符。

直接引用：

- 可以直接定位到目标的指针、偏移量或句柄。
- 与 JVM 运行时内存布局相关。

解析对象包括：

- 类或接口。
- 字段。
- 类方法。
- 接口方法。
- 方法类型。
- 方法句柄。
- 动态调用点等。

解析不一定全部一次性完成，JVM 可以按需解析。

## 初始化

初始化阶段执行类构造器 `<clinit>`。它由编译器收集类变量赋值语句和静态代码块合并生成。

特点：

- `<clinit>` 按源码顺序收集静态变量赋值和静态代码块。
- 父类 `<clinit>` 通常先于子类执行。
- JVM 会保证一个类的 `<clinit>` 在多线程环境下被正确加锁同步。
- 如果 `<clinit>` 执行失败，类可能进入错误状态，后续使用可能抛出 `NoClassDefFoundError`。

示例：

```java
class Parent {
    static int p = init("parent");
}

class Child extends Parent {
    static int c = init("child");
}
```

初始化 `Child` 前，会先初始化 `Parent`。

## 主动引用与被动引用

触发类初始化的常见主动引用：

- `new` 创建类实例。
- 访问或设置类的静态变量，编译期常量除外。
- 调用类的静态方法。
- 使用反射调用类。
- 初始化子类时，父类需要先初始化。
- JVM 启动时指定的主类。
- `MethodHandle`、动态语言支持等场景。

不会触发或不一定触发初始化的被动引用：

```java
System.out.println(Child.parentStaticField); // 只定义在父类，可能只初始化父类
Parent[] arr = new Parent[10];               // 创建数组不会初始化 Parent
System.out.println(ConstClass.VALUE);         // 编译期常量可能被内联
```

## 类初始化顺序

实例化对象时常见顺序：

1. 父类静态变量和静态代码块。
2. 子类静态变量和静态代码块。
3. 父类实例变量和实例代码块。
4. 父类构造方法。
5. 子类实例变量和实例代码块。
6. 子类构造方法。

同一类中，静态变量赋值和静态代码块按源码顺序执行；实例变量赋值和实例代码块也按源码顺序执行。

## 类卸载

一个类要被卸载，通常需要满足：

- 该类所有实例都已不可达。
- 加载该类的类加载器不可达。
- 该类对应的 `Class` 对象不可达。

Bootstrap ClassLoader 加载的核心类通常不会卸载。类卸载更多出现在自定义类加载器、热部署、插件化、脚本引擎等场景。

## 排查和调优方法

类加载问题常见表现：

- `ClassNotFoundException`：通常是加载阶段找不到类，常见于 classpath 缺失。
- `NoClassDefFoundError`：编译期存在或曾经加载过，但运行时无法使用，可能是依赖缺失或初始化失败。
- `LinkageError`：类冲突、版本不兼容、重复定义等。
- `ClassCastException`：同名类由不同类加载器加载，类型身份不同。
- `OutOfMemoryError: Metaspace`：类或类加载器过多，无法卸载。

常用命令：

```bash
jcmd <pid> VM.classloader_stats
jcmd <pid> GC.class_histogram
jcmd <pid> VM.system_properties
jcmd <pid> VM.command_line
java -verbose:class ...
```

JDK 9 以后也可以使用统一日志：

```bash
-Xlog:class+load=info,class+unload=info
```

## 常见追问

### 准备阶段和初始化阶段有什么区别？

准备阶段给类变量分配内存并设置零值；初始化阶段执行 `<clinit>`，按代码给静态变量赋真实值并执行静态代码块。

### `static final` 常量什么时候赋值？

编译期可确定的基本类型和字符串常量可能在准备阶段被设置为常量值，也可能被调用方内联。运行期才能确定的 `static final` 对象或方法返回值仍要在初始化阶段执行。

### 为什么类加载要验证？

因为 JVM 执行的是 class 字节码，不一定来自 Java 编译器。验证可以防止恶意或错误字节码破坏内存安全和类型安全。

### `ClassNotFoundException` 和 `NoClassDefFoundError` 区别？

`ClassNotFoundException` 是受检异常，通常由显式加载类失败触发，例如 `Class.forName`。`NoClassDefFoundError` 是错误，常见于类在编译期存在但运行时缺失、依赖缺失、初始化失败后再次使用。

### 一个类能否被加载多次？

同一个类加载器对同一个类名通常只加载一次。但同一个全限定名的 class 可以被不同类加载器分别加载，JVM 会认为它们是不同类型。

## 易错点

- 把加载阶段和初始化阶段混为一谈。
- 认为准备阶段会执行静态代码块。
- 认为 `static final` 一定都在初始化阶段赋值。
- 忽略类加载器参与类身份判断。
- 认为解析一定发生在初始化之前且一次性完成。
- 不区分 `ClassNotFoundException`、`NoClassDefFoundError`、`LinkageError`。

## 项目表达

可以这样表达类加载排查：

> 我们遇到过一个线上 `ClassCastException`，两个对象打印出来类名完全一样，但无法强转。后来通过打印 `obj.getClass().getClassLoader()` 和类加载路径，发现插件框架中同一个 Jar 被宿主和插件类加载器各加载了一份。最终把公共 API 包上移到父加载器，并隔离插件私有依赖，解决了类身份不一致的问题。

Metaspace 方向：

> 某次热部署后 Metaspace 持续上涨，我们用 `jcmd VM.classloader_stats` 观察类加载器数量，发现旧版本类加载器没有释放。继续查 GC Roots，定位到线程上下文类加载器和静态缓存持有引用。清理引用后，类卸载恢复正常。
