# 业务背景

## 面试定位

业务背景不是项目开场白，而是解释「为什么这个项目值得做」。面试官通过这一段判断你是否理解业务目标、用户场景、约束条件和技术工作的真实价值。对 Java 后端面试来说，业务背景要能自然引出后面的系统架构、核心链路、技术难点和结果指标；对 AI Agent 类项目，还要讲清楚 Agent 解决的是哪类人工流程、知识流程或自动化流程问题。

一个合格的业务背景应该回答四个问题：

- 业务服务谁：用户、运营、商家、内部研发、审核、客服、数据分析、企业客户。
- 原来有什么问题：效率低、成本高、体验差、错误多、规模上不去、人工依赖重、合规风险高。
- 为什么现在要做：业务增长、组织策略、成本压力、稳定性问题、平台化诉求、AI 能力成熟、竞品压力。
- 技术系统承担什么价值：提升吞吐、降低延迟、保证一致性、支持快速迭代、沉淀平台能力、减少人工操作。

## 核心目标

准备业务背景时，目标不是把业务讲大，而是把技术项目放到清楚的业务上下文里。

- 让面试官知道项目不是孤立编码任务，而是业务链路中的关键环节。
- 让后续所有技术选择都有依据，例如为什么要异步、为什么要缓存、为什么要拆服务、为什么要加 Agent。
- 让个人贡献具备业务价值锚点，而不是只停留在「我写了几个接口」。
- 让结果指标可以被理解，例如 QPS 提升、P99 降低、人工耗时减少、任务成功率提升。

## 准备要点

### 业务域

先用一句话定义业务域。

示例：

- 这是一个面向 C 端用户的权益发放系统，负责在用户完成任务后计算、锁定并发放对应权益。
- 这是一个内部研发 AI Agent 平台，负责把需求分析、代码检索、测试触发、文档生成等流程工具化和自动化。
- 这是一个面向运营配置的活动平台，负责支持多活动、多策略、多渠道的快速上线和效果追踪。

### 用户和角色

至少准备三类角色：

- 直接用户：真正使用系统的人，例如 C 端用户、运营、研发、客服、审核人员。
- 业务负责人：关心转化、成本、履约、效率和风险的人。
- 系统依赖方：调用你的接口、消费你的消息、依赖你的数据或配置的人。

### 原有痛点

痛点要具体，最好能落到链路、数据或成本。

常见表达：

- 人工操作多：每次活动上线都需要研发改代码，配置周期长。
- 数据链路长：状态散落在多个系统，排查一次问题需要跨库、跨日志、跨消息。
- 高峰压力大：活动集中触发时同步链路变长，接口超时和下游限流明显。
- 一致性难保障：权益、库存、订单、任务状态存在跨系统更新，异常时容易出现重复发放或漏发。
- Agent 场景下流程不可控：大模型能生成答案，但缺少工具权限、执行校验、结果验证和审计。

### 业务规模

准备数字时不需要夸张，但要有量级。

可准备的维度：

- 日活、月活、请求量、峰值 QPS。
- 单日任务数、订单数、消息数、数据量。
- 活动数量、策略数量、配置数量、接入方数量。
- 人工处理时长、研发排期周期、运营等待时间。
- AI Agent 的日均任务数、工具调用次数、人工采纳率、失败重试率。

如果不能透露真实数字，可以用区间或量级：

- 峰值 QPS 在千级。
- 单日处理量在百万级。
- 配置项从几十个增长到数百个。
- 单次流程从小时级人工处理压缩到分钟级自动化。

### 技术约束

业务背景里要埋下技术约束，让后面技术方案有根。

常见约束：

- 稳定性：核心链路不能影响主交易、主履约或核心用户路径。
- 性能：高峰请求集中，不能把压力打到数据库或下游服务。
- 一致性：不能重复发奖、不能漏发、不能出现用户状态和资产状态不一致。
- 可运营：策略要能配置、灰度、回滚、追踪。
- 合规和权限：涉及用户数据、资金、权益、内部工具调用时要做权限和审计。
- 交付周期：业务窗口固定，技术方案必须兼顾长期演进和短期上线。
- AI Agent 约束：模型输出不稳定，工具调用要可控，关键动作要有确认、回放和兜底。

## 展开框架

### 三段式框架

面试中推荐用三段式讲业务背景：

1. 业务是什么：一句话定义业务域和服务对象。
2. 问题是什么：说明原链路的痛点、规模和约束。
3. 项目要解决什么：说明目标、边界和预期收益。

示例结构：

```text
这个项目服务于什么业务场景。
当时业务增长到什么阶段，原有链路出现了哪些具体问题。
我们这个项目的目标是解决哪些问题，系统上主要负责哪一段能力。
```

### 业务问题到技术问题映射

| 业务问题 | 技术问题 | 可引出的后续内容 |
| --- | --- | --- |
| 活动配置上线慢 | 配置模型、规则引擎、灰度发布 | 系统架构、方案权衡 |
| 高峰请求超时 | 缓存、异步化、限流、线程池隔离 | 技术难点、性能指标 |
| 权益重复发放 | 幂等、分布式锁、事务边界、状态机 | 核心链路、故障复盘 |
| 排查问题慢 | traceId、日志规范、监控告警、数据看板 | 指标、复盘 |
| 人工流程重 | Agent 编排、工具调用、审批确认、审计 | AI Agent 项目表达 |
| 接入方越来越多 | API 契约、版本兼容、扩展点、平台化 | 可扩展性 |

### 30 秒版本

适合面试刚开始或项目列表介绍。

```text
这个项目是一个面向某类业务场景的后端系统，主要解决原链路在效率、稳定性和扩展性上的问题。业务上，当时面临请求规模增长、策略快速变化和人工处理成本高的压力；技术上，需要在保证核心链路稳定的前提下，把配置、计算、发放、监控和异常处理能力平台化。我主要负责其中的核心链路设计和后端实现，后续通过异步化、幂等控制、缓存和监控建设，把接口稳定性和交付效率提升到了可量化水平。
```

### 2 分钟版本

适合正式展开。

```text
项目背景可以分成三块。第一，业务侧要支持更多活动和策略，原来很多逻辑依赖人工配置或研发硬编码，迭代成本高。第二，用户规模和请求峰值上来后，同步链路变长，下游服务和数据库压力增大，稳定性风险变高。第三，运营和客服需要更好的可观测性，出现异常时要能快速定位用户、策略、订单或权益状态。

所以这个项目的目标不是单纯新增几个接口，而是把核心业务流程抽象成稳定的后端能力：入口统一、规则可配置、状态可追踪、发放可幂等、异常可补偿、结果可监控。我在项目中主要负责核心服务设计、关键接口实现、异步消息链路、幂等和监控建设。
```

## 常见追问

### 这个项目为什么重要

回答重点：

- 连接业务目标和技术价值。
- 说明如果不做会有什么损失。
- 不只说「领导重视」或「业务需要」。

示例：

```text
重要性主要体现在两点。业务上，它影响活动上线效率和用户履约体验；技术上，它处在核心链路里，一旦链路不稳定会直接带来用户投诉、重复发放或运营无法追踪的问题。所以项目价值不只是支持一个功能，而是把原来容易出错、难扩展的链路变成可配置、可观测、可兜底的系统能力。
```

### 你们为什么不用现成系统

回答重点：

- 先承认复用是首选。
- 再讲现有系统不满足的边界。
- 最后说明新建或改造的必要性。

示例：

```text
我们一开始也评估过复用现有能力。现有系统能覆盖通用配置和基础发放，但这个场景里有更强的实时性、状态一致性和策略组合要求，而且还要和已有用户状态、任务系统、资产系统打通。如果直接复用，会把大量定制逻辑塞进通用系统，反而影响稳定性和后续维护。因此最后选择在通用能力之上封装一层领域服务，把特殊业务规则留在领域层。
```

### 业务规模到底有多大

回答重点：

- 用量级，不要编造精确值。
- 讲峰值和平均值的差异。
- 讲规模对技术方案的影响。

示例：

```text
规模大概是日常请求在稳定量级，活动或批量任务时会出现明显峰值。这个特点决定了方案不能只看平均 QPS，而要重点处理瞬时流量、下游限流和消息堆积。因此我们在入口做了限流和参数校验，在核心写链路做幂等，在非强实时部分做异步化，并补充了积压和失败率监控。
```

### AI Agent 项目的业务价值怎么讲

回答重点：

- 不要只说用了大模型。
- 要讲原流程、人工成本、工具边界和风险控制。

示例：

```text
这个 Agent 项目的业务背景不是为了展示模型能力，而是解决内部研发流程中重复、高上下文成本的问题。原来研发要自己查需求、搜代码、跑命令、整理文档和触发测试，这些步骤都依赖人手工串联。Agent 的价值是把这些步骤编排成可控流程，通过工具调用、权限控制、执行日志和结果校验，让模型从回答问题变成完成任务的协作入口。
```

## 优秀回答结构

优秀回答一般包含这些层次：

1. 一句话业务定义：项目服务的业务场景是什么。
2. 背景变化：业务规模、策略复杂度、交付周期或稳定性要求发生了什么变化。
3. 核心痛点：原链路哪里慢、哪里不稳、哪里不可控。
4. 技术承担：后端系统负责把哪些能力平台化或服务化。
5. 个人入口：你负责哪一段关键能力。
6. 量化结果：后面可以用哪些指标证明项目价值。

可复用模板：

```text
这个项目服务于【业务场景】，核心用户是【用户/运营/研发/系统接入方】。
项目启动前，主要问题是【痛点一】、【痛点二】和【痛点三】。
这些问题带来的直接影响是【业务影响】和【技术风险】。
因此项目目标是建设【系统能力】，在【约束条件】下支持【业务目标】。
我主要负责【个人模块】，后续可以从【核心链路/难点/指标】展开。
```

## 常见误区

- 只讲业务愿景，不讲技术系统承担什么。
- 只讲技术名词，不讲业务问题从哪里来。
- 把项目说得过大，但个人负责范围说不清。
- 背景里堆概念，例如「赋能」「闭环」「中台」，但没有具体链路。
- 没有量级，导致后面的性能、架构和权衡缺乏依据。
- 把 AI Agent 项目讲成「接了一个大模型接口」，没有说明工具、流程、权限和评估。
- 不能解释为什么现在做、为什么不用原方案、为什么这件事值得投入。

## 自检清单

- 能否用 30 秒讲清楚项目业务背景。
- 能否用 2 分钟讲清楚问题、目标、约束和个人入口。
- 是否准备了业务规模的量级表达。
- 是否能把业务痛点映射到技术问题。
- 是否能说明项目不做会有什么业务或系统风险。
- 是否能解释为什么这个项目适合 Java 后端能力展示。
- 如果是 AI Agent 项目，是否讲清楚 Agent 解决的是流程问题、工具问题还是知识问题。
- 是否避免了空泛词汇，所有背景描述都能落到真实链路。
- 是否为后续「系统架构」「核心链路」「技术难点」「指标结果」埋好了线索。
