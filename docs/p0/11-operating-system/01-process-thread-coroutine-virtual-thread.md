# 进程、线程、协程、虚拟线程的区别

## 面试定位

这是操作系统和 Java 并发的入口题。面试官通常不是只要定义，而是要看你能不能把资源隔离、调度单位、上下文切换成本、阻塞行为和 Java 工程实践串起来。

高频追问：

- 进程和线程谁是资源分配单位，谁是调度单位。
- 为什么线程切换比进程切换轻。
- 协程为什么轻量，阻塞系统调用会发生什么。
- Java 虚拟线程和协程、平台线程有什么关系。
- 虚拟线程适合 IO 密集还是 CPU 密集。
- 线上线程数过多、线程池打满、CPU 上下文切换高怎么排查。

## 一句话对比

| 概念 | 资源归属 | 调度方 | 切换成本 | 并行能力 | 典型场景 |
| --- | --- | --- | --- | --- | --- |
| 进程 | 独立地址空间、文件表、信号等 | OS 内核 | 高 | 多核并行 | 服务进程、隔离部署 |
| 线程 | 共享进程地址空间，独立栈和寄存器上下文 | OS 内核 | 中 | 多核并行 | Java 请求处理、线程池 |
| 协程 | 通常共享线程资源，独立协程栈或状态机 | 用户态运行时 | 低 | 依赖承载线程 | 高并发 IO、异步流程 |
| 虚拟线程 | Java 虚拟线程对象，运行时映射到平台线程 | JVM 调度，底层仍依赖 OS 线程 | 低于平台线程阻塞模型 | 依赖 carrier 平台线程并行 | Java 高并发阻塞式 IO |

核心结论：

- 进程是资源隔离边界。
- 线程是操作系统 CPU 调度的基本单位。
- 协程是用户态调度的轻量执行流。
- Java 虚拟线程是 JVM 提供的轻量线程，让阻塞式代码获得接近异步模型的伸缩性。

## 进程

进程是程序的一次运行实例，也是操作系统进行资源分配和保护的基本单位。

进程通常拥有：

- 独立虚拟地址空间。
- 代码段、数据段、堆、栈、mmap 区。
- 文件描述符表。
- 信号处理信息。
- 当前工作目录、环境变量。
- 用户、权限、命名空间、cgroup 等运行上下文。

进程间地址空间隔离，一个进程崩溃一般不会直接破坏另一个进程内存。但隔离带来成本，进程创建、进程切换、进程间通信通常比线程更重。

常见进程间通信方式：

- 管道、命名管道。
- Unix Domain Socket。
- TCP/UDP socket。
- 共享内存。
- 消息队列。
- 信号。
- mmap 文件。

Java 后端中，一个服务实例通常对应一个或多个 JVM 进程。进程级问题常见于：

- JVM 进程 CPU 高。
- 进程内存 RSS 持续上涨。
- 进程文件描述符耗尽。
- 进程被 OOM Killer 杀掉。
- 容器 cgroup 限制导致资源不足。

## 线程

线程是进程内的执行流，也是 Linux 调度器实际调度 CPU 的基本实体。Linux 下线程和进程都以 task 形式存在，只是线程组共享更多资源。

线程拥有：

- 独立程序计数器。
- 独立寄存器上下文。
- 独立线程栈。
- 独立线程本地存储。
- 独立调度状态和优先级。

同一进程内线程共享：

- 虚拟地址空间。
- 堆内存。
- 方法区或元空间等 JVM 进程资源。
- 文件描述符表。
- 信号处理配置的一部分。

线程的优点：

- 创建和切换比进程轻。
- 共享内存，通信方便。
- 能利用多核并行。

线程的风险：

- 共享内存带来数据竞争。
- 锁竞争、死锁、可见性问题。
- 线程过多会增加上下文切换、栈内存和调度成本。
- 阻塞式 IO 会占住平台线程。

Java 平台线程通常一对一映射到底层 OS 线程。`Thread`、线程池、`synchronized`、`ReentrantLock`、`park/unpark` 最终都要和 OS 调度、阻塞、唤醒发生关系。

## 协程

协程是用户态的轻量执行流。它通常由语言运行时或框架调度，不直接作为 OS 调度实体。

协程的关键特点：

- 创建成本低。
- 切换不需要陷入内核，通常只保存少量寄存器、栈指针或状态机状态。
- 可以用少量 OS 线程承载大量并发任务。
- 需要运行时配合非阻塞 IO 或异步调度。

协程调度方式常见两类：

- 有栈协程：每个协程有自己的栈，可在函数调用深处挂起。
- 无栈协程：编译器把异步流程改写成状态机，例如 async/await。

协程的核心问题是阻塞边界。如果协程中直接调用会阻塞 OS 线程的系统调用，而运行时没有做 hook 或替换，承载它的线程会被卡住，导致同一线程上的其他协程也无法执行。

工程上要关注：

- 阻塞 API 是否被协程运行时适配。
- CPU 密集任务是否会霸占调度线程。
- 是否需要主动让出执行权。
- 调度器线程数和任务队列是否合理。

## Java 虚拟线程

虚拟线程是 JDK 提供的轻量线程能力。它保留 `Thread` 编程模型，但不再让每个 Java 线程长期独占一个 OS 平台线程。

基本用法：

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 100_000; i++) {
        executor.submit(() -> {
            String body = httpClient.send(request, BodyHandlers.ofString()).body();
            handle(body);
        });
    }
}
```

虚拟线程的关键机制：

- 虚拟线程由 JVM 调度。
- 真正执行时挂载到 carrier 平台线程。
- 遇到可感知的阻塞点时，虚拟线程可以卸载，carrier 线程去执行其他虚拟线程。
- 业务代码可以继续写同步阻塞风格，避免 callback 或复杂异步链。

适合场景：

- 请求量高、每个请求大部分时间在等待数据库、RPC、HTTP、文件 IO。
- 想保留同步代码可读性。
- 原线程池模型因阻塞导致线程数膨胀。

不适合或收益有限：

- CPU 密集计算。CPU 核数固定，虚拟线程不能让 CPU 计算突破物理上限。
- 大量持有 monitor 锁或本地方法导致虚拟线程 pinning 的场景。
- 依赖固定线程身份的代码，例如滥用 `ThreadLocal` 保存大对象。

## 底层机制对比

### 地址空间和资源隔离

进程之间地址空间隔离，线程和协程通常共享进程地址空间。隔离越强，故障影响越小，但通信和切换成本越高。

Java 服务部署时，进程隔离常用于：

- 多实例部署。
- 灰度和回滚。
- 容器资源限制。
- 故障隔离。

线程和协程更适合进程内部并发。

### 栈内存

平台线程通常有较大的线程栈。Java 可通过 `-Xss` 设置线程栈大小。线程数很多时，即使堆不大，线程栈也可能导致内存压力。

虚拟线程栈不是一次性分配一大块连续内存，而是按需管理，创建大量虚拟线程的内存成本远低于平台线程。

### 调度和切换

进程或平台线程切换一般需要内核参与，保存和恢复寄存器、栈指针、程序计数器、内存映射相关状态，并可能引发 TLB、缓存命中率下降。

协程或虚拟线程的用户态切换成本更低，但最终要在有限的平台线程上运行，真正并行度仍受 CPU 核数和 carrier 线程数限制。

### 阻塞行为

| 场景 | 平台线程 | 协程 | 虚拟线程 |
| --- | --- | --- | --- |
| 阻塞式 socket read | OS 线程阻塞 | 如果未适配，会阻塞调度线程 | 可卸载虚拟线程，释放 carrier |
| CPU 长计算 | 占用 CPU | 占用调度线程 | 占用 carrier |
| synchronized 竞争 | 线程阻塞或等待 | 取决于运行时 | 可能发生 pinning，需注意 |
| 非阻塞 IO | 需要事件循环配合 | 常见基础 | JVM 或库层适配 |

## 命令和示例

查看进程：

```bash
ps -ef | grep java
top
```

查看线程：

```bash
top -H -p <pid>
ps -T -p <pid>
```

查看线程数：

```bash
cat /proc/<pid>/status | grep Threads
ls /proc/<pid>/task | wc -l
```

查看 Java 线程栈：

```bash
jstack -l <pid> > /tmp/jstack.txt
jcmd <pid> Thread.print -l > /tmp/thread.txt
```

定位高 CPU 线程：

```bash
top -H -p <pid>
printf "%x\n" <tid>
grep -n "nid=0x<hex>" -A 40 /tmp/jstack.txt
```

## 线上排查

### 线程数过多

现象：

- `Threads` 持续增长。
- `unable to create new native thread`。
- RSS 上升但堆不一定高。
- 上下文切换增加。

排查：

```bash
cat /proc/<pid>/status | egrep "Threads|VmRSS|VmSize"
jstack -l <pid> | grep '^"' | sed 's/" .*//' | sort | uniq -c | sort -nr | head
```

常见原因：

- 线程池无界创建。
- 每个请求创建线程。
- 定时任务重复初始化线程池。
- 第三方 SDK 内部线程未关闭。
- 连接池、调度器、监听器重复注册。

### CPU 高但线程很多

先区分是单个线程打满，还是大量线程频繁切换：

```bash
top -H -p <pid>
vmstat 1
pidstat -w -p <pid> 1
```

如果 `cs` 很高、CPU 利用率高但业务吞吐不涨，可能是线程数过多、锁竞争或频繁 park/unpark。

### 虚拟线程排查关注点

- 不要只看 Java 线程数量判断 OS 线程压力，虚拟线程数量可以很大。
- 关注 carrier 线程是否被 CPU 密集任务占满。
- 检查是否存在大量 `ThreadLocal` 大对象。
- 检查同步块、native 调用、长时间持锁导致的 pinning。

## 常见追问

### 进程和线程的根本区别是什么

进程是资源分配和隔离单位，拥有独立地址空间；线程是 CPU 调度单位，同一进程内线程共享地址空间和大部分资源。

### 为什么线程切换比进程切换轻

线程共享进程地址空间，切换时不需要切换完整地址空间和大量进程级资源，TLB 和缓存影响通常也小一些。但线程切换仍需要内核调度，不是零成本。

### 协程为什么不能替代线程

协程不是 OS 调度实体，真正执行还要依赖线程。协程能降低并发等待成本，但不能突破 CPU 核数限制；遇到未适配的阻塞调用还可能卡住承载线程。

### 虚拟线程和线程池是什么关系

传统线程池复用少量平台线程，避免平台线程创建成本。虚拟线程创建便宜，推荐按任务创建虚拟线程，而不是把虚拟线程再池化。虚拟线程底层仍由 JVM 调度到平台线程执行。

### 虚拟线程能否提升 CPU 密集任务性能

不能。CPU 密集任务瓶颈是 CPU 核数和算法效率。虚拟线程主要改善大量阻塞等待场景的资源占用和编程模型。

## 易错点

- 把进程说成调度单位而不提线程。
- 认为线程之间完全隔离，忽略共享堆和文件描述符。
- 认为协程一定不会阻塞，忽略阻塞系统调用和运行时适配。
- 认为虚拟线程就是操作系统线程。
- 用固定大小线程池处理大量长时间阻塞请求，却不设置超时和队列保护。
- 线程数越多不等于吞吐越高，过多线程会增加切换和内存成本。
- `ThreadLocal` 在线程池和虚拟线程场景都要注意清理和对象大小。

## 自检清单

- 能否说清进程、线程、协程、虚拟线程各自的资源和调度边界。
- 能否解释线程切换为什么比进程切换轻。
- 能否说明协程遇到阻塞调用时的风险。
- 能否解释虚拟线程适合 IO 密集而不是 CPU 密集。
- 能否用 `top -H`、`jstack`、`/proc/<pid>/status` 排查线程问题。
- 能否把 Java 线程池、虚拟线程和 OS 调度联系起来。
