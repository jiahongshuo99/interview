# 上下文切换

## 面试定位

上下文切换是连接 CPU 调度、线程模型、锁竞争和线上性能排查的关键概念。Java 后端面试中常见问法是：线程越多越好吗，为什么锁竞争会导致性能下降，`vmstat` 里的 `cs` 高说明什么，协程和虚拟线程为什么切换成本低。

回答时要覆盖：

- 什么是上下文。
- 进程切换、线程切换、中断切换有什么区别。
- 上下文切换有什么成本。
- 如何用 Linux 命令观察。
- Java 服务中哪些场景会导致切换变多。

## 核心概念

上下文是 CPU 继续执行某个任务所需的状态，通常包括：

- 程序计数器。
- 通用寄存器。
- 栈指针。
- CPU 标志寄存器。
- 内存管理相关状态。
- 调度状态。
- 内核栈。

上下文切换是操作系统暂停当前任务，保存其上下文，再恢复另一个任务上下文的过程。

常见类型：

| 类型 | 触发原因 | 成本 |
| --- | --- | --- |
| 进程上下文切换 | 调度另一个进程运行 | 较高，可能切换地址空间 |
| 线程上下文切换 | 同进程或跨进程线程切换 | 中等，同进程线程较轻 |
| 中断上下文切换 | 设备中断、时钟中断 | 与中断处理相关 |
| 用户态到内核态切换 | 系统调用、异常、中断 | 特权级切换，不一定换线程 |
| 协程切换 | 用户态运行时调度 | 通常较低 |

## 触发上下文切换的原因

### 时间片用完

线程运行时间片耗尽，调度器选择其他可运行线程执行。

### 主动阻塞

线程调用阻塞 API：

- `read` 等待网络数据。
- `accept` 等待连接。
- `sleep` 休眠。
- `wait`、`park` 等待条件。
- `join` 等待线程结束。

### 锁竞争

线程获取锁失败，进入等待队列，当前线程被挂起，调度器运行其他线程。

### 更高优先级任务就绪

更高优先级或调度器认为更合适的任务变为可运行。

### 中断和异常

硬件中断、缺页异常、系统调用等都会导致 CPU 切入内核处理。

## 上下文切换成本

直接成本：

- 保存和恢复寄存器。
- 进入内核调度逻辑。
- 更新调度队列和任务状态。
- 切换内核栈。

间接成本：

- CPU cache 局部性变差。
- TLB 失效或命中率下降。
- 分支预测状态受影响。
- 锁竞争导致唤醒后继续抢锁。
- 大量线程导致调度器负担增加。

所以线程数不是越多越好。对于 CPU 密集任务，线程数接近 CPU 核数通常更合理；对于 IO 密集任务，可以更多，但必须受连接池、超时、队列和下游容量约束。

## 进程切换和线程切换

进程切换通常更重，因为可能涉及：

- 切换虚拟地址空间。
- 切换页表基址。
- 刷新或影响 TLB。
- 切换更多进程级资源。

同进程内线程切换较轻，因为共享地址空间和文件描述符等资源。但线程切换仍然需要内核调度，仍然有成本。

## 系统调用与上下文切换的区别

系统调用会让 CPU 从用户态进入内核态，执行内核代码，然后返回用户态。这是特权级切换。

但系统调用不一定发生任务上下文切换：

- `getpid()` 这类快速系统调用可能很快返回，同一线程继续执行。
- `read()` 如果数据已在缓冲区，可能很快返回。
- `read()` 如果没有数据，线程可能阻塞，此时才会调度其他线程，发生任务上下文切换。

面试中要避免把“系统调用”等同于“线程切换”。

## Java 中的典型来源

### 线程池过大

大量线程竞争 CPU，时间片切换频繁，`cs` 上升，吞吐反而下降。

### 锁竞争严重

大量线程在同一把锁上阻塞和唤醒，出现 `BLOCKED`、`WAITING`、`futex` 增多。

### 连接池耗尽

线程等待数据库连接、HTTP 连接、Redis 连接，频繁阻塞和唤醒。

### 阻塞 IO

平台线程在 socket read、文件读写、下游 RPC 上阻塞。

### 频繁 sleep 或短周期轮询

用 `sleep(1)` 轮询状态，容易造成无意义唤醒和切换。

### 日志或监控过度同步

日志锁、队列锁、指标上报锁竞争也会引起切换。

## 命令和示例

### vmstat

```bash
vmstat 1
```

关注字段：

- `r`：可运行队列长度。
- `b`：不可中断睡眠任务数，常见于 IO 等待。
- `us`：用户态 CPU。
- `sy`：内核态 CPU。
- `id`：空闲 CPU。
- `wa`：IO wait。
- `cs`：每秒上下文切换次数。
- `in`：每秒中断次数。

判断：

- `r` 长期大于 CPU 核数很多，说明 CPU 竞争明显。
- `cs` 很高且吞吐不高，可能线程过多或锁竞争。
- `sy` 很高，可能系统调用、网络栈、锁、内核开销较大。
- `wa` 高，可能 IO 瓶颈。

### pidstat

查看进程上下文切换：

```bash
pidstat -w -p <pid> 1
```

字段：

- `cswch/s`：自愿上下文切换，通常是主动阻塞。
- `nvcswch/s`：非自愿上下文切换，通常是时间片耗尽或被抢占。

查看线程级：

```bash
pidstat -w -t -p <pid> 1
```

### top

```bash
top
top -H -p <pid>
```

`top -H` 可以看到进程内线程 CPU 分布，配合 `jstack` 定位 Java 线程。

### perf

```bash
perf stat -p <pid> sleep 10
```

可观察 context-switches、cpu-migrations、cache-misses 等指标。

## 线上排查路径

### 上下文切换高

1. 看系统整体：

```bash
vmstat 1
```

2. 找进程：

```bash
pidstat -w 1
```

3. 看线程：

```bash
pidstat -w -t -p <pid> 1
top -H -p <pid>
```

4. 看 Java 栈：

```bash
jstack -l <pid> > /tmp/jstack.txt
```

5. 判断类型：

- `cswch/s` 高：大量阻塞、等待、sleep、IO。
- `nvcswch/s` 高：线程数多、CPU 竞争、时间片抢占。
- 大量 `BLOCKED`：monitor 锁竞争。
- 大量 `WAITING parking`：AQS 锁、线程池、队列或连接池等待。

### 修复方向

- 降低线程池大小，拆分 CPU 和 IO 线程池。
- 减少共享锁，缩短临界区。
- 批量处理，降低频繁唤醒。
- 用阻塞队列替代忙等轮询。
- 设置下游超时和连接池上限。
- 对 IO 密集场景考虑异步化、NIO 或虚拟线程。
- 对 CPU 密集场景优化算法和并行度，而不是盲目加线程。

## 常见追问

### 自愿上下文切换和非自愿上下文切换有什么区别

自愿上下文切换是线程主动让出 CPU，例如等待 IO、锁、条件、sleep。非自愿上下文切换是线程还想继续运行，但时间片用完或被更高优先级任务抢占。

### 上下文切换高一定有问题吗

不一定。高并发 IO 服务天然会有较多切换。要结合 CPU 使用率、吞吐、延迟、线程数、锁竞争和业务基线判断。如果切换高且吞吐下降、延迟上升，就需要排查。

### 为什么锁竞争会导致上下文切换

线程获取锁失败后可能进入阻塞等待，内核或 JVM 会挂起它并运行其他线程。锁释放后等待线程被唤醒，再参与调度和竞争。竞争越激烈，阻塞、唤醒、再竞争越频繁。

### 协程切换为什么轻

协程切换通常在用户态完成，不需要内核调度，不需要切换完整 OS 线程上下文。但它仍要保存协程执行状态，并且最终并行度受承载线程限制。

## 易错点

- 把系统调用、用户态内核态切换、线程上下文切换混为一谈。
- 认为线程数越多吞吐越高。
- 只看 `cs` 数字，不结合 CPU 核数和业务基线。
- 看到 `WAITING` 就认为异常，线程池工作线程空闲等待很正常。
- 忽略锁竞争和连接池等待也会造成大量切换。
- CPU 密集任务使用过大的线程池。

## 自检清单

- 能否解释上下文包含哪些状态。
- 能否区分进程切换、线程切换和系统调用。
- 能否说明上下文切换的直接和间接成本。
- 能否用 `vmstat`、`pidstat -w`、`top -H` 定位切换问题。
- 能否结合 Java 线程 dump 判断锁竞争、IO 等待、线程池等待。
- 能否给出降低上下文切换的工程手段。
