# 流量控制

## 面试定位

流量控制常被拿来和拥塞控制对比。面试官会问：流量控制保护谁、接收窗口怎么工作、零窗口是什么、为什么应用读慢会影响 TCP、线上看到 zero window 怎么排查。

核心回答：

TCP 流量控制通过接收方通告窗口限制发送方发送速度，避免发送方发送过快导致接收方缓冲区溢出。

## 核心概念

流量控制保护的是接收方。

接收方维护接收缓冲区：

```text
接收缓冲区总大小
  = 已收到但应用未读取的数据
  + 剩余可接收空间
```

剩余空间通过 TCP Header 中的窗口字段通告给发送方：

```text
rwnd = receive buffer free space
```

发送方根据 rwnd 控制未确认数据量。

## 工作流程

```text
发送方发送数据
  -> 接收方内核放入接收缓冲区
  -> 应用从 Socket 读取数据
  -> 接收缓冲区释放空间
  -> ACK 中通告新的窗口大小
  -> 发送方根据窗口继续发送
```

如果应用读取慢：

```text
接收缓冲区被占满
  -> rwnd 变小
  -> 发送方减少发送
  -> rwnd = 0 时暂停发送
```

## 零窗口

零窗口表示接收方暂时没有缓冲区空间：

```text
Window size = 0
```

发送方会停止发送普通数据，但会周期性发送零窗口探测，确认窗口是否恢复。

常见触发原因：

- 应用线程阻塞，不读 Socket。
- Java GC 暂停。
- Netty EventLoop 被阻塞。
- 下游处理慢导致上游请求体读取慢。
- 大响应写入时对端消费慢。

## 糊涂窗口综合征

如果接收方每释放很小的空间就通知发送方，发送方每次也只发送很小的数据，会导致大量小包和低效率，这被称为糊涂窗口综合征。

常见优化：

- 接收方避免通告过小窗口。
- 发送方使用 Nagle 算法合并小包。

工程上不用展开太深，知道它是“小窗口小包导致效率低”的问题即可。

## 和拥塞控制的区别

| 对比项 | 流量控制 | 拥塞控制 |
| --- | --- | --- |
| 保护对象 | 接收方 | 网络链路 |
| 控制变量 | 接收窗口 rwnd | 拥塞窗口 cwnd |
| 反馈来源 | 接收方通告 | 丢包、RTT、ACK 节奏 |
| 常见现象 | zero window、window full | 重传、cwnd 降低、吞吐下降 |

实际发送窗口：

```text
min(rwnd, cwnd)
```

## Java 后端典型场景

### 服务端不及时读请求体

例如上传大文件时，服务端业务线程被阻塞，没有持续读取请求体。客户端会看到写入变慢或写超时。

### 客户端不及时读响应体

客户端拿到响应后不消费 body，服务端发送缓冲区可能堆积，最终受对端窗口限制。

### Netty EventLoop 阻塞

如果在 EventLoop 线程执行慢业务、同步 RPC、阻塞 IO，会导致读写事件不能及时处理，引发窗口缩小、连接堆积和延迟抖动。

## 命令示例

抓零窗口：

```bash
sudo tcpdump -i any host 203.0.113.20 and tcp port 8080 -w /tmp/flow.pcap
```

Wireshark 过滤：

```text
tcp.analysis.zero_window
tcp.analysis.zero_window_probe
tcp.analysis.window_update
```

查看连接：

```bash
ss -ti dst 203.0.113.20
```

查看 Java 线程：

```bash
jstack <pid> > /tmp/jstack.txt
```

查看 GC：

```bash
jstat -gcutil <pid> 1000
```

## 线上排查

### 看到 zero window

判断方向：

- 哪一端通告了零窗口，哪一端就是读不动的一方。
- 看该端应用线程是否阻塞。
- 看 GC 是否长暂停。
- 看 CPU 是否打满。
- 看连接对应的请求是否大请求或大响应。

### 写超时

写超时可能不是本端网络发不出去，而是对端不读导致窗口变小。

排查：

- 抓包看对端窗口。
- 看对端应用日志和线程。
- 看是否大 body。
- 看代理或网关是否限速。

### 上传下载慢

可能受：

- 接收窗口。
- 拥塞窗口。
- RTT。
- 带宽。
- 应用读写速度。

需要结合抓包和应用指标，而不是只调 HTTP 超时。

## 常见追问

### 接收窗口由谁决定

由接收方内核根据接收缓冲区可用空间决定，并通过 ACK 通告给发送方。

### 应用层不读数据会发生什么

内核接收缓冲区逐渐填满，接收窗口变小，最终可能变为零窗口，发送方被迫暂停发送。

### 流量控制会不会导致超时

会。如果对端长期不读，本端写入或请求整体耗时可能超过应用配置的写超时、读超时或整体超时。

## 易错点

- 把流量控制说成防止网络拥塞。
- 看到写超时只查发送端，不查接收端是否读慢。
- 认为接收方 ACK 了就代表应用已经处理。ACK 只代表进入接收缓冲区，不代表业务处理完成。
- 忽略 GC 和线程阻塞对 TCP 读写的影响。

## 自检清单

- 能否说明流量控制保护接收方。
- 能否解释接收窗口和接收缓冲区的关系。
- 能否判断 zero window 是哪一方的问题。
- 能否区分写超时、读超时和流量控制现象。
- 能否从 Java 线程、GC、抓包三方面排查读慢。
