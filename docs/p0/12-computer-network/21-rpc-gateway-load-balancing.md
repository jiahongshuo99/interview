# RPC、网关、负载均衡

## 面试定位

RPC、网关、负载均衡是后端系统设计和网络排查的交叉题。面试常问：RPC 和 HTTP 有什么关系、网关做什么、负载均衡有哪些算法、长连接如何影响负载均衡、服务发现和 DNS 有什么区别。

核心回答：

RPC 是服务间调用抽象，底层可以基于 TCP、HTTP/2、HTTP/1.1 或自定义协议；网关负责统一入口治理；负载均衡把请求分配到多个后端实例，并要结合连接、健康检查和故障摘除处理。

## RPC 是什么

RPC，Remote Procedure Call，远程过程调用。它让调用远程服务像调用本地方法一样：

```java
User user = userService.getUser(userId);
```

底层实际包含：

```text
接口代理
  -> 序列化
  -> 协议编码
  -> 服务发现
  -> 负载均衡
  -> 网络传输
  -> 超时和重试
  -> 反序列化
```

常见 RPC 技术：

- gRPC：基于 HTTP/2 和 Protobuf。
- Dubbo：支持多协议和注册中心。
- Thrift：IDL 和多语言 RPC。
- 内部 RPC 框架：常基于 Netty、自定义协议或 HTTP。

## RPC 和 HTTP

HTTP 是应用层协议，RPC 是调用模型。

关系：

- RPC 可以基于 HTTP，例如 gRPC 基于 HTTP/2。
- RPC 也可以基于自定义 TCP 协议。
- REST API 通常直接暴露 HTTP 资源语义。
- RPC 更强调方法调用、IDL、强类型和服务治理。

对比：

| 维度 | HTTP/REST | RPC |
| --- | --- | --- |
| 语义 | 资源和状态转移 | 方法调用 |
| 协议 | HTTP | HTTP/2、自定义 TCP 等 |
| 契约 | OpenAPI、文档 | IDL、接口定义 |
| 治理 | 网关、客户端库 | 注册发现、负载均衡、熔断 |
| 场景 | 开放 API、前后端 | 服务间高频调用 |

## 网关职责

网关是统一入口，常见职责：

- 路由。
- 鉴权。
- 限流。
- 熔断。
- 协议转换。
- TLS 终止。
- 灰度发布。
- 日志和审计。
- 统一错误处理。
- 跨域处理。

典型链路：

```text
Client
  -> DNS
  -> LB
  -> API Gateway
  -> Service
```

网关本身也可能成为瓶颈或故障点。

## 负载均衡层次

常见层次：

| 层次 | 示例 | 特点 |
| --- | --- | --- |
| DNS 负载均衡 | 多 A 记录、GSLB | 简单，但缓存影响故障切换 |
| 四层负载均衡 | LVS、NLB | 基于 IP、端口、TCP/UDP |
| 七层负载均衡 | Nginx、Envoy、API Gateway | 基于 HTTP Header、路径、权重 |
| 客户端负载均衡 | RPC SDK、服务发现客户端 | 调用方直接选择实例 |

## 常见算法

### 轮询

请求按顺序分配到实例。

优点：简单。

缺点：不考虑实例性能和连接状态。

### 加权轮询

按权重分配。

适合实例规格不同或灰度流量控制。

### 随机

随机选择实例，流量大时分布接近均匀。

### 最少连接

选择当前连接数较少的实例。

适合长连接或请求耗时差异较大的场景，但连接数不等于负载。

### 一致性哈希

同一 key 尽量路由到同一实例。

适合：

- 缓存命中。
- 有状态会话。
- 分片。

风险：

- 热点 key。
- 节点变更带来迁移。

### P2C

Power of Two Choices，随机选两个实例，再选负载较低的一个。

在大规模服务中常用于平衡效果和成本。

## 健康检查

负载均衡必须配合健康检查：

- TCP 端口检查。
- HTTP `/health` 检查。
- 应用自定义健康检查。
- 主动探测和被动异常摘除。

健康检查要注意：

- 端口通不代表业务健康。
- 健康检查过重会影响服务。
- 检查周期过长会导致故障摘除慢。
- 发布时要支持优雅上下线。

## 长连接对负载均衡的影响

长连接会让流量绑定在连接建立时选中的实例上。

影响：

- 新实例上线后流量进入慢。
- 老实例下线仍有连接。
- HTTP/2 单连接承载大量请求，负载更容易不均。
- 客户端连接池可能绕过 DNS 新结果。

治理：

- 连接最大生命周期。
- 连接排水。
- 优雅下线。
- 客户端感知实例变更。
- 限制单连接最大并发流。

## 超时、重试、熔断在 RPC 中的位置

RPC 框架通常内置：

- 连接池。
- 请求超时。
- 重试。
- 熔断。
- 限流。
- 负载均衡。
- 服务发现。

风险：

- 框架重试和业务重试叠加。
- 客户端超时比服务端处理短，造成后台无效工作。
- 熔断维度太粗，影响无故障接口。
- 负载均衡没有及时摘除慢实例。

## 命令示例

查看网关响应和 upstream 耗时，取决于日志格式。

curl 指定 Host：

```bash
curl -v -H 'Host: api.example.com' http://1.2.3.4/path
```

HTTPS 指定 IP 但保留域名和 SNI：

```bash
curl -v --resolve api.example.com:443:1.2.3.4 https://api.example.com/path
```

查看连接分布：

```bash
ss -antp | awk '{print $5}' | sort | uniq -c | sort -nr | head
```

测试端口：

```bash
nc -vz 203.0.113.20 8080
```

## 线上排查

### 某个实例流量异常高

可能原因：

- 长连接粘住。
- 权重配置错误。
- 一致性哈希热点。
- 客户端缓存服务列表过旧。
- HTTP/2 单连接复用过多。

### 网关 502

排查：

- upstream 是否可连。
- TLS 或协议是否匹配。
- 上游是否返回 RST。
- 上游是否端口未监听。
- 网关连接池是否异常。

### 网关 503

排查：

- 是否无健康实例。
- 是否限流、熔断。
- 是否发布摘除。
- 是否网关自身过载。

### 网关 504

排查：

- upstream 处理是否超过网关超时。
- 上游线程池或连接池是否满。
- 网络是否重传。
- 网关 timeout 是否过短。

## 常见追问

### 四层和七层负载均衡区别

四层基于 IP、端口和连接转发，不理解 HTTP 语义。七层理解应用协议，可按域名、路径、Header、Cookie 路由。

### 服务发现和 DNS 区别

DNS 是通用域名解析，受缓存和 TTL 影响。服务发现通常面向服务实例列表，支持健康检查、变更推送、元数据和客户端负载均衡。

### RPC 一定比 HTTP 快吗

不一定。RPC 常通过二进制协议、长连接、连接池和 IDL 获得性能优势，但最终取决于协议、序列化、网络、框架实现和业务处理。

## 易错点

- 认为 RPC 是一种固定网络协议。
- 认为端口健康就代表服务健康。
- 忽略长连接导致负载不均。
- 多层重试叠加。
- 网关 502、503、504 不区分就查应用日志。
