# 重传机制

## 面试定位

重传机制是 TCP 可靠传输的关键。面试常问：TCP 如何判断丢包、超时重传和快速重传区别是什么、RTO 怎么估算、重复 ACK 是什么、SACK 有什么用、重传对线上延迟有什么影响。

回答时要强调：重传不是免费恢复，重传通常意味着延迟升高、吞吐下降、拥塞控制收缩窗口。

## 为什么需要重传

IP 网络是不可靠的，TCP 段可能：

- 丢失。
- 乱序。
- 被校验失败后丢弃。
- ACK 丢失。
- 中间设备拥塞丢包。

TCP 通过重传补齐缺失字节，让接收方最终获得连续有序字节流。

## 超时重传

发送方发送数据后会启动定时器。如果在重传超时时间 RTO 内没有收到确认，就认为可能丢包并重发。

简化流程：

```text
发送 seq=1000,len=100
  -> 等待 ACK 1100
  -> RTO 到期仍未收到
  -> 重传 seq=1000,len=100
```

RTO 不是固定值，会根据 RTT 动态估算。网络抖动越大，RTO 通常越保守。

## RTT、SRTT、RTO

概念：

- RTT：一次发送到收到 ACK 的往返时间。
- SRTT：平滑后的 RTT。
- RTTVAR：RTT 波动。
- RTO：重传超时时间。

直观理解：

```text
RTO = 平滑 RTT + 波动补偿
```

如果 RTO 太短，会误判丢包导致不必要重传。如果 RTO 太长，真实丢包恢复太慢。

## 快速重传

超时重传要等 RTO，可能较慢。快速重传利用重复 ACK 提前判断丢包。

例子：

```text
发送：1000-1099、1100-1199、1200-1299、1300-1399
丢失：1100-1199
接收方收到 1200-1299、1300-1399，但缺 1100
接收方持续 ACK 1100
发送方收到多个重复 ACK 1100
触发快速重传 1100-1199
```

重复 ACK 表示接收方仍在等待某个缺口。

## SACK

SACK 是 Selective Acknowledgment，选择确认。它允许接收方告诉发送方哪些非连续数据块已经收到。

没有 SACK 时，发送方可能不知道哪些后续段已到，只能保守重传。

有 SACK 时，接收方可以表达：

```text
累计 ACK 1100
SACK: 1200-1400 已收到
```

发送方就可以优先重传 1100-1199，而不是重复发送已经到达的数据。

查看是否开启：

```bash
sysctl net.ipv4.tcp_sack
```

## 重传和拥塞控制

丢包常被视为网络拥塞信号。发生重传后，TCP 拥塞控制通常会降低发送窗口：

- 超时重传通常更严重，拥塞窗口大幅下降。
- 快速重传配合快速恢复，下降相对温和。

线上表现：

- P99 延迟升高。
- 吞吐下降。
- 同连接请求排队。
- HTTP/2 单连接多路复用下多个流一起变慢。

## 重传不一定是请求失败

少量重传可能被 TCP 自动恢复，应用无感，只表现为耗时略高。大量重传才会导致明显超时、断连或吞吐下降。

因此排查时要看：

- 重传率。
- RTT。
- 请求耗时分位数。
- 是否集中在某个机房、节点、网卡或下游。

## 命令示例

查看 TCP 连接细节：

```bash
ss -ti dst 203.0.113.20
```

输出中可能包含：

```text
rtt:20.1/5.2 rto:208 retrans:1/3 cwnd:10
```

含义：

- `rtt`：平滑 RTT 和波动。
- `rto`：重传超时。
- `retrans`：重传相关统计。
- `cwnd`：拥塞窗口。

抓重传包：

```bash
sudo tcpdump -i any host 203.0.113.20 and tcp port 8080 -w /tmp/retrans.pcap
```

Wireshark 过滤：

```text
tcp.analysis.retransmission
tcp.analysis.fast_retransmission
tcp.analysis.duplicate_ack
```

模拟网络丢包：

```bash
sudo tc qdisc add dev eth0 root netem loss 3% delay 50ms
sudo tc qdisc del dev eth0 root
```

查看网卡丢包：

```bash
ip -s link
ethtool -S eth0
```

## 线上排查

### 请求超时但服务端处理很快

可能是网络重传导致客户端等待响应慢。

排查：

- 客户端和服务端同时看日志时间。
- 在客户端侧抓包看响应是否迟到。
- 看链路 RTT 和重传。
- 检查 LB、网关、跨机房链路。

### 偶发长尾延迟

可能原因：

- 单个 TCP 包丢失触发 RTO。
- HTTP/2 连接上某个包丢失影响多个流。
- 接收端窗口变小导致发送受限。

治理：

- 降低单连接承载的过高并发。
- 合理配置连接池和 HTTP/2 最大并发流。
- 优化网络链路和跨机房调用。
- 对幂等请求设置合理超时和重试。

### 大量重传

优先检查：

- 是否某台机器网卡异常。
- 是否某个交换机、宿主机、容器节点有丢包。
- 是否 MTU 配置不一致。
- 是否链路拥塞。
- 是否接收端 CPU 高导致处理 ACK 变慢。

## 常见追问

### ACK 丢了会怎样

如果数据已到但 ACK 丢了，发送方可能超时重传。接收方识别重复数据后丢弃，并再次 ACK。

### 快速重传一定需要三个重复 ACK 吗

经典描述是三个重复 ACK 触发快速重传。具体实现可能有优化，但面试回答按经典机制即可。

### 重传会导致应用收到重复数据吗

不会。TCP 根据序号去重，应用看到的是有序字节流。

### 重传是否说明服务端慢

不一定。重传是传输层现象，可能是网络丢包、链路拥塞、接收端处理慢，也可能和服务端业务逻辑无关。

## 易错点

- 把超时重传和业务读超时混为一谈。
- 认为快速重传不影响吞吐。它通常仍会触发拥塞窗口调整。
- 只看服务端耗时，不看网络重传导致的客户端耗时。
- 忽略 ACK 丢失也会引发重传。
- 看到一次重传就判断网络故障，需要结合比例和影响。

## 自检清单

- 能否说明超时重传和快速重传的区别。
- 能否解释重复 ACK 和 SACK。
- 能否说明 RTO 与 RTT 的关系。
- 能否用 `ss -ti` 和抓包看重传。
- 能否解释重传对延迟和吞吐的影响。
