# HTTP 常见状态码

## 面试定位

HTTP 状态码是后端接口设计、网关排查和线上定位的基础。面试常问：301 和 302 区别、401 和 403 区别、502/503/504 怎么区分、哪些请求可以重试、状态码应该由网关还是应用返回。

回答时要结合语义、调用方动作和排查方向，而不是只背数字。

## 分类

| 范围 | 类别 | 含义 |
| --- | --- | --- |
| 1xx | 信息响应 | 请求已收到，继续处理 |
| 2xx | 成功 | 请求成功接收、理解并处理 |
| 3xx | 重定向 | 需要进一步操作完成请求 |
| 4xx | 客户端错误 | 请求有问题或权限不足 |
| 5xx | 服务端错误 | 服务端或网关处理失败 |

## 2xx

### 200 OK

请求成功，最常见。

适用：

- 查询成功。
- 普通同步操作成功。

### 201 Created

资源创建成功。

适用：

- 创建订单。
- 创建用户。
- 返回 Location 指向新资源。

### 202 Accepted

请求已接受，但尚未处理完成。

适用：

- 异步任务提交。
- 批处理导入。
- AI Agent 长任务排队。

### 204 No Content

处理成功，但没有响应体。

适用：

- 删除成功。
- 更新成功但无需返回内容。

## 3xx

### 301 Moved Permanently

永久重定向。客户端和搜索引擎可缓存新地址。

### 302 Found

临时重定向。历史上很多客户端会把 POST 改成 GET，使用时要谨慎。

### 307 Temporary Redirect

临时重定向，但保持原请求方法和 Body。

### 308 Permanent Redirect

永久重定向，同时保持原请求方法和 Body。

工程建议：

- 页面跳转常见 301/302。
- API 重定向如需保持方法，优先考虑 307/308。

## 4xx

### 400 Bad Request

请求格式错误或参数非法。

例子：

- JSON 解析失败。
- 参数类型错误。
- 必填字段缺失。

### 401 Unauthorized

未认证或认证失败。

例子：

- 未带 token。
- token 过期。
- 签名错误。

### 403 Forbidden

已认证，但没有权限。

例子：

- 用户无资源访问权限。
- IP 不在白名单。
- 账号被禁用。

### 404 Not Found

资源不存在或路由不存在。

排查：

- URL 是否正确。
- 网关路由是否配置。
- 服务是否发布对应接口。

### 405 Method Not Allowed

请求方法不允许。

例子：

- 接口只支持 POST，却用 GET。

### 408 Request Timeout

服务端等待客户端请求超时。与客户端读超时不同，408 是 HTTP 响应状态码。

### 409 Conflict

资源状态冲突。

适用：

- 版本号冲突。
- 幂等键冲突。
- 重复创建。

### 410 Gone

资源曾经存在，但现在永久删除。

### 413 Payload Too Large

请求体过大。

常见于：

- 上传文件超过 Nginx 或网关限制。
- JSON body 太大。

### 415 Unsupported Media Type

Content-Type 不支持。

### 422 Unprocessable Entity

语义校验失败。部分 API 用它表达字段业务校验不通过。

### 429 Too Many Requests

请求过多，被限流。

调用方动作：

- 按 Retry-After 等待。
- 指数退避。
- 降低并发。

## 5xx

### 500 Internal Server Error

服务端内部错误。通常是应用异常或未分类错误。

### 501 Not Implemented

服务端不支持请求功能。

### 502 Bad Gateway

网关作为代理访问上游失败，收到无效响应或连接失败。

常见原因：

- 上游连接拒绝。
- 上游连接重置。
- 上游返回非法协议。
- 网关和上游 TLS 或 HTTP 版本不兼容。

### 503 Service Unavailable

服务暂不可用。

常见原因：

- 服务过载。
- 熔断降级。
- 发布重启。
- 网关无可用实例。

### 504 Gateway Timeout

网关等待上游响应超时。

常见原因：

- 上游处理慢。
- 上游线程池满。
- 网络慢。
- 网关 upstream timeout 过短。

## 502、503、504 区分

| 状态码 | 重点含义 | 排查方向 |
| --- | --- | --- |
| 502 | 网关连上游或解析上游响应失败 | 上游端口、RST、协议、TLS |
| 503 | 服务不可用或被保护 | 实例健康、限流、熔断、发布 |
| 504 | 网关等上游超时 | 上游耗时、线程池、网络、超时配置 |

## 重试策略

通常可考虑重试：

- 408。
- 429，遵守 `Retry-After`。
- 502。
- 503。
- 504。
- 网络连接失败。

但必须结合幂等性：

- GET 通常可重试。
- PUT、DELETE 视语义和幂等设计。
- POST 默认不安全，除非有幂等键或业务保证。

## 命令示例

查看状态码：

```bash
curl -i https://example.com/api
curl -s -o /dev/null -w '%{http_code} %{time_total}\n' https://example.com/api
```

查看重定向：

```bash
curl -IL https://example.com/
```

看详细过程：

```bash
curl -v https://example.com/api
```

## 线上排查

### 4xx 激增

优先看：

- 是否发布了参数校验变更。
- 前端或调用方是否改了请求格式。
- 鉴权服务是否异常。
- 网关路由是否变化。
- 是否限流触发 429。

### 5xx 激增

优先看：

- 应用异常日志。
- 线程池、连接池、数据库连接池。
- 下游错误率。
- 网关 upstream 日志。
- 发布和扩缩容事件。

### 504 增多

重点看耗时拆分：

- 网关等待上游多久。
- 上游是否收到请求。
- 上游处理到哪一步。
- 客户端超时是否短于网关或服务端超时。

## 常见追问

### 401 和 403 区别

401 是未认证或认证失败；403 是已认证但无权限。

### 301 和 302 区别

301 是永久重定向，302 是临时重定向。API 场景要注意方法是否被客户端改写，必要时使用 307/308。

### 业务错误应该用 200 还是 4xx

取决于 API 规范。参数错误、权限错误更适合 4xx；业务可预期失败有些团队用 200 加业务码。关键是约定一致，便于监控和调用方处理。

## 易错点

- 把客户端读超时说成 408。读超时是客户端异常，408 是服务端返回状态码。
- 看到 502 就直接查应用异常，忽略网关到上游连接。
- 认为所有 5xx 都可以无脑重试。
- 401 和 403 混用。
- API 重定向用 302 导致 POST 变 GET。

## 自检清单

- 能否按 2xx、3xx、4xx、5xx 解释常见状态码。
- 能否区分 401/403、301/302/307/308、502/503/504。
- 能否说明哪些状态适合重试以及幂等前提。
- 能否用 curl 快速查看状态码和耗时。
- 能否根据状态码给出网关和应用排查方向。
