# 滑动窗口

## 面试定位

滑动窗口是 TCP 高吞吐的基础，也是理解流量控制和拥塞控制的前置知识。面试常问：为什么不能发一个包等一个 ACK、滑动窗口怎么滑、发送窗口和接收窗口是什么、窗口大小受哪些因素影响。

一句话回答：

滑动窗口允许发送方在未收到全部 ACK 前连续发送多个 TCP 段，从而提升链路利用率，同时通过窗口边界控制未确认数据量。

## 为什么需要滑动窗口

如果每发送一段数据都等待 ACK：

```text
send segment 1
wait ack 1
send segment 2
wait ack 2
```

高 RTT 链路吞吐会很低。滑动窗口允许：

```text
send segment 1
send segment 2
send segment 3
...
wait ack and slide window
```

这样可以让网络中始终有一定数量的数据在传输。

## 发送窗口

发送方维护几个区域：

```text
已确认 | 已发送未确认 | 可发送未发送 | 不可发送
```

窗口右移依赖 ACK：

```text
收到 ack = 5000
表示 5000 之前的字节都已确认
发送窗口左边界移动到 5000
```

窗口越大，最多可在网络中飞行的未确认数据越多，吞吐潜力越大。

## 接收窗口

接收方通过 TCP Header 中的窗口字段告诉发送方自己还能接收多少数据。

接收窗口由接收缓冲区决定：

```text
接收窗口 = 接收缓冲区剩余空间
```

如果应用读取慢，接收缓冲区被占满，接收窗口会变小，甚至变为 0。

## 实际发送窗口

发送方不是只看接收窗口，还要看拥塞窗口：

```text
实际可发送窗口 = min(接收窗口 rwnd, 拥塞窗口 cwnd)
```

- rwnd 来自接收方，防止压垮对端。
- cwnd 来自发送方拥塞控制，防止压垮网络。

## 窗口滑动示例

假设发送窗口大小为 400 字节，起始序号 1000：

```text
可发送范围：1000-1399
发送：1000-1099、1100-1199、1200-1299、1300-1399
收到 ACK 1200
窗口左边界移动到 1200
新的可发送范围：1200-1599
```

已确认数据从窗口移出，新的字节进入可发送范围。

## 窗口大小和吞吐

链路最大吞吐受带宽时延积影响：

```text
BDP = bandwidth * RTT
```

如果窗口小于 BDP，就无法充分利用带宽。

例子：

```text
带宽 100Mbps
RTT 100ms
BDP = 100Mbps * 0.1s = 10Mb = 1.25MB
```

窗口至少接近 1.25MB 才能较好填满链路。

## 窗口扩大

TCP Header 原始窗口字段只有 16 位，最大 65535 字节。现代 TCP 使用 Window Scale 选项扩大窗口。

抓包中可看到：

```text
Window size value
Window scale
Calculated window size
```

面试中知道有窗口扩大机制即可，不必背具体实现细节。

## 零窗口

当接收方缓冲区满时，会通告窗口为 0：

```text
rwnd = 0
```

发送方暂停发送，并通过零窗口探测确认窗口是否恢复。

常见原因：

- 接收方应用读数据慢。
- 接收方线程阻塞。
- GC 或 STW 导致应用暂停读取。
- 服务端处理慢，Socket 缓冲区积压。

## 命令示例

查看连接窗口和拥塞信息：

```bash
ss -ti dst 203.0.113.20
```

抓包看窗口：

```bash
sudo tcpdump -i any host 203.0.113.20 and tcp port 8080 -w /tmp/window.pcap
```

Wireshark 常用过滤：

```text
tcp.analysis.zero_window
tcp.analysis.window_full
tcp.analysis.window_update
```

查看缓冲区参数：

```bash
sysctl net.ipv4.tcp_rmem
sysctl net.ipv4.tcp_wmem
```

## Java 后端关联

滑动窗口问题在线上常表现为：

- 大响应下载慢。
- 上传大文件慢。
- 服务端 CPU 不高但吞吐低。
- 某些客户端链路 RTT 高时速度差。
- 接收端 GC 后出现窗口变小或零窗口。

应用层关注点：

- 不要在 I/O 线程做慢业务逻辑。
- 大响应要合理流式处理，避免阻塞读写。
- Netty EventLoop 不要被业务代码阻塞。
- HTTP Client 连接池不要让单连接承载过大并发。

## 常见追问

### 滑动窗口解决了什么问题

解决停等协议吞吐低的问题，让发送方可以连续发送多个未确认段，提高链路利用率。

### 滑动窗口是否等同于流量控制

不是。滑动窗口是一种机制，流量控制使用接收窗口来限制发送方。拥塞控制也通过拥塞窗口影响实际发送窗口。

### 窗口越大越好吗

不是。窗口过小吞吐不足，窗口过大可能造成缓冲区膨胀、延迟升高，且实际还受拥塞控制限制。

### 零窗口是不是网络丢包

不一定。零窗口更多说明接收方没有足够缓冲区，常见根因是应用读取慢或暂停。

## 易错点

- 把窗口理解成“包个数”，实际是字节范围。
- 只讲接收窗口，不讲拥塞窗口。
- 认为窗口满一定是发送方问题，实际可能是接收方读慢。
- 不知道高 RTT 链路需要更大的窗口才能打满带宽。
- 把 TCP 滑动窗口和算法题滑动窗口混为一谈。
