# 熔断、限流、降级

## 面试定位

熔断、限流、降级是分布式系统高可用治理的核心组合。面试官通常会用“下游变慢怎么办”“大促流量突增怎么办”“非核心功能挂了怎么办”来考察你是否理解故障隔离和资源保护。

常见考点：

- 熔断、限流、降级分别解决什么问题。
- 熔断器关闭、打开、半开的状态转换。
- 限流算法：固定窗口、滑动窗口、漏桶、令牌桶。
- 单机限流和分布式限流的区别。
- 降级数据从哪里来。
- 如何避免雪崩。
- Java 后端如何结合线程池隔离、超时和监控。

回答时要强调：这三者目标都是保护系统，但触发条件和作用位置不同。

## 三者区别

| 能力 | 解决问题 | 典型触发 | 动作 |
| --- | --- | --- | --- |
| 熔断 | 下游故障或变慢拖垮上游 | 错误率、超时率、慢调用比例升高 | 暂停调用下游，快速失败或降级 |
| 限流 | 流量超过系统承载能力 | QPS、并发数、令牌不足 | 拒绝、排队、削峰 |
| 降级 | 非核心能力不可用或成本过高 | 故障、过载、开关、熔断 | 返回兜底结果或关闭功能 |

一句话：

```text
限流挡住过多入口流量，熔断切断不健康依赖，降级保证核心链路继续可用。
```

## 熔断

### 为什么需要熔断

没有熔断时，下游故障会导致：

- 上游线程阻塞等待。
- 连接池耗尽。
- 重试放大流量。
- 调用链路层层堆积。
- 整体雪崩。

熔断的核心是：当失败达到阈值时，短时间不再调用下游，让上游快速失败，也给下游恢复时间。

### 熔断器状态

```text
CLOSED 关闭：正常调用
  -> 错误率或慢调用比例超过阈值
OPEN 打开：拒绝调用，直接失败或降级
  -> 休眠窗口结束
HALF_OPEN 半开：放少量探测请求
  -> 探测成功，回到 CLOSED
  -> 探测失败，回到 OPEN
```

### 熔断指标

常用触发条件：

- 错误率。
- 超时率。
- 慢调用比例。
- 连续失败次数。
- 最小请求数。
- 滑动窗口统计。

为什么需要最小请求数：

- 只有 1 个请求失败就 100% 错误率，直接熔断会误判。
- 必须达到一定请求量后错误率才有统计意义。

### 服务级和实例级熔断

服务级熔断：

- 整个下游服务不可用。
- 所有实例都不再调用。
- 适合下游整体故障。

实例级熔断：

- 只隔离某个异常 Provider。
- 负载均衡避开该实例。
- 适合单机故障、单实例 GC、网络异常。

工程上两者可以同时存在。

## 限流

### 限流目标

限流不是为了让请求失败，而是为了让系统在过载时有控制地失败，保护核心资源。

限流保护：

- CPU。
- 内存。
- 线程池。
- 数据库连接。
- 下游服务。
- 核心业务链路。

### 固定窗口

按固定时间窗口计数，例如每秒最多 1000 次。

优点：

- 实现简单。

缺点：

- 窗口边界可能出现双倍流量。

示例：

```text
00:00:00.900 - 00:00:00.999 通过 1000 次
00:00:01.000 - 00:00:01.100 又通过 1000 次
短时间内实际通过 2000 次
```

### 滑动窗口

把时间窗口拆成多个小格，随时间滑动统计。

优点：

- 比固定窗口平滑。
- 边界突刺更小。

缺点：

- 实现复杂度更高。
- 需要维护多个时间片。

### 漏桶

请求进入桶，以固定速率流出。

特点：

- 输出平滑。
- 超过桶容量的请求被拒绝。
- 对突发流量不友好。

适合需要稳定处理速率的场景。

### 令牌桶

系统按固定速率生成令牌，请求拿到令牌才能通过。

特点：

- 支持一定突发流量。
- 平均速率受令牌生成速度限制。
- 桶容量决定可承接的突发量。

适合大多数接口限流。

### 并发数限流

限制同时执行的请求数。

相比 QPS 限流，并发数限流更能保护线程池和下游资源。

适合：

- 慢请求。
- DB 查询。
- 下游 RPC。
- 文件处理。

公式直觉：

```text
并发数 ≈ QPS * 平均耗时
```

平均耗时上升时，即使 QPS 不变，并发数也会升高。

## 单机限流和分布式限流

### 单机限流

每个实例自己限流。

优点：

- 性能高。
- 不依赖外部组件。
- 实现简单。

缺点：

- 总限流值和实例数相关。
- 流量分配不均时可能局部过载。

适合大部分服务自身保护。

### 分布式限流

多个实例共享一个全局限流额度，常用 Redis、网关或专门限流服务。

优点：

- 能控制全局 QPS。
- 适合租户、用户、接口级配额。

缺点：

- 有额外网络开销。
- 限流组件可能成为瓶颈。
- 要考虑 Redis 故障时的策略。

常见做法：

- 网关做全局粗限流。
- 服务内做单机细粒度限流。
- 核心资源再做并发数保护。

## 降级

### 降级类型

常见降级：

- 返回默认值。
- 返回缓存值。
- 返回静态配置。
- 关闭非核心功能。
- 延迟处理。
- 只保留核心字段。
- 排队异步处理。

### 降级数据来源

可以来自：

- 本地缓存。
- Redis 缓存。
- 配置中心。
- 上一次成功结果。
- 静态默认值。
- 兜底服务。

降级数据要明确时效性，不能让过期数据造成严重业务错误。

### 降级开关

降级通常需要配置开关：

- 按接口开关。
- 按用户或租户开关。
- 按城市、渠道、版本开关。
- 按比例开关。
- 自动触发和人工触发结合。

开关系统要支持快速生效、审计和回滚。

## 工程组合

### 稳定性保护顺序

典型链路：

```text
入口请求
  -> 网关限流
  -> 参数校验
  -> 用户或租户限流
  -> 业务线程池隔离
  -> 下游 RPC 超时
  -> 下游实例级熔断
  -> 服务级熔断
  -> 降级返回
```

### 线程池隔离

不同下游或不同业务使用独立线程池，避免互相拖垮。

例如：

- 订单核心链路线程池。
- 推荐服务调用线程池。
- 通知服务调用线程池。
- 报表导出线程池。

非核心下游变慢时，不应耗尽核心业务线程。

### 和超时重试的关系

- 超时：限制等待时间。
- 重试：处理短暂失败。
- 熔断：失败过多时停止调用。
- 限流：流量过大时拒绝或排队。
- 降级：保证核心流程可继续。

重试必须受熔断和限流约束，否则会放大故障。

## 典型场景

### 下游推荐服务变慢

处理：

- 推荐调用设置短超时。
- 慢调用比例升高后熔断。
- 降级返回默认推荐或热门列表。
- 核心交易链路不被推荐拖慢。

### 大促入口流量突增

处理：

- 网关按接口限流。
- 用户维度防刷限流。
- 核心接口优先保障。
- 非核心功能降级。
- DB 写入削峰。

### 某个租户异常刷接口

处理：

- 租户维度分布式限流。
- 触发后只影响该租户。
- 告警并保留审计日志。

## 线上风险

### 熔断阈值太敏感

少量请求失败就熔断，会导致可用服务被误切。

解决：

- 设置最小请求数。
- 使用滑动窗口。
- 区分错误类型。
- 半开探测。

### 限流值配置过大

限流值超过系统真实容量，等于没有保护。

要根据：

- 压测结果。
- 线程池容量。
- DB 连接池容量。
- 下游限额。
- 业务峰值。

### 限流值配置过小

会误伤正常请求，影响业务转化。

需要：

- 监控被限流请求数。
- 区分核心和非核心流量。
- 支持动态调整。

### 降级结果不可信

降级不是随便返回假数据。

风险：

- 金额、库存、权限等关键字段不能随意默认。
- 过期缓存可能导致错误决策。
- 降级状态没有对用户说明。

### 降级依赖同一个故障源

如果降级数据仍依赖故障下游，降级无效。

例如下游服务挂了，降级逻辑还实时调用同一个服务查询默认值。

## 常见追问

### 熔断和限流有什么区别

可以这样答：

```text
限流主要处理流量超过系统容量的问题，通常在入口或资源访问前控制请求数量；熔断主要处理下游故障或慢调用问题，当失败率达到阈值时停止继续调用下游。限流是挡流量，熔断是切故障依赖。
```

### 限流算法怎么选

- 简单接口 QPS 控制：滑动窗口或令牌桶。
- 允许突发：令牌桶。
- 要求稳定速率：漏桶。
- 保护慢资源：并发数限流。
- 全局配额：分布式限流。

### 熔断后什么时候恢复

熔断打开后等待一段休眠窗口，进入半开状态，放少量探测请求。如果探测成功率达标，恢复关闭；如果失败，重新打开。

### 降级是不是直接返回空

不是。降级要符合业务语义。非核心推荐可以返回空或热门列表，但库存、余额、权限、支付结果等核心数据不能随便默认成功。

## 易错点

- 把熔断、限流、降级混为一谈。
- 只有限流没有降级提示和监控。
- 熔断统计没有最小请求数。
- 所有异常都计入熔断，包括业务参数错误。
- 令牌桶容量过大导致突刺打垮下游。
- 分布式限流依赖 Redis，却没有 Redis 故障策略。
- 降级数据没有时效性说明。
- 非核心下游和核心链路共用线程池。
