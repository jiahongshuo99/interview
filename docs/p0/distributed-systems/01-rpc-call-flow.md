# RPC 调用链路

## 面试定位

RPC 调用链路是分布式系统面试的入口题。面试官通常不是只想听“像调用本地方法一样调用远程服务”，而是希望你能讲清楚一次 Java 后端 RPC 请求从客户端代理到服务端执行再回到客户端的完整路径，以及链路上每个组件解决什么问题。

常见考点：

- RPC 和 HTTP REST 的区别。
- 动态代理、序列化、网络传输、连接池、线程模型分别在哪一层。
- 服务发现、负载均衡、超时、重试、熔断、限流如何串起来。
- 请求如何关联 traceId、日志、监控和异常。
- RPC 调用为什么会出现超时、重复执行、雪崩和线程池耗尽。

回答时建议先给主链路，再拆机制，最后讲工程治理。

## 总体链路

一次典型 RPC 调用：

```text
业务代码调用接口方法
  -> 客户端动态代理
  -> 构造 RPC 请求
  -> 服务发现获取实例列表
  -> 负载均衡选择实例
  -> 连接池获取连接
  -> 编码和序列化
  -> 网络发送
  -> 服务端解码和反序列化
  -> 服务端线程池分发
  -> 反射或方法句柄调用目标方法
  -> 序列化响应
  -> 网络返回
  -> 客户端解码
  -> Future/Promise 唤醒调用线程
  -> 返回结果或抛异常
```

可以用一句话概括：

```text
RPC 框架把本地接口调用转换成网络请求，客户端负责代理、寻址、编码、发送和容错，服务端负责解码、分发、执行和响应，中间通过注册中心、连接池、线程池和治理能力保证可用性。
```

## 核心概念

### 调用方和提供方

- Consumer：服务调用方。
- Provider：服务提供方。
- Registry：注册中心，维护服务名到实例列表的映射。
- Protocol：传输协议和编码格式。
- Stub/Proxy：客户端代理对象。
- Invoker：框架内部对可调用远程节点的抽象。

### RPC 请求包含什么

一个 RPC 请求通常包含：

| 字段 | 作用 |
| --- | --- |
| serviceName | 服务接口名或服务标识 |
| methodName | 目标方法名 |
| parameterTypes | 参数类型，用于定位重载方法 |
| arguments | 参数值 |
| requestId | 请求唯一标识，用于响应匹配 |
| version/group | 服务版本、分组或环境隔离 |
| timeout | 调用超时时间 |
| attachments | traceId、用户信息、灰度标签等上下文 |

### RPC 响应包含什么

常见响应结构：

- requestId：对应请求。
- statusCode：成功、业务异常、框架异常、超时等状态。
- result：返回值。
- exception：异常信息。
- attachments：服务端透传的上下文或调试信息。

## 客户端底层机制

### 动态代理

业务代码通常只依赖接口：

```java
public interface OrderService {
    OrderDTO getOrder(Long orderId);
}
```

注入到业务代码里的不是本地实现类，而是代理对象。代理拦截方法调用，把方法调用转换成 RPC 请求。

常见代理方式：

- JDK 动态代理：基于接口。
- CGLIB：基于子类。
- ByteBuddy/Javassist：字节码增强。

面试重点：

- 代理对象屏蔽远程调用细节。
- 方法签名用于构造服务名、方法名、参数类型。
- 代理层通常也是过滤器链入口，处理 trace、监控、鉴权、超时和重试。

### 服务发现

客户端根据服务名从本地缓存或注册中心拿到 Provider 列表。

通常不会每次调用都远程查注册中心，而是：

- 启动时拉取全量或订阅目标服务。
- 注册中心变更时推送实例列表。
- 客户端保留本地缓存。
- 注册中心短时不可用时继续使用旧缓存。

### 负载均衡

拿到实例列表后，客户端需要选择一个实例。

常见输入：

- 实例权重。
- 连接状态。
- 最近失败次数。
- 响应耗时。
- 一致性哈希 key。
- 灰度标签和机房标签。

负载均衡不是孤立策略，它要和服务发现、熔断、重试、路由规则一起工作。

### 连接池和网络发送

RPC 框架通常使用长连接，而不是每次调用都创建 TCP 连接。

客户端会维护：

- 每个 Provider 的连接。
- 连接健康状态。
- 最大连接数。
- 心跳和空闲检测。
- 写队列和 pending 请求表。

在 Netty 这类 NIO 框架中，一个连接可以承载多个未完成请求，通过 requestId 匹配响应。

### 序列化和编码

序列化把 Java 对象转换成字节。

编码通常还会加协议头：

```text
magic
version
serializerType
messageType
requestId
bodyLength
body
```

协议头的作用：

- 快速识别协议。
- 判断消息类型。
- 知道 body 长度，解决粘包拆包。
- 根据 requestId 匹配响应。
- 支持不同序列化方式。

### 异步转同步

底层网络通常是异步的，但业务接口可能是同步返回。

典型做法：

```text
发送请求
  -> 创建 CompletableFuture
  -> 放入 pendingMap，key 为 requestId
  -> I/O 线程收到响应
  -> 根据 requestId 找到 future
  -> complete(result)
  -> 调用线程被唤醒
```

如果超过超时时间还没有响应：

- future 超时完成。
- pendingMap 清理 requestId。
- 调用方收到 TimeoutException。
- 服务端可能仍在执行，所以要考虑幂等。

## 服务端底层机制

### 服务暴露

服务端启动时会：

- 扫描 RPC 服务注解或配置。
- 建立 serviceName 到实现类的映射。
- 启动网络监听端口。
- 向注册中心注册实例信息。
- 定期发送心跳或续约。

服务暴露不仅是“开放端口”，还包括版本、分组、权重、协议、方法元数据和健康状态。

### 解码和反序列化

服务端收到字节流后：

- 按协议拆包。
- 读取 requestId、服务名、方法名、参数类型。
- 反序列化参数。
- 构造框架内部请求对象。

风险点：

- 请求体过大导致内存膨胀。
- 反序列化漏洞。
- 参数类型不兼容导致调用失败。
- I/O 线程执行重逻辑导致整体阻塞。

### 线程模型

服务端一般会区分：

- I/O 线程：负责网络读写和轻量编解码。
- 业务线程池：负责执行真正业务逻辑。

不要在 I/O 线程里执行慢业务，否则少量慢请求就能拖垮整个服务端网络处理能力。

### 方法调用

服务端根据 serviceName、methodName、parameterTypes 找到目标方法。

调用方式：

- 反射。
- MethodHandle。
- 预生成调用器。

执行过程中可能抛出：

- 业务异常。
- 参数异常。
- 权限异常。
- 线程池拒绝异常。
- 服务端超时或中断异常。

框架需要把这些异常转换成统一响应，不能直接丢连接。

## 过滤器链

成熟 RPC 框架通常有过滤器或拦截器机制。

客户端过滤器常做：

- traceId 注入。
- 日志采样。
- 请求鉴权。
- 路由和灰度。
- 超时设置。
- 重试。
- 熔断。
- 指标统计。

服务端过滤器常做：

- traceId 接收。
- 参数校验。
- 权限校验。
- 限流。
- 线程池隔离。
- 慢调用日志。
- 异常包装。

过滤器顺序很重要。例如监控最好覆盖完整耗时，限流要尽量靠前，重试要包住一次或多次真实调用。

## 工程方案

### 超时设置

RPC 必须设置超时。没有超时的远程调用等价于把本地线程交给不稳定下游。

建议：

- 区分连接超时、请求超时、读写超时。
- 调用方超时要小于入口整体超时。
- 不同接口按 P99 和业务重要性设置不同超时。
- 不要所有 RPC 统一写一个很大的默认值。

### 重试策略

只对可重试错误重试：

- 连接失败。
- 短暂网络抖动。
- 服务端返回可重试错误。
- 请求尚未到达服务端的明确失败。

谨慎对以下场景重试：

- 写请求。
- 非幂等接口。
- 服务端已经开始执行但客户端超时。
- 下游过载。

### 连接治理

需要关注：

- 连接数上限。
- 空闲连接回收。
- 心跳失败摘除。
- 半开连接检测。
- 连接建立失败的退避。
- Provider 上下线时连接清理。

### 可观测性

RPC 链路至少要有：

- 调用量 QPS。
- 成功率。
- 平均耗时、P95、P99。
- 超时数。
- 重试次数。
- 熔断和限流次数。
- 线程池活跃数、队列长度、拒绝数。
- 下游实例维度的错误率和耗时。

日志中要保留：

- traceId。
- serviceName/methodName。
- provider 地址。
- timeout。
- 耗时。
- 错误类型。

## 典型场景

### 一次读请求

读请求通常可以设置较短超时，允许有限重试。

示例：

```text
查询商品详情
  -> 调商品服务
  -> 超时 100ms
  -> 失败后重试另一个实例一次
  -> 仍失败则降级返回基础信息或空结果
```

### 一次写请求

写请求必须先确认幂等能力，再决定是否重试。

示例：

```text
创建订单
  -> 请求携带 orderNo 或 requestId
  -> 服务端以唯一键保证幂等
  -> 客户端超时后查询结果或用同一幂等号重试
```

### 服务端线程池满

如果业务线程池满：

- 服务端应快速拒绝，而不是让请求无限排队。
- 客户端应识别过载错误，减少重试。
- 监控应暴露拒绝数和队列长度。

## 线上风险

### 超时不等于失败

客户端超时只代表调用方没有在规定时间内拿到结果，不代表服务端没有执行。

影响：

- 写接口可能已经成功。
- 客户端重试可能造成重复扣款、重复发券、重复发消息。
- 补偿任务可能误判状态。

解决：

- 写请求设计幂等键。
- 状态以服务端事实源为准。
- 超时后优先查询状态。

### 重试放大流量

如果一次请求最多重试 2 次，理论流量可能变成 3 倍。下游已经慢时，重试会进一步压垮下游。

治理：

- 限制重试次数。
- 设置重试预算。
- 只对部分异常重试。
- 配合熔断和退避。

### 大对象序列化

大对象会导致：

- 序列化 CPU 高。
- 网络耗时高。
- 堆内存抖动。
- GC 压力上升。

治理：

- 限制请求和响应体大小。
- 分页或分批。
- 避免返回全量复杂对象。
- 压缩要评估 CPU 成本。

### I/O 线程阻塞

在 Netty I/O 线程做慢操作会影响同一 EventLoop 上的所有连接。

常见错误：

- 在编解码器里访问数据库。
- 在回调里做复杂 JSON 处理。
- 在 I/O 线程等待 Future。

## 常见追问

### RPC 和 HTTP 有什么区别

可以这样答：

```text
RPC 是一种远程调用抽象，关注像调用本地接口一样调用远程服务；HTTP 是一种应用层协议。RPC 可以基于 TCP、自定义协议或 HTTP/2，也可以使用 JSON、Hessian、Protobuf 等序列化。HTTP REST 更强调资源语义和通用性，RPC 更强调接口调用、性能和服务治理。实际工程中，RPC 通常内置服务发现、负载均衡、超时、重试和熔断。
```

### 为什么需要 requestId

因为同一个连接上可能有多个并发请求，响应返回顺序不一定和请求发送顺序一致。requestId 用来在客户端 pendingMap 中找到对应 Future。

### 客户端超时后服务端还会执行吗

可能会。客户端超时只影响客户端等待，服务端如果已经收到请求，可能继续执行。除非框架支持取消并且业务线程能响应中断，但工程上不能依赖这一点。

### 为什么不能无限排队

无限排队会导致：

- 请求等待时间超过业务可接受范围。
- 队列占用大量内存。
- 老请求即使执行也已经无意义。
- 故障恢复后处理过期请求，影响新请求。

所以线程池和队列要有界，满了要快速失败或降级。

### 如何定位一次 RPC 慢调用

排查顺序：

1. 看调用方耗时和超时配置。
2. 看 Provider 实例维度耗时是否集中。
3. 看网络连接是否异常。
4. 看服务端线程池队列和拒绝数。
5. 看服务端业务方法内部耗时。
6. 看 DB、缓存、下游 RPC 是否变慢。
7. 用 traceId 串起全链路。

## 易错点

- 把 RPC 等同于 TCP 长连接，忽略服务治理。
- 认为客户端超时就是服务端失败。
- 对所有接口统一重试，不区分读写和幂等。
- 在 I/O 线程中执行慢逻辑。
- 不设置请求体大小限制。
- 只监控总体成功率，不看实例维度。
- 注册中心不可用时没有本地缓存兜底。
- 异常被统一包装后丢失真实错误类型。

## 自检清单

- 能否画出从代理到服务端执行的完整链路。
- 能否说明动态代理在 RPC 中的作用。
- 能否解释 requestId、pendingMap 和异步转同步。
- 能否说明客户端超时后服务端可能继续执行。
- 能否说清服务发现、负载均衡、超时、重试、熔断的关系。
- 能否解释为什么 I/O 线程不能做慢业务。
- 能否给出 RPC 慢调用排查步骤。
- 能否说明写请求为什么必须设计幂等。
