# 超时、重试、退避

## 面试定位

超时、重试、退避是分布式系统稳定性治理的基础。面试官通常关注你能否理解远程调用的不确定性，以及如何避免“为了提高成功率的重试”反过来把系统打垮。

常见考点：

- 为什么所有远程调用都必须设置超时。
- 超时和失败有什么区别。
- 哪些请求可以重试，哪些不能重试。
- 固定间隔、指数退避、随机抖动的作用。
- 重试风暴如何形成。
- 多层服务都有重试时如何控制放大。
- Java 后端如何设计整体超时预算。

回答时要强调：超时是资源保护，重试是有条件的成功率补偿，退避是避免集中冲击。

## 核心概念

### 超时

超时是调用方愿意等待下游的最长时间。

超时保护的是：

- 调用线程。
- 连接资源。
- 线程池队列。
- 用户请求整体耗时。
- 上游系统稳定性。

没有超时的远程调用会导致线程无限等待，最终拖垮调用方。

### 重试

重试是在一次调用失败后再次尝试。

重试适合处理：

- 短暂网络抖动。
- 连接建立失败。
- 单个实例异常。
- 偶发超时。
- 明确可重试的服务端错误。

重试不适合盲目处理：

- 非幂等写请求。
- 参数错误。
- 权限错误。
- 余额不足等业务失败。
- 下游整体过载。

### 退避

退避是在重试前等待一段时间，避免立即再次冲击下游。

常见方式：

- 固定间隔。
- 线性退避。
- 指数退避。
- 指数退避加随机抖动。

## 超时分类

### 连接超时

建立连接允许等待的时间。

过长风险：

- 下游不可达时大量线程卡在连接建立。
- 故障发现慢。

过短风险：

- 网络轻微抖动时误判失败。

### 读超时

请求发出后等待响应的时间。

这是业务 RPC 最常见的超时配置。

### 写超时

发送请求数据允许等待的时间。

大请求体、网络拥塞或对端接收慢时可能触发。

### 获取连接超时

从连接池获取连接允许等待的时间。

如果连接池耗尽，应快速失败或限流，而不是无限等待。

### 总体超时

一个入口请求的整体 deadline。

例如接口总超时 800ms，内部多个 RPC、DB、缓存调用都应该共享这个预算，而不是每个下游都独立等待 800ms。

## 超时预算

### 为什么需要预算

假设入口接口目标 P99 是 500ms，内部调用链：

```text
网关 -> A 服务 -> B 服务 -> C 服务 -> DB
```

如果每层都设置 1s 超时，最终用户请求可能远超预期，线程也会长期占用。

更合理的方式：

```text
入口总预算 500ms
  -> 参数校验和本地逻辑 30ms
  -> 缓存读取 20ms
  -> B 服务 RPC 150ms
  -> C 服务 RPC 150ms
  -> 结果组装 30ms
  -> 预留 120ms
```

### Deadline 传递

上游可以把剩余时间传给下游：

```text
requestDeadline = now + 500ms
A 调 B 时剩余 320ms
B 调 C 时剩余 180ms
C 查询 DB 时剩余 100ms
```

好处：

- 避免下游继续处理已经无意义的请求。
- 防止多层超时叠加。
- 更符合用户实际等待时间。

### 超时设置依据

可以参考：

- 下游历史 P95/P99。
- 业务可接受等待时间。
- 上游整体 SLA。
- 是否可降级。
- 是否可重试。
- 高峰期延迟波动。

不要拍脑袋统一设置 3 秒或 10 秒。

## 重试条件

### 可以考虑重试

- TCP 连接失败。
- 连接被对端关闭。
- 短暂网络超时。
- 读取超时但接口是幂等读。
- 服务端返回限流之外的临时错误。
- 单个实例熔断后换实例。

### 不应该重试

- 参数校验失败。
- 认证授权失败。
- 业务规则失败。
- 明确库存不足、余额不足。
- 非幂等写请求且没有幂等键。
- 下游返回过载并要求停止。

### 写请求怎么处理

写请求不是绝对不能重试，而是必须满足幂等条件。

常见方案：

- 请求携带全局唯一 requestId。
- 服务端用唯一键防重。
- 状态机只允许合法状态推进。
- 超时后先查询结果。
- 使用同一个幂等号重试。

## 重试放大

### 单层放大

如果一次调用最多重试 2 次，总尝试次数是 3 次。

下游原本承接 1000 QPS，故障时可能变成 3000 次尝试。

### 多层放大

如果每层都重试 2 次：

```text
A -> B -> C -> D
```

理论尝试次数可能变成：

```text
3 * 3 * 3 = 27
```

这就是重试风暴。

### 控制方法

- 限制重试次数。
- 只在最上层或明确边界层重试。
- 使用全链路重试预算。
- 下游过载时禁止重试。
- 重试配合退避和熔断。
- 监控 retry_count。

## 退避策略

### 固定间隔

每次重试间隔固定，例如 50ms。

优点：

- 简单。
- 可预测。

缺点：

- 大量请求同时失败时，会同时重试，形成周期性冲击。

### 指数退避

间隔逐渐增大：

```text
50ms -> 100ms -> 200ms -> 400ms
```

优点：

- 给下游恢复时间。
- 减少连续冲击。

缺点：

- 可能增加整体耗时。
- 要设置最大间隔和总超时。

### 随机抖动

在退避时间上加入随机值：

```text
delay = base * 2^retry + random(0, jitter)
```

作用：

- 打散大量客户端同时重试。
- 避免同步脉冲。
- 降低重试风暴概率。

工程上，指数退避加随机抖动比纯固定间隔更稳。

## 工程方案

### RPC 调用

建议：

- 默认必须有超时。
- 读接口允许有限重试。
- 写接口必须有幂等键才重试。
- 重试时换实例。
- 总耗时不能超过入口 deadline。
- 对超时、重试、最终失败分别打点。

### HTTP 调用

要区分：

- connectTimeout。
- connectionRequestTimeout。
- socketTimeout/readTimeout。
- 整体调用 deadline。

连接池要有上限，避免无限建连接。

### MQ 消费

MQ 消费失败重试更常见，但也要退避。

建议：

- 区分可重试和不可重试异常。
- 重试次数有上限。
- 超过上限进死信队列或人工处理。
- 消费逻辑幂等。
- 重试间隔逐步增大。

### 定时任务补偿

补偿任务也属于重试。

要注意：

- 扫描范围有限。
- 每批数量有限。
- 按状态和更新时间过滤。
- 避免多个任务实例重复补偿。
- 对失败原因分类。

## 典型场景

### 查询接口偶发超时

方案：

- 设置较短超时。
- 重试 1 次。
- 换实例重试。
- 失败后返回降级数据或空结果。
- 监控超时率和重试成功率。

### 创建订单接口超时

方案：

- 请求携带 orderNo。
- 服务端 orderNo 唯一键。
- 客户端超时后查询订单状态。
- 状态未知时用同一 orderNo 重试。
- 不生成新的 orderNo 重试。

### 下游整体变慢

如果下游所有实例 P99 升高：

- 继续重试没有意义。
- 应快速失败、熔断或降级。
- 上游减少并发。
- 告警并定位下游。

## 线上风险

### 超时设置过长

影响：

- 调用线程堆积。
- 连接长期占用。
- 用户等待时间长。
- 故障扩散慢。

### 超时设置过短

影响：

- 正常慢请求被误判失败。
- 重试增加流量。
- 下游实际压力更大。
- 成功率下降。

### 客户端超时但服务端成功

这是写请求最典型风险。

解决：

- 幂等键。
- 查询确认。
- 状态机。
- 唯一约束。

### 重试打到过载下游

如果下游已经因为容量不足变慢，重试会进一步增加请求量。

解决：

- 下游返回明确过载错误。
- 调用方识别过载不重试。
- 熔断打开后快速失败。
- 退避和限流。

## 常见追问

### 超时是不是失败

不是。超时是调用方在规定时间内没有拿到结果，服务端可能没收到、正在执行、已经成功或执行失败。业务状态必须以服务端事实源为准。

### 为什么要加随机抖动

大量客户端如果按固定间隔重试，会在同一时间再次冲击下游。随机抖动把重试时间打散，降低瞬时峰值。

### 重试次数设置多少合适

没有固定答案。要看接口类型、幂等能力、下游 SLA 和入口超时预算。内部同步 RPC 通常控制在 0 到 2 次，写接口更谨慎。重试次数必须受总 deadline 约束。

### 在哪一层重试更好

优先在最了解业务语义、幂等语义和整体超时预算的层重试。底层框架可以提供能力，但不能无脑对所有调用重试。

## 易错点

- 所有下游统一超时。
- 只设置读超时，不设置连接池获取超时。
- 把超时当成失败并直接补偿。
- 非幂等写请求自动重试。
- 多层服务都默认重试。
- 固定间隔重试没有抖动。
- 重试不受总 deadline 控制。
- 不监控重试次数和重试成功率。

## 自检清单

- 能否解释超时、失败和未知状态的区别。
- 能否说明连接超时、读超时、获取连接超时。
- 能否设计入口请求的超时预算。
- 能否判断哪些错误可以重试。
- 能否说明写请求重试为什么需要幂等键。
- 能否解释重试风暴和多层重试放大。
- 能否讲清指数退避和随机抖动。
- 能否给出 RPC、MQ、补偿任务的重试治理方案。
