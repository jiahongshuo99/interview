# 接口版本兼容

## 面试定位

接口版本兼容是分布式系统演进能力的核心。面试官通常关注你是否理解多服务独立发布下的兼容风险，以及如何通过协议设计、字段演进、灰度发布、双写双读和契约测试避免线上事故。

常见考点：

- 为什么微服务接口必须向前、向后兼容。
- 新增字段、删除字段、修改字段类型分别有什么风险。
- RPC、HTTP、MQ 消息、数据库字段的兼容策略。
- 如何做接口版本管理。
- 如何灰度发布调用方和提供方。
- 如何处理老客户端长期不升级。
- 如何用契约测试和监控发现兼容问题。

回答时要强调：分布式系统中调用方和提供方不可能严格同时发布，所以接口变更必须考虑过渡期。

## 核心概念

### 向后兼容

新版本服务能兼容老版本调用方。

示例：

- 老客户端不传新字段，新服务仍能处理。
- 老客户端使用旧枚举值，新服务仍识别。

### 向前兼容

老版本服务或客户端能容忍新版本数据。

示例：

- 老客户端收到响应中的未知字段，不报错。
- 老消费者收到消息中的新增字段，忽略即可。

### 兼容窗口

新旧版本并存的时间。

在兼容窗口内：

- 老调用方可能调用新提供方。
- 新调用方可能调用老提供方。
- 消息可能被老消费者处理。
- 数据库中可能同时存在新旧格式数据。

兼容窗口越长，接口演进成本越高。

## 接口变更类型

### 安全变更

通常安全：

- 响应新增可选字段。
- 请求新增可选字段且服务端有默认值。
- 新增接口。
- 新增枚举值但老逻辑能忽略。
- 放宽字段长度或限制。

仍要注意老客户端是否严格反序列化未知字段。

### 危险变更

高风险：

- 删除字段。
- 字段改名。
- 字段类型变化。
- 必填字段新增。
- 枚举语义变化。
- 返回结构层级变化。
- 错误码语义变化。
- 接口路径或方法变化。

这些变更通常需要新版本接口或分阶段迁移。

## HTTP API 兼容

### 请求字段

新增请求字段应设计为可选：

```json
{
  "orderId": 123,
  "channel": "APP"
}
```

如果新增 `scene` 字段：

- 老客户端不传时服务端有默认值。
- 新客户端传时老服务端最好能忽略未知字段。
- 不要直接把新字段设为必填。

### 响应字段

响应新增字段一般兼容，但要确认客户端反序列化策略。

客户端应：

- 忽略未知字段。
- 不依赖字段顺序。
- 对缺失字段有默认处理。
- 对 null 做兼容。

### URL 和版本

常见版本方式：

- 路径版本：`/api/v1/orders`。
- Header 版本：`Accept-Version: 1`。
- 参数版本：`?version=1`。
- 服务名或接口名版本。

路径版本直观，但版本膨胀明显。Header 版本灵活，但排查和缓存要注意。

### 错误码兼容

错误码也是接口契约。

风险：

- 老客户端不识别新错误码。
- 错误码语义被复用。
- 文案变化影响客户端判断。

建议：

- 错误码只增不改语义。
- 新错误码归类到可识别大类。
- 客户端对未知错误码有默认处理。

## RPC 接口兼容

### 方法签名

RPC 接口中，直接修改方法签名风险很高。

危险操作：

- 改方法名。
- 改参数顺序。
- 改参数类型。
- 改返回值类型。
- 删除方法。

更稳妥：

- 新增方法。
- 使用请求对象封装参数。
- 新字段可选。
- 保留旧方法到调用方迁移完成。

### 参数对象

推荐：

```java
OrderQueryResponse queryOrder(OrderQueryRequest request);
```

比多个散参数更利于演进：

```java
Order queryOrder(Long orderId, Integer type, String scene);
```

请求对象新增字段时，老调用方不传也能处理。

### 序列化协议

不同协议兼容性不同。

要关注：

- 字段 ID 是否稳定。
- 是否允许未知字段。
- 是否允许字段缺失。
- 默认值语义。
- 枚举新增是否兼容。

例如 Protobuf：

- 不要复用已删除字段编号。
- 新增字段要使用新编号。
- 删除字段要 reserved。
- 老版本会忽略未知字段。

### 服务版本

RPC 常用：

- interface version。
- group。
- namespace。
- tag。

版本策略：

- 小兼容变更不升大版本。
- 不兼容变更新增版本。
- 老版本保留到调用方迁移完成。
- 注册中心支持按版本路由。

## MQ 消息兼容

MQ 消息兼容容易被忽略，因为生产者和消费者更难同步发布。

### 消息新增字段

要求：

- 消费者忽略未知字段。
- 新字段有默认值。
- 不影响老消费者处理。

### 消息删除字段

高风险。老消费者可能依赖该字段。

迁移步骤：

1. 消费者先升级，不再依赖旧字段。
2. 观察所有消费者版本。
3. 生产者再停止发送旧字段。
4. 最后清理代码和文档。

### 消息语义变更

比字段变更更危险。

例如：

```text
status=1 原来表示已支付
后来改成处理中
```

这会导致老消费者错误处理。语义变化应新增字段或新增事件类型。

### 事件版本

可以在消息中带：

```json
{
  "eventType": "ORDER_PAID",
  "eventVersion": 2,
  "eventId": "xxx",
  "payload": {}
}
```

消费者根据版本处理，或对不支持版本进入死信和告警。

## 数据库字段兼容

服务多版本发布时，数据库 schema 也要兼容。

### 新增字段

推荐步骤：

1. 新增 nullable 字段或带默认值字段。
2. 先发布能兼容字段为空的代码。
3. 灰度写入新字段。
4. 回填历史数据。
5. 观察读写一致性。
6. 最后视情况改为非空约束。

### 删除字段

不能直接删除。

步骤：

1. 代码先停止读取该字段。
2. 代码停止写入该字段。
3. 观察一段时间。
4. 确认无老版本依赖。
5. 再删除字段。

### 字段类型变化

不要直接原地改类型。

推荐：

- 新增新字段。
- 双写新旧字段。
- 读逻辑优先读新字段，缺失读旧字段。
- 回填历史数据。
- 切读。
- 停写旧字段。
- 删除旧字段。

## 发布顺序

### Provider 兼容发布

新增可选能力时：

```text
先发布 Provider，兼容老请求
再发布 Consumer，开始传新字段
```

### Consumer 兼容发布

删除旧字段时：

```text
先发布 Consumer，不再依赖旧字段
确认所有 Consumer 完成迁移
再发布 Provider，停止返回旧字段
```

### 双写双读

字段迁移常用：

```text
阶段 1：新增新字段
阶段 2：写旧字段 + 新字段
阶段 3：读新字段，缺失读旧字段
阶段 4：回填历史数据
阶段 5：只写新字段
阶段 6：删除旧字段
```

## 兼容性治理

### 契约测试

契约测试用于验证提供方是否满足调用方依赖的接口契约。

关注：

- 字段是否存在。
- 类型是否一致。
- 错误码是否符合预期。
- 必填字段是否兼容。
- 响应结构是否稳定。

### 灰度发布

接口变更必须灰度：

- 小流量验证。
- 按调用方或用户分组。
- 监控错误率和反序列化异常。
- 支持快速回滚。

### 监控指标

要监控：

- 接口错误率。
- 反序列化失败。
- 参数校验失败。
- 未知枚举值。
- 版本分布。
- 老接口调用量。
- 字段缺失比例。

### 文档和变更通知

接口是契约，变更要有：

- 变更原因。
- 兼容性说明。
- 影响范围。
- 迁移时间。
- 下线时间。
- 回滚方案。

## 典型场景

### 响应新增字段

一般兼容。

仍要确认：

- App 老版本是否忽略未知字段。
- 前端 TypeScript 类型是否严格。
- JSON 解析是否开启 fail on unknown。
- 字段为 null 时是否崩溃。

### 请求新增必填字段

不要直接新增必填。

方案：

- 先新增可选字段。
- 服务端默认旧逻辑。
- 新客户端逐步传字段。
- 观察新字段覆盖率。
- 覆盖率达标后再考虑强校验。

### 枚举新增值

风险：

- 老客户端 switch 没有 default。
- 老消费者把未知枚举当异常。

方案：

- 客户端必须有 UNKNOWN/default 分支。
- 服务端灰度发送新枚举。
- 监控未知枚举处理。

### 删除旧接口

步骤：

- 统计调用量。
- 通知调用方。
- 返回 deprecation 告警或日志。
- 等调用量归零。
- 保留一段观察期。
- 下线。

## 线上风险

### 提供方先删除字段

老调用方可能直接空指针或反序列化失败。

### 调用方先传新必填语义

老提供方不认识新字段，按旧逻辑处理，可能造成业务错误。

### 枚举语义复用

同一个枚举值在新旧版本含义不同，会造成静默错误，比直接失败更危险。

### 数据库变更不兼容老代码

如果直接改字段类型或非空约束，老版本代码可能写入失败，回滚也失败。

### 长尾客户端

移动端、外部客户、第三方系统可能长期不升级。

要保留兼容层或明确下线协议。

## 常见追问

### 如何做到接口平滑升级

可以这样答：

```text
核心原则是不做原地破坏性变更。新增能力先保持可选，Provider 先兼容老请求，Consumer 再逐步使用新字段；删除字段则反过来，先让 Consumer 不再依赖，确认调用量归零后 Provider 再停止返回。涉及存储字段迁移时使用双写双读、回填和灰度切流。
```

### 为什么推荐请求对象而不是多个参数

请求对象更容易新增可选字段，不需要改变方法签名，也能承载默认值、校验规则和扩展字段。多个散参数改动容易破坏 RPC 方法匹配和调用方编译兼容。

### 枚举新增为什么有风险

老客户端可能没有处理新枚举值，尤其是 switch 没有 default，可能抛异常或走错误逻辑。枚举要预留 UNKNOWN，客户端对未知值要有兜底。

### 如何确认旧接口可以下线

要看真实调用量和调用方清单，而不是只看代码搜索。

步骤：

- 接口维度打点。
- 调用方身份统计。
- 日志确认。
- 通知调用方迁移。
- 连续观察无调用。
- 再下线。

## 易错点

- 直接删除字段或接口。
- 新增请求必填字段。
- 修改字段类型或枚举语义。
- 老客户端不忽略未知字段。
- 数据库 schema 先做破坏性变更。
- MQ 消息只考虑生产者，不考虑所有消费者。
- 没有统计老版本调用量。
- 回滚时发现新数据格式老代码不认识。

## 自检清单

- 能否解释向前兼容和向后兼容。
- 能否判断新增字段、删除字段、改类型的风险。
- 能否设计 HTTP、RPC、MQ 的版本兼容方案。
- 能否说明请求对象为什么利于演进。
- 能否设计数据库字段双写双读迁移。
- 能否给出 Provider 和 Consumer 发布顺序。
- 能否说明枚举新增和错误码变化风险。
- 能否用契约测试、灰度和监控治理接口兼容。
