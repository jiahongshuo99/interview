# CAP、BASE

## 面试定位

CAP 和 BASE 是分布式系统理论面试的基础题。面试官通常不是想听缩写释义，而是看你能否把理论和工程实践连接起来：网络分区为什么不可避免，一致性和可用性如何取舍，为什么很多业务选择最终一致，如何设计可接受的中间状态。

常见考点：

- CAP 三个字母分别是什么。
- 为什么分布式系统发生网络分区时只能在 C 和 A 之间取舍。
- CP、AP 系统分别适合什么场景。
- BASE 和 ACID 的关系。
- 最终一致性如何落地。
- 注册中心、配置中心、订单支付、缓存分别如何取舍。

回答时要避免机械地说“三选二”。更准确的说法是：在发生网络分区时，必须在一致性和可用性之间取舍。

## CAP

### Consistency 一致性

CAP 中的一致性通常指线性一致性。所有客户端读到的都是最新写入结果，系统表现得像只有一个副本。

示例：

```text
写入 x = 1 成功后
任何客户端随后读取 x
都必须读到 1
```

### Availability 可用性

每个非故障节点收到请求后，都必须在有限时间内返回响应。

注意：

- 返回错误通常不算满足 CAP 里的可用性。
- 可用性强调节点不能因为等待一致性而拒绝响应。

### Partition Tolerance 分区容忍性

网络分区指节点之间通信失败或延迟过大，导致系统被切成多个无法互通的部分。

分布式系统必须考虑网络分区，因为：

- 网络会抖动。
- 机器会故障。
- 交换机、机房、跨地域链路都可能异常。
- GC、CPU 飙高也可能表现为通信超时。

所以 P 不是想不想要，而是必须面对。

## CAP 正确理解

### 不是简单三选二

单机系统不涉及 P。

在正常无分区时，系统可以同时提供较好的一致性和可用性。

真正取舍发生在网络分区时：

- 选择 C：为了保证一致性，部分请求必须等待或失败，牺牲可用性。
- 选择 A：为了保证可用性，各分区继续响应，可能读写到旧数据，牺牲一致性。

### CP

发生分区时优先保证一致性。

做法：

- 只有多数派可写。
- 少数派拒绝写入。
- 读写需要确认最新主节点或多数派。

适合：

- 配置中心关键配置。
- 分布式协调。
- 主节点选举。
- 金融账务核心状态。

代价：

- 分区时部分节点不可用。
- 写入延迟可能更高。

### AP

发生分区时优先保证可用性。

做法：

- 各分区都尽量响应请求。
- 允许短时间数据不一致。
- 分区恢复后通过冲突解决、同步、补偿达到最终一致。

适合：

- 服务发现。
- 缓存。
- 用户动态、点赞、浏览量。
- 可容忍短暂不一致的业务状态。

代价：

- 读到旧值。
- 冲突合并复杂。
- 需要最终一致机制。

## 一致性模型

CAP 的 C 是强一致，但工程里还有多种一致性。

### 强一致

写成功后，后续读一定读到最新值。

适合：

- 余额。
- 库存核心扣减。
- 权限。
- 唯一约束。

### 顺序一致

所有节点看到的操作顺序一致，但不一定和真实时间完全一致。

### 因果一致

有因果关系的操作顺序要被所有节点看到。

例如先评论再回复，回复不能先于评论出现。

### 最终一致

如果没有新的写入，经过一段时间后，所有副本最终会收敛到相同状态。

适合：

- 缓存同步。
- 搜索索引。
- 数据仓库。
- 订单异步履约。

## BASE

BASE 是 Basically Available、Soft state、Eventually consistent。

它是对高可用分布式系统的一种工程思想，常和 ACID 对比。

### Basically Available 基本可用

系统在故障或高峰时，保证核心功能可用，允许非核心能力降级。

示例：

- 推荐服务不可用时返回热门列表。
- 大促时关闭非核心入口。
- 页面部分模块降级。
- 写操作进入处理中状态。

基本可用不是完全可用，而是有损可用。

### Soft State 软状态

系统允许存在中间状态，这些状态不会影响整体可用性。

示例：

- 订单支付中。
- 发券处理中。
- 消息待消费。
- 搜索索引同步中。

软状态要求业务和产品能接受并表达这些中间状态。

### Eventually Consistent 最终一致

系统不要求每个时刻都一致，但经过重试、补偿、同步后最终一致。

前提：

- 有可靠事件。
- 有幂等消费。
- 有补偿任务。
- 有监控告警。
- 有人工兜底。

## ACID 和 BASE

### ACID

数据库事务特性：

- Atomicity 原子性。
- Consistency 一致性。
- Isolation 隔离性。
- Durability 持久性。

适合单库事务内强一致操作。

### BASE

适合跨服务、跨库、高可用场景。

不是说 BASE 比 ACID 更好，而是边界不同：

- 单服务单库内部尽量用 ACID。
- 跨服务链路根据业务选择最终一致。
- 核心账务仍应尽量强一致。
- 非核心链路可以 BASE。

## 工程取舍示例

### 注册中心

服务发现通常偏 AP。

原因：

- Consumer 有本地缓存。
- 服务列表短时间不一致可以接受。
- 注册中心短时不可用不应影响已有调用。

风险：

- 故障实例短时间残留。
- 上下线变更延迟。

需要客户端熔断和健康检查兜底。

### 配置中心

关键配置更偏 CP。

原因：

- 错误配置可能造成全局故障。
- 配置版本要一致。
- 发布、回滚、审计很重要。

但客户端通常也有本地缓存，以保证配置中心短时不可用时服务不崩。

### 订单支付

核心支付流水要强一致或尽量强一致。

订单展示可以最终一致：

```text
支付成功
  -> 支付流水强一致落库
  -> 订单状态异步更新
  -> 用户短时间看到支付处理中
  -> 补偿任务保证最终更新
```

### 缓存

缓存通常最终一致。

原因：

- 数据库是事实源。
- 缓存允许短暂旧值。
- 通过 TTL、删除缓存、异步刷新收敛。

但库存、余额、权限等关键判断不能只依赖可能过期的缓存。

## 最终一致性落地

最终一致需要完整链路：

```text
本地事务写业务状态
  -> 可靠记录事件
  -> 异步发送消息
  -> 消费端幂等处理
  -> 失败重试
  -> 补偿任务扫描
  -> 告警和人工处理
```

关键设计：

- 状态机。
- 幂等键。
- 唯一约束。
- 本地消息表或事务消息。
- 死信队列。
- 最大不一致时长监控。

## 典型场景

### 支付成功但订单未更新

这是最终一致中间状态。

处理：

- 支付流水为事实源。
- 订单显示支付处理中。
- 消息驱动订单更新。
- 补偿任务扫描支付成功但订单未支付记录。
- 用户刷新后最终看到成功。

### 库存扣减

如果是秒杀库存：

- 库存扣减本身要强约束，不能超卖。
- 订单创建和后续履约可以最终一致。

如果是展示库存：

- 可以缓存，允许短暂不准。

### 搜索索引同步

数据库写成功后，搜索索引异步更新。

短时间内搜索不到新数据可以接受，但要监控同步延迟和失败队列。

## 线上风险

### 把最终一致当作无需治理

最终一致不是“迟早会好”，必须有明确机制让它变好。

缺少补偿、重试和告警时，最终一致会变成永久不一致。

### 中间状态没有产品语义

用户看到异常状态却无法理解，会造成投诉。

需要：

- 明确处理中状态。
- 支持查询最终结果。
- 避免展示冲突信息。

### 一致性边界不清

要区分：

- 哪些数据必须强一致。
- 哪些数据允许最终一致。
- 最大可接受不一致时间。
- 出现不一致时谁是事实源。

### 为了强一致牺牲过多可用性

所有链路都强一致会导致：

- 性能差。
- 发布和扩展困难。
- 下游小故障拖垮核心流程。

要按业务价值取舍。

## 常见追问

### CAP 为什么不是简单三选二

可以这样答：

```text
CAP 的取舍主要发生在网络分区时。分布式系统必须容忍分区，所以真正要选的是分区发生时优先保证一致性还是可用性。正常情况下，系统可以同时提供较好的一致性和可用性，但一旦分区，就不能同时保证线性一致和所有节点都响应。
```

### 服务发现为什么通常偏 AP

因为服务发现数据允许短暂不一致，Consumer 本地有缓存。分区时继续使用旧服务列表通常比完全不可用更好。故障实例残留可以通过客户端失败隔离、熔断和健康检查收敛。

### BASE 是不是放弃一致性

不是。BASE 放弃的是每个时刻强一致，追求最终一致。它要求通过消息、重试、补偿、幂等和监控让系统最终收敛。

### 怎么判断一个业务能不能最终一致

看几个问题：

- 用户能否接受处理中状态。
- 是否有明确事实源。
- 是否能补偿。
- 最大不一致时间是多少。
- 重复执行是否幂等。
- 失败是否能人工修复。

## 易错点

- 机械背 CAP 三选二。
- 把 CAP 的一致性和数据库 ACID 一致性混淆。
- 认为 AP 就是不保证数据正确。
- 认为最终一致不用补偿。
- 所有业务都套强一致，忽略可用性和性能。
- 缓存旧值参与关键写判断。
- 没有定义事实源。
- 中间状态对用户不可解释。

## 自检清单

- 能否准确解释 CAP 的 C、A、P。
- 能否说明为什么分区时只能在 C 和 A 之间取舍。
- 能否举例说明 CP 和 AP 场景。
- 能否解释 BASE 三个概念。
- 能否区分 ACID 和 BASE 的适用边界。
- 能否设计最终一致闭环。
- 能否说明服务发现、配置中心、缓存、支付订单的一致性取舍。
- 能否识别最终一致的线上风险。
