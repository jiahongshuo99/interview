# 分布式锁

## 面试定位

分布式锁是 Java 后端面试中非常高频但容易答错的题。面试官通常不是想听“Redis setnx 就可以”，而是要看你是否理解锁的互斥性、过期时间、续期、误删、主从切换、数据库锁、ZooKeeper 锁以及业务兜底。

常见考点：

- 为什么需要分布式锁。
- Redis 分布式锁正确写法。
- 为什么释放锁要校验 value。
- 锁过期但业务没执行完怎么办。
- Redisson 看门狗原理。
- Redis 锁和 ZooKeeper 锁区别。
- 分布式锁能否保证幂等。
- 分布式锁失效会造成什么线上问题。

回答时要强调：分布式锁只是并发控制手段，核心业务正确性还要靠唯一约束、状态机和事务兜底。

## 使用场景

适合使用分布式锁的场景：

- 多实例定时任务只允许一个实例执行。
- 同一个资源同一时间只允许一个请求修改。
- 防止缓存击穿时多个实例同时回源。
- 控制同一个用户或订单的并发状态推进。
- 分布式环境下保护短临界区。

不适合滥用的场景：

- 高吞吐核心写链路全部串行化。
- 长时间任务持锁。
- 用锁替代数据库唯一约束。
- 用锁替代状态机幂等。

## 分布式锁基本要求

一个可用的分布式锁至少要考虑：

- 互斥性：同一时间只有一个客户端持有锁。
- 防死锁：持锁客户端崩溃后锁能释放。
- 防误删：不能释放别人的锁。
- 可重入性：是否支持同一线程重复获得。
- 自动续期：业务执行超过过期时间怎么办。
- 容错：锁服务故障时如何处理。
- 性能：加锁释放不能成为瓶颈。

## Redis 分布式锁

### 基本写法

加锁：

```text
SET lock_key unique_value NX PX 30000
```

含义：

- NX：key 不存在才设置。
- PX 30000：过期时间 30 秒。
- unique_value：当前锁持有者标识，例如 UUID + threadId。

释放锁必须使用 Lua 脚本保证判断和删除原子性：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

### 为什么 value 要唯一

如果不校验 value，会出现误删：

```text
A 获取锁，过期时间 10s
A 执行业务超过 10s，锁过期
B 获取同一把锁
A 执行完 del lock_key
A 删除了 B 的锁
C 又获取锁
B 和 C 并发执行
```

所以释放时必须确认锁还是自己持有的。

### 过期时间怎么设置

过期时间太短：

- 业务没执行完锁就过期。
- 其他实例进入临界区。

过期时间太长：

- 持锁实例崩溃后恢复慢。
- 阻塞其他请求。

设置依据：

- 临界区 P99 耗时。
- 最大可接受阻塞时间。
- 是否有自动续期。
- 业务是否允许失败重试。

### 自动续期

如果业务耗时不确定，可以使用看门狗机制：

```text
加锁成功
  -> 后台定时任务在锁过期前续期
  -> 业务执行完成后释放锁
  -> 持锁进程崩溃后不再续期，锁最终过期
```

注意：

- 续期线程本身也可能异常。
- 长任务持锁会降低吞吐。
- JVM STW 时间过长可能导致续期不及时。

### Redis 主从切换风险

单 Redis 主节点加锁后，如果数据还没同步到从节点，主节点宕机并发生切换，新主节点没有锁 key，其他客户端可能再次加锁成功。

这会破坏强互斥。

因此 Redis 锁适合大多数工程互斥场景，但不能用于要求绝对强一致的金融级关键互斥。关键业务仍要数据库约束兜底。

## Redisson 锁

Redisson 封装了 Redis 锁常见能力：

- 可重入锁。
- Lua 原子释放。
- 看门狗自动续期。
- 公平锁。
- 读写锁。
- 联锁。

可重入大致原理：

- value 中记录持有线程标识。
- hash 结构记录重入次数。
- 同一线程再次加锁，计数加一。
- 解锁时计数减一，减到零才删除。

看门狗：

- 默认给锁设置过期时间。
- 持锁客户端还活着时定期续期。
- 客户端宕机后停止续期，锁自动释放。

面试中要说明 Redisson 降低了误用概率，但不代表分布式锁绝对可靠。

## ZooKeeper 分布式锁

### 基本原理

ZooKeeper 锁通常基于临时顺序节点。

流程：

```text
客户端在 /locks/order 下创建临时顺序节点
  -> /locks/order/lock-00000001
  -> /locks/order/lock-00000002

序号最小的客户端获得锁
其他客户端监听自己前一个节点
前一个节点删除后，重新判断自己是否最小
```

### 特点

优点：

- 临时节点随会话断开自动删除。
- 顺序节点天然支持排队。
- 监听前一个节点避免惊群。
- 一致性语义更强。

缺点：

- 性能通常不如 Redis。
- 对 ZooKeeper 集群依赖较重。
- 会话超时配置影响锁释放。
- 使用复杂度更高。

适合：

- 分布式协调。
- 主节点选举。
- 对锁语义要求更强、并发量不是特别高的场景。

## 数据库锁

### 唯一索引抢锁

建锁表：

```text
lock_name unique
owner
expire_time
```

通过插入同一个 lock_name 抢锁，唯一键保证互斥。

优点：

- 依赖已有数据库。
- 强一致。

缺点：

- 性能较低。
- 需要处理过期锁。
- 容易增加数据库压力。

### select for update

在事务中锁定某行：

```sql
SELECT * FROM lock_table WHERE lock_name = ? FOR UPDATE;
```

适合单库事务内资源互斥，不适合长时间跨服务锁。

## 分布式锁和幂等

分布式锁不能替代幂等。

示例：

```text
A 获取锁，执行业务成功
A 释放锁
重复请求 B 获取锁
B 再次执行业务
```

锁保证的是同一时间只有一个执行者，不保证同一业务只执行一次。

核心写操作仍需要：

- 唯一索引。
- 幂等表。
- 状态机。
- 乐观锁。

## 工程方案

### 缓存击穿

场景：

```text
热点 key 过期
大量请求同时访问
都发现缓存为空
都回源数据库
```

方案：

- 只有一个请求获得锁并回源。
- 其他请求短暂等待后读缓存。
- 获取锁失败不要无限等待。
- 回源失败要释放锁并设置短暂空值或降级。

### 定时任务单实例执行

方案：

- 任务开始前抢锁。
- 锁 value 记录实例和线程。
- 设置合理过期时间或续期。
- finally 中释放锁。
- 任务逻辑幂等。

### 订单状态推进

可以用锁降低并发冲突，但最终要靠状态机：

```sql
UPDATE orders
SET status = 'PAID'
WHERE order_id = ?
  AND status = 'CREATED';
```

锁失效时，状态机仍能防止非法重复推进。

## 典型风险

### 锁过期业务未完成

后果：

- 其他客户端获取锁。
- 多个客户端同时执行临界区。
- 数据重复更新。

解决：

- 设置合理过期时间。
- 自动续期。
- 缩短临界区。
- 业务幂等兜底。

### 释放别人锁

后果：

- 锁互斥失效。
- 多个请求并发执行。

解决：

- value 唯一。
- Lua 校验再删除。

### 持锁时间过长

后果：

- 吞吐下降。
- 请求阻塞。
- 锁竞争激烈。
- 故障恢复慢。

解决：

- 只锁必要临界区。
- 不在锁内做慢 RPC。
- 尽量用状态机或乐观锁替代。

### 锁服务不可用

要明确策略：

- 快速失败。
- 降级。
- 走数据库兜底。
- 禁止关键操作。

不能在锁服务不可用时默认放行所有请求，除非业务能承受并发执行。

## 常见追问

### Redis 分布式锁正确写法是什么

可以这样答：

```text
加锁用 SET key uniqueValue NX PX expireMillis，保证加锁和设置过期时间是一个原子命令。释放锁不能直接 del，要用 Lua 脚本先判断 value 是否还是自己的，再删除，避免锁过期后误删别人的锁。
```

### 锁过期但业务没执行完怎么办

可以设置足够的过期时间，或使用看门狗自动续期。但更重要的是缩短临界区，并用唯一约束、状态机等业务手段兜底，因为续期也可能受 JVM STW、网络抖动影响。

### Redis 锁和 ZooKeeper 锁怎么选

- Redis：性能高，适合短临界区、高并发、允许业务兜底的场景。
- ZooKeeper：一致性语义更强，适合分布式协调、选主、并发量较低但锁语义要求更高的场景。

### RedLock 是否一定可靠

RedLock 试图通过多个独立 Redis 节点提升可靠性，但工程界对它是否满足严格分布式锁语义有争议。面试中可以说：普通业务更常用单 Redis/Redis 集群加业务兜底；强一致关键场景不应只依赖 Redis 锁。

## 易错点

- `setnx` 后再单独 `expire`，中间宕机会死锁。
- 解锁直接 `del`，没有校验 value。
- 锁 value 固定，无法区分持有者。
- 锁过期时间小于业务执行时间。
- 在锁里做慢 RPC 或长事务。
- 用分布式锁替代数据库唯一约束。
- 认为 Redisson 锁绝对可靠。
- 没有处理加锁失败和锁服务不可用。
