# 幂等、防重、去重

## 面试定位

幂等、防重、去重是 Java 后端高频面试题，尤其常出现在支付、订单、发券、消息消费、回调、补偿任务等场景。面试官通常希望你能区分“同一个请求重复进来”和“业务上重复操作”，并能给出数据库、Redis、MQ、状态机等工程方案。

常见考点：

- 幂等是什么，和防重、去重有什么区别。
- 为什么 RPC 超时和 MQ 重投会导致重复执行。
- 唯一索引、状态机、幂等号、去重表如何使用。
- Redis `setnx` 做防重有什么风险。
- 消息消费如何保证幂等。
- 幂等记录何时删除、保留多久。
- 如何处理并发重复请求。

回答时要强调：分布式系统无法完全避免重复请求，只能让重复请求产生相同业务结果。

## 核心概念

### 幂等

同一个操作执行一次和执行多次，对业务最终状态的影响相同。

示例：

- 查询订单天然幂等。
- 按 orderId 更新订单状态为 PAID，在状态机约束下可以幂等。
- 按 requestId 发券，同一个 requestId 只发一次，可以幂等。
- 扣余额如果每次都 `balance = balance - amount`，不是幂等。

### 防重

防止同一请求在短时间内重复提交。

常见场景：

- 用户连续点击按钮。
- 前端超时后重复提交。
- 网关重放请求。
- 定时任务重复触发。

防重偏入口控制，不等于完整幂等。

### 去重

识别重复数据或重复消息，只处理一次。

常见场景：

- MQ 消息重复投递。
- 回调通知重复到达。
- 批量导入中重复记录。
- 日志或事件流重复消费。

去重偏数据处理。

## 为什么会重复

### 网络不可靠

客户端发出请求后可能出现：

- 请求没有到服务端。
- 请求到了服务端但响应丢失。
- 服务端处理成功但客户端超时。
- 客户端不知道最终结果而重试。

所以“客户端没收到成功”不代表“服务端没执行”。

### RPC 重试

调用方为了提升成功率可能重试。如果写接口没有幂等，重复请求会造成重复扣款、重复创建、重复发放。

### MQ 至少一次投递

很多 MQ 保证至少一次投递，不保证只投递一次。

消息可能因为以下原因重复：

- 消费成功但 ack 失败。
- 消费者处理超时。
- Broker 重平衡。
- 消费者重启。
- 生产者重复发送。

### 补偿任务

补偿任务为了修复异常状态，会重复扫描和重试。如果没有幂等，补偿可能造成二次伤害。

## 常见幂等方案

### 唯一索引

用数据库唯一约束保证同一业务键只能写一次。

示例：

```sql
CREATE UNIQUE INDEX uk_order_no ON orders(order_no);
```

创建订单：

```text
第一次 insert 成功
重复 insert 触发唯一键冲突
查询已有订单并返回同一结果
```

优点：

- 强一致。
- 简单可靠。
- 数据库兜底，不怕应用并发。

缺点：

- 依赖数据库写入。
- 高并发下唯一键冲突也有成本。
- 业务键设计错误会误拦截。

适合：

- 创建类操作。
- 订单号、支付单号、发放流水号。

### 幂等号表

单独建幂等记录表。

表设计示例：

```text
id
biz_type
idempotent_key
request_hash
status
result
expire_time
created_at
updated_at
```

唯一键：

```text
biz_type + idempotent_key
```

流程：

```text
收到请求
  -> 插入幂等记录 PROCESSING
  -> 插入成功，执行业务
  -> 业务成功，更新幂等记录 SUCCESS 和结果
  -> 重复请求发现已有记录
     -> SUCCESS 返回历史结果
     -> PROCESSING 返回处理中或稍后重试
     -> FAILED 根据业务决定是否允许重试
```

适合：

- 需要返回相同响应。
- 请求体复杂，需要校验重复请求参数是否一致。
- 需要记录处理状态。

### 状态机幂等

通过状态流转约束保证重复操作无副作用。

示例：

```text
订单状态：
CREATED -> PAID -> DELIVERED -> FINISHED
```

支付成功回调：

```sql
UPDATE orders
SET status = 'PAID'
WHERE order_id = ?
  AND status = 'CREATED';
```

如果重复回调再次执行，影响行数为 0，再查询订单发现已经是 PAID，可以返回成功。

优点：

- 符合业务语义。
- 对回调、补偿、消息消费很有效。

风险：

- 状态流转必须完整。
- 不能用无条件 update。
- 要处理非法状态和重复状态。

### 乐观锁

使用 version 字段或 CAS 条件更新。

示例：

```sql
UPDATE account
SET balance = balance - 100,
    version = version + 1
WHERE account_id = ?
  AND version = ?
  AND balance >= 100;
```

适合：

- 防并发覆盖。
- 状态更新。
- 库存扣减。

乐观锁不天然解决业务重复扣减，仍需要业务幂等键。

### Redis 防重

常用：

```text
SET key value NX EX 60
```

表示 key 不存在才设置，并设置过期时间。

适合：

- 短时间防重复点击。
- 限制同一任务重复执行。
- 降低数据库压力。

风险：

- Redis 成功后业务失败，key 仍存在，可能误拦截。
- 过期时间太短，业务未完成时重复请求进入。
- 过期时间太长，失败后无法重试。
- Redis 主从切换可能导致短暂不一致。

核心业务不要只依赖 Redis 防重，数据库唯一键或状态机要兜底。

## MQ 消费幂等

### 消费者幂等原则

不要依赖 MQ “绝对只投递一次”。消费者必须自己保证重复消息不会重复产生业务副作用。

### 基于消息 ID 去重

维护消费记录表：

```text
message_id
consumer_group
status
created_at
updated_at
```

唯一键：

```text
consumer_group + message_id
```

流程：

```text
收到消息
  -> 插入消费记录
  -> 插入成功，处理业务
  -> 处理成功，更新 SUCCESS 并 ack
  -> 插入失败，说明处理过或正在处理
```

### 基于业务键幂等

比消息 ID 更推荐，因为同一业务事件可能被不同消息 ID 重复发送。

例如发券消息：

```text
biz_key = activityId + userId + couponType
```

发放流水表对 biz_key 建唯一索引。

### ack 时机

一般应在业务处理成功后 ack。

如果先 ack 再处理：

- 业务失败后消息不会重投，造成丢失。

如果处理成功但 ack 失败：

- 消息会重投，所以业务必须幂等。

## 防重和锁的区别

防重关注同一业务请求只能处理一次，锁关注同一时间只能有一个执行者。

不要把分布式锁当成唯一幂等方案。

原因：

- 锁过期后重复请求仍可能进入。
- 持锁者业务成功但释放失败，后续请求行为复杂。
- 锁只能保护临界区，不能证明业务已经处理过。

核心写操作需要业务唯一键或状态机兜底。

## 幂等键设计

### 好的幂等键

应该满足：

- 全局唯一或业务域内唯一。
- 与一次业务意图绑定。
- 重试时保持不变。
- 不同业务意图不能共用。
- 可追踪。

示例：

- orderNo。
- paymentNo。
- refundNo。
- requestId。
- outBizNo。
- messageBizKey。

### 常见错误

- 使用时间戳作为幂等键，重试时变了。
- 使用 userId 做幂等键，导致一个用户只能下一单。
- 使用随机 UUID，但每次重试重新生成。
- 幂等键不包含业务类型，跨业务误冲突。

### 请求参数一致性

同一个幂等键重复请求时，要校验参数是否一致。

例如：

```text
第一次 requestId=1, amount=100
第二次 requestId=1, amount=200
```

不能直接返回第一次结果，应该识别为幂等键冲突或非法请求。

可以存 request_hash 做校验。

## 工程方案

### 创建类接口

推荐：

- 客户端传业务唯一号。
- 数据库唯一索引兜底。
- 重复请求查询已有记录返回。
- 不让服务端每次生成新业务号。

### 状态推进类接口

推荐：

- 明确状态机。
- 条件更新。
- 重复状态返回成功。
- 非法状态返回明确错误。

### 发放类接口

推荐：

- 发放流水表。
- 业务唯一键。
- 唯一索引。
- 发放结果落库。
- 补偿任务按流水状态重试。

### 回调接口

推荐：

- 验签。
- 回调流水记录。
- 业务单号唯一。
- 状态机推进。
- 重复回调返回成功。

### MQ 消费

推荐：

- 以业务键做幂等。
- 消费成功后 ack。
- 失败可重试。
- 不可重试错误进入死信或人工处理。

## 典型场景

### 用户重复点击提交订单

方案：

- 前端按钮置灰只是体验优化。
- 后端以 orderNo 或 requestId 做唯一约束。
- Redis 可做短时间防抖。
- 数据库唯一键兜底。

### 支付回调重复通知

方案：

- 验签。
- 根据 paymentNo 查询支付单。
- `CREATED -> PAID` 条件更新。
- 已经 PAID 时返回成功。
- 金额不一致时告警。

### MQ 重复发券

方案：

- 发券流水唯一键：activityId + userId + couponId。
- 插入流水成功才发券。
- 重复消息查询流水状态。
- 发券失败保留 FAILED，补偿任务重试。

## 线上风险

### 幂等记录和业务操作不在同一事务

如果幂等记录成功但业务失败，或者业务成功但幂等记录失败，会出现不一致。

解决：

- 尽量放同一数据库事务。
- 或以业务事实表唯一键为准。
- 设计 PROCESSING、SUCCESS、FAILED 状态。
- 补偿任务修复中间状态。

### 幂等 key 过期太早

请求晚到或 MQ 延迟重投时，幂等记录已经过期，会重复执行。

解决：

- 核心业务幂等记录保留足够久。
- 和消息最大重试周期、回调周期一致。
- 关键流水永久保留或归档。

### 幂等范围过大

幂等键设计太粗，会误伤正常请求。

例如 userId 作为下单幂等键，一个用户无法创建多个订单。

### 只靠前端防重

前端防重无法防：

- 网络重试。
- 恶意请求。
- MQ 重投。
- 服务端补偿重复。

后端必须兜底。

## 常见追问

### 幂等和分布式锁有什么区别

可以这样答：

```text
分布式锁解决同一时间只有一个执行者，幂等解决同一业务请求执行多次最终结果相同。锁不能替代幂等，因为锁过期、释放失败或请求重放后，业务仍可能重复执行。核心写操作应该用唯一键、状态机或幂等表兜底。
```

### Redis setnx 能保证幂等吗

只能做短时间防重，不能单独保证核心业务幂等。因为 Redis key 过期、业务失败、主从切换、锁误删等问题都可能导致重复或误拦截。核心结果要以数据库唯一约束或状态机为准。

### MQ 如何做到 exactly once

工程上通常不追求端到端绝对 exactly once，而是通过至少一次投递加消费者幂等，达到业务效果上的 exactly once。

### 重复请求正在处理中怎么办

可以返回处理中，也可以短暂等待后查询结果。不要让重复请求并发进入业务临界区。幂等表可用 PROCESSING 状态表达处理中。

## 易错点

- 把防重当成幂等。
- 认为 MQ 不会重复投递。
- 非幂等写请求开启自动重试。
- 幂等 key 每次重试重新生成。
- 只用 Redis，不做数据库兜底。
- 状态更新没有带旧状态条件。
- 同一幂等 key 不校验请求参数一致性。
- 幂等记录过期时间短于消息重试周期。
