# 分布式事务：2PC、TCC、Saga、本地消息表、最终一致性

## 面试定位

分布式事务是 Java 后端面试中区分工程经验的重要题。面试官通常希望你能说明为什么本地事务不够用，2PC、TCC、Saga、本地消息表分别解决什么问题，以及实际项目中如何在强一致和最终一致之间取舍。

常见考点：

- 本地事务和分布式事务的区别。
- 2PC 的流程、阻塞和单点问题。
- TCC 的 Try、Confirm、Cancel 怎么设计。
- Saga 正向事务和补偿事务。
- 本地消息表如何保证消息和数据库写入一致。
- 事务消息和最终一致性的关系。
- 如何处理悬挂、空回滚、幂等。
- 面试中如何结合订单、支付、库存、发券场景讲方案。

回答时要强调：分布式事务没有银弹，要基于业务一致性要求、性能、复杂度和可补偿性选择方案。

## 为什么需要分布式事务

单体应用中，一个数据库本地事务可以保证多张表一致：

```text
BEGIN
  update order
  update inventory
COMMIT
```

微服务拆分后：

```text
订单服务写订单库
库存服务写库存库
支付服务写支付库
```

这些操作跨服务、跨数据库，本地事务无法直接覆盖。

问题：

- 订单创建成功，库存扣减失败怎么办。
- 支付成功，订单状态更新失败怎么办。
- 发券成功，活动流水更新失败怎么办。
- MQ 消息发送成功但数据库事务回滚怎么办。

## 一致性选择

### 强一致

所有参与方要么都成功，要么都失败，中间状态对业务不可见。

优点：

- 语义简单。
- 数据短时间内一致。

缺点：

- 性能差。
- 可用性受限。
- 跨服务耦合强。

### 最终一致

允许短时间中间状态，通过消息、补偿、重试让系统最终达到一致。

优点：

- 可用性更高。
- 性能更好。
- 服务耦合较低。

缺点：

- 状态复杂。
- 用户可能看到处理中。
- 需要补偿、幂等、监控和人工处理。

大多数互联网业务更常采用最终一致。

## 2PC 两阶段提交

### 基本流程

2PC 有协调者和参与者。

第一阶段：Prepare。

```text
协调者询问所有参与者能否提交
参与者执行本地事务但不提交
参与者锁定资源并返回 yes/no
```

第二阶段：Commit/Rollback。

```text
如果所有参与者 yes
  -> 协调者通知所有参与者 commit
否则
  -> 协调者通知所有参与者 rollback
```

### 优点

- 提供较强一致性。
- 模型直观。
- 适合数据库 XA 场景。

### 缺点

- 同步阻塞：参与者在 prepare 后持有资源锁。
- 协调者单点：协调者故障会影响事务完成。
- 性能差：多轮网络交互。
- 可用性低：一个参与者慢会拖慢整体。
- 不适合长事务和高并发核心链路。

### 适用场景

- 强一致要求高。
- 参与方少。
- 事务时间短。
- 并发量不高。
- 能接受资源锁定。

互联网核心链路一般谨慎使用 2PC/XA。

## TCC

TCC 是 Try、Confirm、Cancel。

### Try

预留资源，检查业务条件。

例如冻结库存：

```text
available_stock -= count
frozen_stock += count
```

### Confirm

真正提交业务。

例如确认扣减库存：

```text
frozen_stock -= count
```

### Cancel

取消预留，释放资源。

例如解冻库存：

```text
available_stock += count
frozen_stock -= count
```

### TCC 特点

优点：

- 一致性较强。
- 业务可控。
- 不依赖数据库长时间锁。

缺点：

- 业务侵入大。
- 每个参与方都要实现三个接口。
- 要处理空回滚、悬挂、幂等。
- Confirm/Cancel 必须可靠重试。

### TCC 三大问题

#### 空回滚

Cancel 先于 Try 到达，或者 Try 没成功却执行了 Cancel。

解决：

- 记录事务分支状态。
- Cancel 时如果没有 Try 记录，插入一条空回滚记录。
- 后续 Try 到来时识别已空回滚并拒绝。

#### 悬挂

由于网络延迟，Cancel 先执行，之后 Try 又到达。如果 Try 成功，就会出现资源被预留但全局事务已经回滚。

解决：

- Try 前检查事务状态。
- 如果已经 Cancel，Try 不能再成功。

#### 幂等

Confirm 或 Cancel 可能重复调用。

解决：

- 分支事务唯一键。
- 状态机。
- 重复 Confirm 返回成功。
- 重复 Cancel 返回成功。

### TCC 适用场景

- 业务需要较强一致。
- 资源可以预留。
- 参与方数量有限。
- 能接受业务改造。

典型：

- 库存冻结。
- 优惠券锁定。
- 账户资金冻结。

## Saga

Saga 把一个长事务拆成多个本地事务，每个本地事务都有对应补偿事务。

### 基本流程

```text
T1 创建订单
T2 扣库存
T3 扣余额
T4 发券
```

如果 T3 失败，需要补偿已成功步骤：

```text
C2 释放库存
C1 取消订单
```

### 编排式 Saga

由中心协调器控制流程。

优点：

- 流程清晰。
- 状态集中可见。
- 容易做监控和人工介入。

缺点：

- 协调器复杂。
- 参与方依赖协调器。

### 协同式 Saga

各服务通过事件驱动推进。

优点：

- 服务更解耦。
- 没有强中心流程。

缺点：

- 流程分散。
- 排查困难。
- 容易形成事件链路复杂度。

### Saga 特点

优点：

- 适合长事务。
- 不长时间锁资源。
- 可用性高。

缺点：

- 只能最终一致。
- 补偿不一定能完全抵消。
- 用户可能看到中间状态。
- 每一步都要幂等和可补偿。

### 适用场景

- 流程长。
- 参与服务多。
- 可接受最终一致。
- 每一步有明确补偿动作。

例如：

- 订单履约。
- 旅游预订。
- 多系统开户。
- 审批流程。

## 本地消息表

本地消息表解决数据库写入和消息发送的一致性问题。

### 问题

如果业务写库成功后发送 MQ：

```text
BEGIN
  创建订单
COMMIT
发送订单创建消息
```

如果提交成功后服务宕机，消息没发出去，下游不知道订单创建。

如果先发消息再提交事务，事务回滚后下游可能收到不存在的订单。

### 基本方案

把业务数据和消息记录放在同一个本地事务中。

```text
BEGIN
  创建订单
  插入本地消息表 status=NEW
COMMIT

后台任务扫描 NEW 消息
  -> 发送 MQ
  -> 发送成功后标记 SENT
```

因为业务表和消息表在同一个数据库事务内，所以能保证：

- 订单创建成功，一定有消息记录。
- 订单回滚，消息记录也回滚。

### 消息表字段

常见字段：

```text
id
biz_type
biz_id
message_key
payload
status
retry_count
next_retry_time
last_error
created_at
updated_at
```

### 发送端可靠性

发送任务要：

- 扫描未发送消息。
- 控制批量大小。
- 失败重试并退避。
- 超过次数告警。
- 发送成功后更新状态。

### 消费端幂等

本地消息表只保证消息最终发出去，不保证消费者只处理一次。

消费者仍要幂等：

- 业务唯一键。
- 消费记录表。
- 状态机。

## 事务消息

很多 MQ 提供事务消息能力，简化本地消息表。

典型流程：

```text
发送半消息
  -> 执行业务本地事务
  -> 提交或回滚消息
  -> Broker 未收到最终状态时回查生产者
```

优点：

- 框架封装消息和本地事务一致性。
- 减少手写消息表。

注意：

- 仍然是最终一致。
- 消费端仍要幂等。
- 回查接口必须能根据业务状态判断提交或回滚。

## 最终一致性工程闭环

最终一致不是“发个 MQ 就完了”，需要完整闭环：

- 本地事务保证业务状态和事件记录一致。
- 消息可靠发送。
- 消费端幂等。
- 失败重试。
- 死信队列。
- 补偿任务。
- 状态机。
- 监控告警。
- 人工处理入口。

## 典型场景

### 下单扣库存

方案选择：

- 强一致库存预留：TCC。
- 高并发秒杀：库存服务本地原子扣减 + 订单最终一致。
- 普通电商：订单创建后发消息扣库存，失败则取消订单或进入异常状态。

### 支付成功更新订单

常见方案：

- 支付回调先落支付流水。
- 通过本地事务更新支付单并写消息。
- 消费消息更新订单状态。
- 订单状态机幂等。
- 补偿任务扫描支付成功但订单未更新记录。

### 发券

常见方案：

- 发券请求先写发放流水。
- 发券服务保证业务唯一键。
- 发券失败可重试。
- 超过次数进入人工处理。
- 用户侧展示处理中或失败可重试。

## 线上风险

### 补偿动作不可逆

有些动作不能简单补偿：

- 已发短信无法撤回。
- 已通知外部系统无法完全取消。
- 已打款可能需要反向退款。

这类动作要尽量放在流程后面，或者设计明确反向业务。

### 中间状态用户可见

最终一致下，用户可能看到：

- 订单处理中。
- 支付成功但订单未更新。
- 发券中。

需要：

- 明确状态展示。
- 查询接口聚合真实状态。
- 补偿任务缩短不一致时间。

### 重试造成重复副作用

所有 Confirm、Cancel、消息消费、补偿任务都可能重复执行，必须幂等。

### 本地消息表积压

如果 MQ 故障或发送任务异常，消息表会积压。

要监控：

- NEW 消息数量。
- 最大滞留时间。
- 重试次数。
- 发送失败原因。

## 常见追问

### 2PC 为什么性能差

可以这样答：

```text
2PC 需要协调者和所有参与者进行两轮交互，参与者在 prepare 后要持有本地资源锁，直到收到 commit 或 rollback。任何参与者慢或协调者异常都会拖慢整体事务，所以并发高、链路长的互联网业务通常不适合大规模使用 2PC。
```

### TCC 和 Saga 有什么区别

TCC 是资源预留模型，Try 阶段先冻结资源，Confirm 真正提交，Cancel 释放资源，一致性更强但业务侵入大。Saga 是把长事务拆成本地事务和补偿事务，不长期锁资源，更适合长流程，但只能最终一致。

### 本地消息表解决了什么问题

它解决业务数据库写入和消息发送之间的一致性问题。业务数据和消息记录在同一个本地事务里提交，然后异步发送消息。它不解决消费端重复消费，消费者仍要幂等。

### 最终一致性怎么保证最终一定一致

靠可靠事件记录、重试、幂等、补偿、告警和人工处理闭环。不能只说 MQ，必须说明失败后如何发现、如何重试、如何兜底。

## 易错点

- 把分布式事务等同于 2PC。
- 认为 MQ 能自动保证业务最终一致。
- 本地事务提交后直接发 MQ，没有可靠消息记录。
- 消费者不做幂等。
- TCC 不处理空回滚、悬挂和幂等。
- Saga 补偿动作不可逆却没有说明。
- 最终一致没有状态机和补偿任务。
- 用户可见中间状态没有产品语义。
