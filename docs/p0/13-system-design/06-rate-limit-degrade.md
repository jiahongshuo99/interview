# 限流降级

## 面试定位

限流降级考察的是高并发系统的稳定性设计。面试官希望你能说清：系统过载时如何保护核心链路，如何区分限流、熔断、降级、隔离，如何设置阈值，如何避免故障扩散。

核心回答：

```text
稳定性设计不是让所有请求都成功，而是在资源有限和依赖异常时，优先保证核心业务可用。
```

## 需求澄清

先问：

- 核心链路是什么，哪些请求必须保。
- 哪些能力可以降级，例如推荐、评论、短信、数据分析。
- 限流维度是什么：接口、用户、IP、设备、商家、活动、租户。
- 限流目标是什么：保护应用、DB、Redis、MQ、第三方接口。
- 超限后返回什么：失败、排队、兜底、静态数据。
- 是否允许按用户等级、业务优先级差异化。
- 是否需要动态配置和灰度。
- 是否有监控指标和自动恢复条件。

## 核心概念

### 限流

限制进入系统的请求速率，防止资源被打满。

常见算法：

- 固定窗口计数。
- 滑动窗口计数。
- 令牌桶。
- 漏桶。
- 并发数限制。

### 熔断

当下游错误率或超时率过高时，短时间内停止调用下游，快速失败，避免线程池和连接池被拖死。

典型状态：

```text
CLOSED -> OPEN -> HALF_OPEN -> CLOSED
```

### 降级

在依赖异常或资源不足时，返回简化结果或关闭非核心功能。

例子：

- 商品详情页隐藏推荐模块。
- 评论列表返回空。
- 短信发送进入异步队列。
- 排行榜返回旧缓存。

### 隔离

用线程池、连接池、队列、机器、集群隔离不同业务，避免一个功能拖垮全站。

## 限流算法

### 固定窗口

```text
每 1 秒最多 1000 次
```

优点是简单，缺点是窗口边界可能产生双倍流量。例如 0.9s 和 1.1s 各打满一次。

### 滑动窗口

把时间窗口切成多个小格，统计最近一段时间请求数。比固定窗口平滑，适合接口 QPS 限制。

### 令牌桶

系统按固定速率生成令牌，请求拿到令牌才放行。桶有容量，所以允许短暂突发。

适合：

- 普通接口限流。
- 允许瞬时流量略高于平均速率。

### 漏桶

请求进入桶，系统按固定速率流出。超过桶容量的请求被拒绝。

适合：

- 保护下游稳定速率。
- 削平突发流量。

### 并发数限制

限制正在执行的请求数。

```text
并发数 = QPS * RT
```

当下游 RT 变长时，即使 QPS 不变，并发数也会上升。并发限制能直接保护线程池和连接池。

## 架构方案

限流降级要分层：

```text
CDN/WAF：静态资源、恶意流量
网关：IP、用户、接口、租户限流
应用：业务维度限流、熔断、降级、隔离
资源层：DB/Redis/MQ/第三方连接池和并发限制
```

不要只在应用里做限流。恶意流量和明显超量请求应尽早挡在入口。

## 配置模型

可以把规则配置化：

```sql
CREATE TABLE traffic_rule (
    id BIGINT PRIMARY KEY,
    rule_code VARCHAR(64) NOT NULL,
    resource VARCHAR(128) NOT NULL,
    dimension VARCHAR(64) NOT NULL,
    limit_type VARCHAR(32) NOT NULL,
    threshold BIGINT NOT NULL,
    window_seconds INT NOT NULL,
    fallback_type VARCHAR(32) NOT NULL,
    enabled TINYINT NOT NULL,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    UNIQUE KEY uk_rule_code (rule_code)
);
```

规则示例：

```text
resource = /api/order/create
dimension = userId
limit_type = sliding_window
threshold = 10
window_seconds = 60
fallback_type = reject
```

规则要支持动态推送和本地缓存，配置中心异常时使用最后一次可用配置。

## 读写链路

### 正常请求链路

```text
请求进入网关
  -> 基础限流和鉴权
  -> 应用层业务限流
  -> 调用下游前检查熔断器
  -> 正常处理
  -> 记录指标
```

### 超限链路

```text
请求命中限流
  -> 返回明确错误码
  -> 可提示稍后重试
  -> 记录限流日志和指标
```

秒杀等场景可以把超限请求导向排队页或静态页。

### 下游异常链路

```text
调用下游超时/错误率升高
  -> 熔断打开
  -> 后续请求快速失败或返回兜底
  -> 半开状态少量探测
  -> 恢复后关闭熔断
```

## 缓存、MQ 与一致性

限流降级常与缓存、MQ 配合：

- 缓存：降级时返回旧值、静态配置、本地缓存。
- MQ：非核心写入异步化，削峰填谷。
- 一致性：降级不能破坏核心状态，支付、扣款、发券等关键操作不能随便返回假成功。

示例：

```text
订单创建：
库存和订单写入不能假成功
短信通知、推荐更新、数据埋点可以异步或降级
```

## 容量估算

限流阈值不能拍脑袋，要来自容量：

```text
接口限流阈值 <= 下游稳定处理能力 * 安全系数
```

示例：

```text
DB 稳定写 TPS = 3000
订单创建会写 3 张表
应用集群安全系数 = 0.6
订单入口写 TPS 上限约 = 3000 / 3 * 0.6 = 600
```

熔断阈值常看：

- 错误率。
- 慢调用比例。
- P99 延迟。
- 并发数。
- 连接池等待时间。

## 风险与降级

### 风险

- 阈值过低：误伤正常用户。
- 阈值过高：保护不了系统。
- 限流维度错误：热点用户绕过或普通用户被误伤。
- 降级返回不当：造成资金、库存、权益错误。
- 熔断抖动：打开关闭频繁。
- 配置错误：全站误限流。

### 降级分级

可以设计多级开关：

```text
L0：正常
L1：关闭推荐、评论、非核心模块
L2：部分接口只读，写请求排队
L3：只保核心下单/支付链路
L4：系统保护，拒绝大部分请求
```

每一级要有触发条件、恢复条件和负责人。

## 常见追问

### 令牌桶和漏桶区别？

令牌桶按固定速率生成令牌，允许桶容量内的短突发；漏桶按固定速率处理请求，更强调平滑输出。

### 限流放网关还是应用？

都要放。网关做通用和粗粒度限流，应用做业务维度限流，例如用户、活动、商品、库存。资源层还要有连接池和并发保护。

### 熔断和降级区别？

熔断是检测下游异常后停止调用，避免故障扩散；降级是返回兜底结果或关闭非核心功能。熔断后通常会触发降级。

### 超限请求应该排队还是拒绝？

看业务。订单、支付等强交互请求一般快速失败或提示重试；秒杀可以短暂排队；后台任务可以排队异步处理。排队也要有长度限制，否则会拖垮系统。

### 如何避免误降级？

规则灰度、生效审批、实时监控、自动回滚、手动开关、配置版本管理。核心资金链路的降级必须谨慎。

## 自检清单

- 是否区分限流、熔断、降级、隔离。
- 是否说明限流维度和阈值来源。
- 是否分层设计入口、应用、资源限流。
- 是否覆盖令牌桶、漏桶、滑动窗口、并发限制。
- 是否有动态配置和本地兜底。
- 是否说明核心链路和非核心链路。
- 是否考虑 MQ 削峰和缓存兜底。
- 是否说明监控、告警、恢复条件。
- 是否避免对资金、库存、权益返回假成功。
