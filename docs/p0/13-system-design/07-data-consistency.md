# 数据一致性

## 面试定位

数据一致性是系统设计里最容易被追问的主题。Java 后端面试中，面试官会从订单支付、库存扣减、优惠券发放、MQ 消息、缓存更新、分布式事务等场景切入，看你是否能区分强一致和最终一致，并给出可落地的事务、幂等、补偿、对账方案。

核心回答：

```text
先识别一致性等级，再选择事务边界；跨系统优先用最终一致、幂等、重试、补偿和对账闭环。
```

## 需求澄清

先问：

- 哪些数据必须强一致，哪些允许最终一致。
- 一致性窗口能接受多久：秒级、分钟级、小时级。
- 是否涉及资金、库存、权益、订单状态。
- 是否跨库、跨服务、跨 MQ、跨第三方。
- 失败后能不能重试，是否有副作用。
- 是否需要对账和人工修复。
- 是否允许用户看到中间态，例如处理中。

## 一致性分类

### 强一致

操作返回成功后，所有读取都能看到最新结果。

适合：

- 余额扣减。
- 支付状态关键更新。
- 订单创建主事务。
- 唯一权益发放。

实现通常依赖单库事务、行锁、唯一索引、条件更新。

### 最终一致

短时间内可能不一致，但经过重试、补偿、消息消费后最终达到一致。

适合：

- 积分到账。
- 消息通知。
- 搜索索引更新。
- 数据分析。
- 缓存更新。

### 读己之写

用户刚写完后，希望自己立即能读到最新值。可以通过主库读、会话缓存、写后短时间绕缓存解决。

## 本地事务

单库事务是最可靠、成本最低的一致性边界。

```java
@Transactional
public void createOrder(CreateOrderCommand command) {
    orderRepository.insert(command.toOrder());
    orderItemRepository.batchInsert(command.toItems());
    orderStatusLogRepository.insertCreated(command.getOrderNo());
}
```

原则：

- 能在单库事务内解决，就不要上分布式事务。
- 事务里不要执行长耗时 RPC。
- 用唯一索引和状态条件更新防并发。
- 事务范围越小越好。

## 跨系统一致性

### Outbox 模式

业务表和消息表同事务，后台异步投递 MQ。

```text
本地事务：
  写订单表
  写 outbox 表
提交成功后：
  投递任务扫描 outbox
  发送 MQ
  消费者幂等消费
```

优点：

- 不依赖 MQ 事务能力。
- 消息可审计、可重试、可重放。

缺点：

- 多一张表和投递任务。
- 消息可能重复，消费者必须幂等。

### 事务消息

流程：

```text
发送半消息
执行本地事务
提交/回滚消息
Broker 异常时回查本地事务状态
```

适合 MQ 原生支持事务消息的场景。仍然要求消费者幂等。

### TCC

```text
Try：预留资源
Confirm：确认提交
Cancel：取消释放
```

适合强业务事务，例如账户冻结、库存预占。缺点是接入成本高，每个参与方都要实现三阶段接口，并处理空回滚、悬挂、幂等等问题。

### Saga

把长事务拆成一系列本地事务，每步失败时执行补偿动作。

适合流程长、允许最终一致的业务，例如订单履约、旅行预订。

风险是补偿不一定完全等价于回滚，需要业务可接受。

## 缓存一致性

缓存与 DB 的一致性常用策略：

```text
更新 DB -> 删除缓存 -> 删除失败重试 / binlog 补偿
```

常见问题：

- 删除缓存失败导致脏数据。
- 并发读写导致旧值回填。
- 本地缓存失效传播慢。

增强手段：

- 短 TTL。
- 延迟双删。
- Binlog 订阅删除缓存。
- 缓存 value 带 version。
- 写后读强制查 DB。

缓存通常只能做到最终一致，核心强一致读要绕过缓存或读主库。

## 库存一致性

库存是高频追问。

### DB 条件扣减

```sql
UPDATE sku_stock
SET available = available - 1
WHERE sku_id = ?
  AND available > 0;
```

优点是强一致，缺点是热点行压力大。

### Redis 预扣 + 异步落库

```text
Redis 原子扣库存
  -> 成功后创建订单
  -> MQ 异步扣 DB 库存
  -> 失败补偿 Redis
```

适合秒杀等极高并发，但要处理 Redis 和 DB 最终一致、订单超时释放、重复扣减、补偿对账。

### 库存流水

用库存流水表达每次变更：

```text
sku_id + biz_no + change_type 唯一
```

重复扣减时唯一键兜底。对账时以流水修复库存。

## 状态一致性

状态机是保证业务一致性的核心。

订单状态示例：

```text
CREATED -> PAYING -> PAID -> FULFILLED -> FINISHED
CREATED -> CLOSED
```

支付回调：

```sql
UPDATE orders
SET status = 'PAID'
WHERE order_no = ?
  AND status IN ('CREATED', 'PAYING');
```

重复回调返回成功，非法状态报警。

## 对账与补偿

只要跨系统，就要准备对账。

对账类型：

- 订单和支付渠道对账。
- 订单和库存流水对账。
- 支付成功和权益发放对账。
- MQ outbox 和消费结果对账。
- DB 和缓存抽样校验。

补偿任务设计：

```text
扫描异常状态
  -> 查询事实源
  -> 判断目标状态
  -> 幂等修复
  -> 记录补偿结果
  -> 超过阈值报警人工处理
```

补偿任务必须幂等，不能重复造成二次伤害。

## 容量估算

一致性方案会增加写放大：

- outbox 表：每个业务事件增加一条消息记录。
- 消费记录表：每个消费组增加一条消费记录。
- 流水表：每次库存/余额变更增加一条流水。
- 对账表：每日或每批次保存对账结果。

估算时要把这些辅助表计入存储和索引成本。

示例：

```text
订单日 1000 万
每单 5 个事件
outbox 日增 5000 万
保留 30 天就是 15 亿记录
需要分表、归档和状态索引治理
```

## 风险与降级

- 分布式事务过重：吞吐下降、锁时间长。
- 最终一致无补偿：异常状态永久挂起。
- 重试无幂等：重复扣款、重复发券。
- 缓存脏读：用户看到旧状态。
- MQ 乱序：状态回退。
- 对账缺失：账实不符无法发现。

降级原则：

- 资金、库存、权益不返回假成功。
- 非核心一致性可以延迟，例如通知、推荐、搜索索引。
- 下游异常时主流程进入处理中，后续补偿。

## 常见追问

### 什么时候用分布式事务？

只有业务要求跨资源强一致，并且无法通过状态机、补偿、最终一致解决时才考虑。优先使用本地事务 + MQ 最终一致。

### 最终一致用户会不会看到错误？

要设计中间态。例如订单支付后权益未到账，可以展示“发放中”，并通过补偿任务保证最终到账。

### MQ 消息发送成功但 DB 事务失败怎么办？

不要在事务提交前直接发送可见消息。使用 outbox 或事务消息，确保本地事务状态和消息可见性一致。

### 缓存和 DB 不一致怎么处理？

DB 是事实源。通过删除缓存、短 TTL、重试任务、binlog 订阅和版本号减少不一致窗口。必要时读 DB。

### 对账发现不一致怎么办？

先以事实源为准确认目标状态，再幂等补偿。资金类要保留审计记录，必要时人工复核。

## 自检清单

- 是否区分强一致、最终一致、读己之写。
- 是否优先使用本地事务。
- 是否解释 outbox、事务消息、TCC、Saga 的适用场景。
- 是否说明 MQ 消费幂等和重复消息。
- 是否覆盖缓存一致性。
- 是否用状态机保证业务状态不乱。
- 是否有补偿、对账、人工处理闭环。
- 是否估算辅助表写放大。
- 是否说明核心链路不能假成功。
