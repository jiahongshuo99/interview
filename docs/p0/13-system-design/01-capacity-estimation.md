# 容量估算：QPS、存储、带宽、峰值

## 面试定位

容量估算是系统设计题的开场能力。面试官不是为了考精确算术，而是看你能不能把业务规模转成工程约束：机器数量、DB 压力、缓存容量、MQ 吞吐、带宽、线程池、连接池和峰值保护。

回答时要先讲方法，再代入数字：

```text
业务量 -> 日请求量 -> 平均 QPS -> 峰值 QPS -> 服务/存储/带宽/缓存/MQ 容量
```

不要一上来拍机器数。先问清业务口径，再保守估算，并说明假设。

## 需求澄清

常问这些问题：

- DAU、MAU、峰值在线人数是多少。
- 每个用户每天触发几次核心请求。
- 读写比例是多少，读多写少还是写多读少。
- 请求是否有明显峰谷，例如午晚高峰、大促、秒杀、整点任务。
- 单次请求/响应平均大小是多少。
- 数据保留多久，是否需要归档。
- 强一致、最终一致、可丢弃数据分别有哪些。
- SLA 是多少，P99 延迟目标是多少。
- 是否存在热点用户、热点商品、热点活动。
- 是否有多地域、多机房、容灾要求。

面试里可以主动说：“我先按日活、行为频次、读写比例、峰值系数来估算，后面再根据瓶颈调整架构。”

## 核心公式

### 平均 QPS

```text
平均 QPS = 日请求量 / 86400
日请求量 = DAU * 每用户每日请求次数
```

例如：

```text
DAU = 1000 万
每人每天访问商品详情 20 次
日请求量 = 2 亿
平均 QPS = 2 亿 / 86400 ≈ 2315
```

### 峰值 QPS

互联网系统不能按平均值设计，通常要乘峰值系数。

```text
峰值 QPS = 平均 QPS * 峰值系数
```

常见峰值系数：

- 普通业务：3 到 5 倍。
- 内容/社交热点：5 到 10 倍。
- 大促/秒杀：10 到 100 倍，需要单独隔离。
- 定时任务集中触发：按任务窗口重新估算，不能套全天平均值。

示例：

```text
平均 QPS = 2315
峰值系数 = 5
峰值 QPS ≈ 11575
```

### TPS 和写入量

TPS 更关注写操作。

```text
写 TPS = 峰值 QPS * 写比例
```

如果订单系统峰值 QPS 10000，提交订单占 10%：

```text
写 TPS = 10000 * 10% = 1000
```

数据库设计要按写 TPS、事务复杂度、索引数量、分库分表策略估算。

### 并发数

用 Little 定律估算：

```text
并发数 = QPS * 平均响应时间秒数
```

例如：

```text
QPS = 10000
平均 RT = 100ms = 0.1s
服务端同时处理请求数 ≈ 1000
```

这能帮助估算 Tomcat 线程池、RPC 线程池、连接池和下游压力。RT 越长，系统需要承载的并发越大。

### 带宽

```text
入口带宽 = QPS * 请求大小
出口带宽 = QPS * 响应大小
```

示例：

```text
峰值 QPS = 10000
平均响应 = 20KB
出口带宽 = 10000 * 20KB = 200MB/s ≈ 1.6Gbps
```

如果响应包含图片、视频、文件，应该通过 CDN/Object Storage 分流，不让应用服务承载大流量文件。

### 存储

```text
业务数据量 = 日新增行数 * 单行大小 * 保留天数
实际存储 = 业务数据量 * 副本数 * 索引膨胀系数
```

常见经验：

- MySQL 二级索引会显著放大存储，索引膨胀系数可按 1.3 到 3 粗估。
- Binlog、备份、从库、归档都要另算。
- Kafka/RocketMQ 按消息大小、保留时间、副本数估算。
- Redis 要考虑 value 大小、key 大小、对象开销、过期策略和内存碎片。

示例：

```text
日订单 1000 万
订单主表单行 1KB
保留 3 年约 1000 天
原始数据 = 1000 万 * 1KB * 1000 ≈ 10TB
考虑索引 2 倍、主从 2 份、备份 1 份，实际可能超过 40TB
```

## 架构估算方法

### 应用服务

估算单机承载能力时，不要只看 CPU。Java 后端常见瓶颈包括：

- 业务线程池满。
- HTTP/RPC 连接池满。
- DB 连接池满。
- GC 抖动。
- 下游接口 RT 变长导致线程堆积。
- 日志同步刷盘或日志量过大。

粗略方法：

```text
需要机器数 = 峰值 QPS / 单机稳态 QPS / 安全系数
```

安全系数通常按 0.5 到 0.7，也就是单机压测 2000 QPS，不建议线上长期打满。

### 数据库

数据库估算要按读写分开：

- 读：能否走索引，是否可以读从库，是否能缓存。
- 写：事务复杂度、索引数量、热点行、分库分表、binlog 压力。
- 查询：分页、排序、聚合是否会扫大表。

如果单表预计超过千万到亿级，要提前考虑：

- 分区或分表。
- 冷热分离。
- 归档策略。
- 查询维度是否需要 ES、ClickHouse、Hive 等分析系统。

### 缓存

缓存容量估算：

```text
缓存内存 = key 数量 * (key 大小 + value 大小 + Redis 对象开销) * 副本数
```

热点缓存要额外考虑：

- 热 key 是否会打爆 Redis 单分片。
- 大 key 是否导致网络和序列化开销过大。
- TTL 是否集中失效。
- 是否需要本地缓存或多级缓存。

### MQ

MQ 容量估算：

```text
消息写入 TPS = 业务写 TPS * 每笔业务消息数
消息存储 = 消息 TPS * 单条消息大小 * 保留秒数 * 副本数
消费能力 = 消费者实例数 * 单实例消费 TPS
```

如果生产快于消费，会出现堆积。需要估算：

- 峰值堆积多久可以追平。
- 消费失败和重试会不会放大流量。
- 是否需要按业务 key 分区保证顺序。

## 示例：短链系统估算

假设：

- DAU 1000 万。
- 每人每天点击短链 10 次。
- 创建短链每天 100 万次。
- 点击读请求峰值系数 5。
- 创建写请求峰值系数 3。
- 短链映射单行 500B，保留 3 年。

读：

```text
日点击 = 1 亿
平均读 QPS = 1 亿 / 86400 ≈ 1157
峰值读 QPS ≈ 5785
```

写：

```text
日创建 = 100 万
平均写 TPS = 100 万 / 86400 ≈ 12
峰值写 TPS ≈ 36
```

存储：

```text
3 年短链数 ≈ 100 万 * 1000 = 10 亿
原始映射数据 ≈ 10 亿 * 500B = 500GB
考虑索引、副本、备份，实际按 2TB 以上规划
```

结论：

- 读远大于写，核心是缓存和跳转链路低延迟。
- 写 TPS 不高，但数据总量大，需要分表或按时间归档。
- 热门短链需要本地缓存或 Redis 热 key 保护。

## 风险与降级

容量估算必须落到保护动作：

- 入口限流：按接口、用户、IP、活动维度限流。
- 熔断降级：下游慢时快速失败或返回兜底。
- 异步削峰：非核心写入放 MQ。
- 缓存预热：热点数据提前加载。
- 开关控制：大促前准备降级开关。
- 扩容预案：机器、DB、Redis、MQ 分别有扩容路径。
- 压测验证：用压测结果修正估算，不迷信纸面数字。

## 常见追问

### 为什么不能只按平均 QPS 设计？

用户访问有峰谷，系统故障也常发生在峰值。平均 QPS 会掩盖瞬时压力，尤其是秒杀、推送、定时任务、大促入口。

### 单机能扛多少 QPS？

不能给固定值，要看业务逻辑、下游依赖、RT、序列化、日志、缓存命中率、机器配置和 JVM 参数。面试里可以说先压测单机稳态 QPS，再按 50% 到 70% 作为线上安全水位。

### QPS 高一定要分库分表吗？

不一定。读高可以先缓存、读写分离、索引优化。写高、单表数据量大、热点行严重或单库写入达到瓶颈时才考虑分库分表。

### 带宽估算容易漏什么？

容易漏响应体、图片文件、日志上报、MQ 消息、跨机房复制和备份流量。大文件要走对象存储和 CDN。

### 峰值系数怎么选？

没有历史数据时按业务类型保守估计；有历史数据时看 P95/P99 时段流量、活动峰值、节假日峰值和失败重试放大。

## 自检清单

- 是否先说明了所有关键假设。
- 是否分别估算了读 QPS、写 TPS、峰值 QPS。
- 是否估算了存储、索引、副本、备份。
- 是否估算了带宽和响应体大小。
- 是否把 RT 和并发数联系起来。
- 是否说明缓存、DB、MQ 的容量边界。
- 是否说明单机压测和安全水位。
- 是否考虑了峰值、热点、重试、补偿和定时任务。
- 是否给出限流、降级、扩容和压测验证方案。
