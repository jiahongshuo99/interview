# 热点数据处理

## 面试定位

热点数据处理考察高并发系统的局部瓶颈治理能力。面试官常从热点商品、热点用户、热点库存、热点缓存 Key、热门短链、明星动态、秒杀库存等场景追问，看你是否能识别热点类型，并设计缓存、拆分、限流、异步、隔离和降级。

核心观点：

```text
系统整体 QPS 不高也可能被一个热点打垮，热点治理要定位到 key、行、分片、队列和资源。
```

## 需求澄清

先问：

- 热点对象是什么：商品、库存、用户、短链、文件、活动。
- 热点是读热点还是写热点。
- 热点是否可预测，例如大促活动。
- 热点持续时间多长。
- 是否允许返回旧数据。
- 是否必须强一致。
- 单个热点 key 的峰值 QPS/TPS 是多少。
- 热点是否会导致 DB 行锁、Redis 分片、MQ 分区或应用线程池瓶颈。

## 热点类型

### 热点 Key

大量请求访问同一个 Redis key。

现象：

- Redis 单分片 CPU 飙升。
- 某个 key 网络流量极高。
- Redis 响应变慢但整体内存正常。

### 热点行

大量写请求更新同一行 DB，例如单个 SKU 库存。

现象：

- 行锁等待。
- 更新 TPS 上不去。
- DB CPU 不高但事务等待高。

### 热点分片

分库分表或 Redis Cluster 中，流量集中到某个分片。

原因：

- 分片键低基数。
- 热门用户/商品集中。
- hash tag 使用不当。

### 热点队列

MQ 按业务 key 分区后，某个 key 的消息过多，导致一个分区堆积。

### 热点大对象

热门文件、图片、视频、下载包被集中访问，应用带宽被打满。

## 发现热点

监控指标：

- Redis hotkey、bigkey。
- DB 慢 SQL、行锁等待、热点索引。
- 接口维度 QPS、P99、错误率。
- 业务 ID 维度 TopN。
- MQ 分区堆积。
- CDN 命中率和回源流量。

日志要带业务 key：

```text
traceId, userId, skuId, activityId, shortCode, cacheKey, orderNo
```

没有业务 key，线上很难定位热点。

## 读热点方案

### 多级缓存

```text
本地缓存 Caffeine
  -> Redis 副本 key
  -> DB
```

适合读多写少、允许短暂旧值的数据。

本地缓存要注意：

- TTL 短。
- 最大容量。
- 版本号。
- 失效通知丢失时 TTL 兜底。

### 热点 Key 副本

把一个 key 拆成多个只读副本：

```text
product:1001:0
product:1001:1
product:1001:2
```

读请求随机选择副本，分散 Redis 压力。

写更新时删除或更新所有副本，失败用补偿。

### 逻辑过期

缓存物理不过期或长 TTL，value 内包含逻辑过期时间。过期后异步刷新，读请求继续返回旧值。

适合：

- 热点商品详情。
- 活动配置。
- 榜单。

不适合强一致库存、余额。

### CDN / 对象存储

静态文件和大对象必须从应用层剥离：

```text
客户端 -> CDN -> 对象存储
```

应用只负责鉴权、签名 URL、元数据。

## 写热点方案

### 条件更新

DB 强一致扣减：

```sql
UPDATE sku_stock
SET available = available - 1
WHERE sku_id = ?
  AND available > 0;
```

简单可靠，但热点 SKU 会成为行锁瓶颈。

### 库存分桶

把一个热点库存拆成多个桶：

```text
sku_stock_bucket(sku_id, bucket_no, available)
```

扣减时随机或按用户 hash 选择桶：

```text
bucket = hash(userId) % bucketCount
```

优点是降低单行锁竞争。缺点是总库存统计、补偿和余量合并更复杂。

### Redis 原子扣减

秒杀场景常用 Redis 扣库存：

```lua
local stock = tonumber(redis.call('get', KEYS[1]))
if stock <= 0 then
    return -1
end
redis.call('decr', KEYS[1])
return stock - 1
```

成功后发 MQ 异步落库。需要处理订单超时释放、重复扣减、Redis 和 DB 对账。

### 写聚合

对于计数、点赞、浏览量：

```text
应用本地聚合 / Redis 聚合
  -> 定时批量刷 DB
```

适合可丢少量、可最终一致的指标，不适合资金和库存。

## 架构方案

热点治理分层：

```text
入口：识别热点对象，按对象限流
应用：本地缓存、请求合并、隔离线程池
缓存：热点 key 副本、逻辑过期、预热
DB：分桶、条件更新、归档、读写分离
MQ：按 key 分区、增加分区、拆分 topic
运维：热点发现、动态配置、自动降级
```

## 读写链路

### 热点商品详情

```text
请求商品详情
  -> 判断商品是否热点
  -> 本地缓存命中直接返回
  -> Redis 副本 key 命中返回
  -> 互斥重建缓存
  -> DB 查询
```

### 热点库存扣减

```text
请求下单
  -> 用户/活动限流
  -> Redis 原子预扣库存
  -> 成功后创建订单
  -> MQ 异步扣 DB 或记录库存流水
  -> 超时未支付释放库存
```

## 缓存、MQ 与一致性

读热点可以牺牲短暂一致性换性能；写热点必须看业务类型：

- 库存：不能超卖，可 Redis 原子扣减 + DB 对账。
- 余额：必须强一致，不适合纯缓存扣减。
- 点赞数：可异步聚合，允许最终一致。
- 短链跳转：缓存旧映射通常可接受，但删除/封禁要更快生效。

MQ 在热点场景有两个作用：

- 削峰，把瞬时写入变成后台消费。
- 串行化同一 key 操作，降低并发冲突。

代价是延迟和堆积。

## 容量估算

热点不能只看全站 QPS。

示例：

```text
全站峰值 QPS = 10000
某热点商品占 40%
该商品详情 QPS = 4000
如果都打 Redis 同一 key，单分片会承受 4000 QPS 和网络流量
```

对于写热点：

```text
秒杀 SKU 请求 QPS = 50000
库存只有 1000
真正成功写订单最多 1000
应在入口和 Redis 层拦截绝大多数失败请求
```

## 风险与降级

- 热点发现不及时：要有 TopN 监控和自动识别。
- 本地缓存不一致：短 TTL 和版本号兜底。
- 热点副本更新漏删：补偿任务和短 TTL。
- 分桶库存不均：随机/哈希策略和余量合并。
- Redis 扣减成功但订单失败：释放库存和对账。
- MQ 分区热点：按更细粒度拆 key 或拆 topic。
- 热点大文件回源：CDN 预热和防盗链。

降级：

- 热点详情只返回核心字段。
- 评论、推荐、画像等模块关闭。
- 下单入口排队或限流。
- 库存售罄后本地快速失败。

## 常见追问

### 热点 key 怎么自动发现？

Redis hotkey 统计、客户端埋点 TopN、代理层统计、日志聚合都可以。面试里要强调业务 key 维度监控。

### 热点库存怎么防超卖？

用原子扣减和幂等流水。Redis 扣减要用 Lua 保证原子，DB 落库要有库存流水和对账；DB 扣减用条件更新 `available > 0`。

### 本地缓存会不会脏？

会，所以只适合允许短暂旧值的数据。用短 TTL、版本号、失效通知和主动刷新降低风险。

### 热点 key 副本怎么更新？

写后删除所有副本，失败重试；或使用版本号让旧副本失效。副本数量要可配置，热点结束后回收。

### 分桶库存剩余不均怎么办？

可以随机探测多个桶，或后台把余量合并调整。分桶提升吞吐，但会增加库存管理复杂度。

## 自检清单

- 是否区分读热点和写热点。
- 是否定位到 key、行、分片、队列、大对象。
- 是否说明热点发现和监控。
- 是否有多级缓存、本地缓存、热点副本。
- 是否有写热点分桶、原子扣减、异步削峰。
- 是否说明缓存和 DB 的一致性边界。
- 是否考虑 MQ 分区热点。
- 是否有热点降级和限流。
- 是否按单热点 QPS 估算，而不是只看全站平均值。
