# 缓存设计

## 面试定位

缓存设计是 Java 后端系统设计里的高频能力题。面试官关注的不只是“加 Redis”，而是你能不能说清缓存适用场景、Key 设计、读写链路、一致性、穿透击穿雪崩、热点 Key、大 Key、容量、降级和监控。

核心原则：

```text
缓存提升读性能，但会引入一致性、容量、热点和失效风险。
```

## 需求澄清

先问清：

- 缓存什么数据：配置、商品详情、用户信息、库存、权限、排行榜。
- 读写比例是多少，是否读多写少。
- 数据一致性要求：强一致、秒级最终一致、分钟级可接受。
- 是否有热点 key 或突发活动。
- 数据大小和 key 数量级是多少。
- 是否允许返回旧值。
- 缓存失效时 DB 能不能扛住。
- 是否需要本地缓存、Redis、CDN 多级缓存。

## 核心模型

### Cache Aside

最常用模式：

```text
读：先查缓存，miss 后查 DB，再回填缓存
写：先更新 DB，再删除缓存
```

优点是简单可控，缺点是有短暂不一致，需要删除失败补偿。

Java 示例：

```java
public ProductDTO getProduct(long productId) {
    String key = "product:" + productId;
    String cached = redis.get(key);
    if (cached != null) {
        return JsonUtils.fromJson(cached, ProductDTO.class);
    }

    ProductDTO product = productRepository.findById(productId);
    if (product == null) {
        redis.setex(key, 60, "__NULL__");
        return null;
    }

    redis.setex(key, 600 + ThreadLocalRandom.current().nextInt(60), JsonUtils.toJson(product));
    return product;
}
```

### Read Through / Write Through

由缓存组件统一负责读穿和写穿，业务代码更简单，但对组件能力要求高。Java 后端常见还是 Cache Aside。

### Write Behind

先写缓存，再异步刷 DB。吞吐高，但一致性风险大，适合日志、计数等可补偿场景，不适合订单、支付、余额。

### 多级缓存

```text
本地缓存 Caffeine
  -> Redis
  -> DB
```

适合极热点、低一致性要求的数据，例如配置、商品基础信息、活动规则。风险是本地缓存失效传播慢，要有版本号、短 TTL 或订阅失效通知。

## Key 设计

Key 要可读、可控、可治理：

```text
业务前缀:实体:ID[:版本]
product:detail:1001
user:profile:9527
coupon:stock:activityId:couponId
```

注意：

- 前缀区分业务域，避免冲突。
- key 长度不要过长。
- 不把手机号、身份证等敏感信息直接放 key。
- 包含租户、环境、版本时要有固定顺序。
- 批量删除要能通过业务索引或版本号处理，不依赖线上 `keys` 命令。

## 数据模型

### 字符串缓存

适合对象序列化：

```text
product:detail:1001 -> JSON
```

优点是简单，缺点是局部字段更新需要整体序列化。

### Hash 缓存

适合字段级读取和更新：

```text
HSET user:profile:9527 name age level
```

注意 Hash 过大也会变成大 key。

### Set / ZSet

适合关系和排序：

- 用户关注集合。
- 排行榜。
- 活动参与用户。

### Bitmap / HyperLogLog

适合签到、UV 估算等空间敏感场景。

## 读链路

标准读链路：

```text
请求
  -> 参数校验
  -> 查本地缓存
  -> 查 Redis
  -> 查 DB
  -> 回填 Redis
  -> 回填本地缓存
```

关键点：

- 缓存 miss 后要保护 DB。
- 回填加随机 TTL，避免集中失效。
- 空结果也可短 TTL 缓存。
- 热点 key 可以逻辑过期或主动刷新。
- 缓存反序列化失败要删除异常 key 并降级查 DB。

## 写链路

推荐：

```text
更新 DB 成功
  -> 删除缓存
  -> 删除失败写重试任务
  -> 订阅 binlog 或补偿任务兜底
```

为什么不是先删缓存再更新 DB：

- 先删缓存后，另一个请求可能读到旧 DB 并回填旧缓存。
- 更新 DB 后删缓存，短暂不一致窗口更小。

为什么通常删除缓存而不是更新缓存：

- 更新缓存容易漏字段、漏组合 key。
- 多个缓存副本更新顺序复杂。
- 删除后由读请求按最新 DB 回填，更简单。

## 一致性设计

常见方案：

| 方案 | 一致性 | 成本 | 适用场景 |
| --- | --- | --- | --- |
| 更新 DB 后删缓存 | 最终一致 | 低 | 通用 |
| 延迟双删 | 最终一致 | 中 | 并发读写高 |
| Binlog 订阅删缓存 | 最终一致 | 中高 | 多服务共写 |
| 版本号缓存 | 防旧值覆盖 | 中 | 高并发回填 |
| 强制读 DB | 强一致 | 高 | 支付、余额等关键路径 |

版本号思路：

```text
DB 中有 version / updated_at
缓存 value 带 version
回填缓存前比较版本，避免旧请求覆盖新值
```

## 穿透、击穿、雪崩

### 缓存穿透

不存在的数据持续打到 DB。

方案：

- 参数校验。
- 缓存空值，短 TTL。
- 布隆过滤器。
- 网关或应用限流。

### 缓存击穿

热点 key 过期瞬间大量请求打到 DB。

方案：

- 互斥锁重建。
- 逻辑过期异步刷新。
- 热点 key 不过期，后台主动刷新。
- 本地缓存兜底。

### 缓存雪崩

大量 key 同时失效或 Redis 整体不可用。

方案：

- TTL 加随机抖动。
- 缓存预热。
- Redis 高可用集群。
- 限流降级保护 DB。
- 核心数据本地缓存兜底。

## 热点 Key 和大 Key

### 热点 Key

现象：

- 单个 Redis 分片 CPU 高。
- 某个 key QPS 极高。
- 网络流量集中。

方案：

- 本地缓存。
- 热点 key 拆分副本，例如 `hot:product:1001:0..N`。
- 读请求随机打到副本。
- 提前预热，后台刷新。
- 入口限流。

### 大 Key

风险：

- 删除阻塞。
- 网络传输慢。
- 序列化反序列化慢。
- 主从同步压力大。

方案：

- 拆分 value。
- 使用分页结构。
- 控制集合元素数量。
- 删除用异步 unlink。
- 监控 bigkey。

## 容量估算

估算公式：

```text
缓存内存 = key 数量 * (key 大小 + value 大小 + Redis 对象开销) * 副本数
```

示例：

```text
商品 1000 万
每个商品缓存 2KB
原始 value 约 20GB
考虑 key、对象开销、碎片、副本，至少按 50GB 到 80GB 规划
```

还要估算：

- 峰值 QPS。
- 命中率。
- Redis 单分片吞吐。
- 网络带宽。
- 本地缓存 JVM 内存和 GC 影响。

## 风险与降级

- Redis 故障：核心链路限流，非核心返回兜底。
- DB 被打穿：熔断缓存 miss 请求。
- 缓存脏读：关键操作强制查 DB。
- 反序列化失败：删除异常 key，记录监控。
- 热点过载：本地缓存和副本 key。
- 内存不足：淘汰策略、容量扩容、key 生命周期治理。

降级示例：

```text
商品详情页：
核心字段从本地缓存返回
库存/评价/推荐模块降级为空
后台异步刷新缓存
```

## 常见追问

### 更新 DB 后删除缓存失败怎么办？

记录重试任务，或订阅 binlog 补偿删除。核心数据可以使用短 TTL 降低脏数据持续时间。

### Redis 能不能保证强一致？

通常不能。缓存是性能层，不是事实源。强一致场景要以 DB/事务为准，必要时绕过缓存。

### 本地缓存怎么失效？

短 TTL、版本号、消息通知、配置中心推送都可以。要考虑通知丢失，所以仍要有 TTL 兜底。

### 互斥锁重建缓存有什么坑？

锁要有过期时间，解锁要校验 token，拿锁后 double check，等待方要有超时和降级，避免锁服务故障导致雪崩。

### 缓存命中率低怎么办？

先看 key 设计、TTL、访问分布、数据是否适合缓存。低频随机数据不适合缓存；热点数据可以预热和延长 TTL。

## 自检清单

- 是否说明缓存对象、读写比例和一致性要求。
- 是否选择了合适缓存模式。
- 是否有清晰 key 设计。
- 是否说明读链路、写链路和删除失败补偿。
- 是否覆盖穿透、击穿、雪崩。
- 是否考虑热点 key、大 key。
- 是否估算内存、QPS、带宽。
- 是否说明 Redis 故障时的限流降级。
- 是否强调 DB 是事实源，缓存是性能层。
