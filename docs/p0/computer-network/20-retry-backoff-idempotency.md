# 重试、退避、幂等

## 面试定位

重试、退避、幂等是网络故障和分布式系统的交叉重点。面试官常问：哪些错误可以重试、为什么要指数退避、POST 能不能重试、幂等键怎么设计、重试风暴如何避免。

核心回答：

重试用于处理瞬时故障，但必须受超时预算、退避、最大次数、熔断和幂等约束。没有幂等保障的写操作不能盲目重试。

## 为什么需要重试

网络和下游故障可能是短暂的：

- 单次丢包。
- 连接被重置。
- 实例滚动重启。
- 负载均衡切换。
- 网关短暂 502/503/504。
- DNS 或连接池拿到失效连接。

有限重试可以提高成功率。

## 重试的风险

重试不是免费能力：

- 放大请求量。
- 增加下游压力。
- 拉长上游耗时。
- 造成重复写入。
- 引发重试风暴。
- 掩盖真实故障。

因此重试必须受控。

## 哪些错误适合重试

通常可考虑重试：

- 连接超时。
- 连接重置。
- 连接池复用失效连接。
- 502、503、504。
- 429，遵守 `Retry-After`。
- 部分读超时，前提是请求幂等。

通常不应重试：

- 400 参数错误。
- 401 认证失败。
- 403 无权限。
- 404 资源不存在，除非有最终一致性场景。
- 明确业务校验失败。
- 非幂等写操作且没有幂等键。

## 幂等

幂等指同一个操作执行一次和执行多次的最终效果一致。

常见幂等方法：

- GET：通常幂等。
- PUT：按资源完整更新，通常幂等。
- DELETE：删除同一资源，通常幂等。
- POST：默认不幂等，需要业务设计。

注意：

幂等不等于响应完全相同。第二次删除可能返回 404 或业务码，但资源最终状态仍是“已删除”。

## 幂等键设计

对创建订单、发券、扣款等写操作，需要幂等键。

常见方案：

```text
idempotency_key = 业务方 + 操作类型 + 请求唯一号
```

服务端处理：

1. 收到请求，检查幂等键是否存在。
2. 不存在则创建处理记录，状态为处理中。
3. 执行业务操作。
4. 保存结果。
5. 重复请求返回第一次结果或当前状态。

关键点：

- 幂等键必须由调用方稳定生成。
- 幂等记录要有唯一约束。
- 处理中状态要有超时和补偿。
- 幂等结果要能覆盖成功、失败和处理中。

## 退避

如果失败后立即重试，可能加剧拥塞。退避让重试间隔逐步增加。

固定间隔：

```text
100ms, 100ms, 100ms
```

指数退避：

```text
100ms, 200ms, 400ms
```

加抖动：

```text
100ms +/- random
200ms +/- random
400ms +/- random
```

抖动可以避免大量客户端同时重试形成流量尖峰。

## 重试预算

重试必须纳入整体超时。

错误示例：

```text
单次 read timeout 2s
重试 3 次
总耗时可能 6s+
上游预算只有 3s
```

推荐：

```text
整体超时 3s
最多重试 1 次
单次超时 1s
退避 100ms
超过预算直接失败
```

## 重试位置

重试可能发生在多层：

- SDK 客户端。
- RPC 框架。
- 网关。
- 业务代码。
- 消息队列消费。

风险是多层叠加：

```text
业务重试 3 次
SDK 重试 3 次
网关重试 2 次
实际最多 18 次
```

需要统一重试策略，避免重复放大。

## Java 示例

伪代码：

```java
Duration deadline = Duration.ofSeconds(3);
int maxAttempts = 2;

for (int attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
        return callDownstream();
    } catch (IOException e) {
        if (attempt == maxAttempts || !isRetryable(e)) {
            throw e;
        }
        sleepWithJitter(attempt);
    }
}
```

要点：

- 判断异常是否可重试。
- 判断请求是否幂等。
- 遵守整体 deadline。
- 记录 attempt 次数和最终错误。

## 线上排查

### 下游故障时流量暴涨

可能是重试风暴。

排查：

- 看入口 QPS 和下游 QPS 是否倍增。
- 看重试次数指标。
- 看错误码是否集中。
- 看是否多层重试叠加。

治理：

- 限制最大重试次数。
- 指数退避和抖动。
- 熔断。
- 限流。
- 快速失败。

### 重试导致重复订单或重复发奖

说明写操作缺幂等。

排查：

- 是否有唯一业务键。
- 是否有幂等表。
- 是否保存首次处理结果。
- 是否存在并发重复请求。

治理：

- 唯一约束。
- 幂等记录。
- 分布式锁，谨慎使用。
- 状态机防重复流转。

## 常见追问

### 读超时能不能重试

要看操作是否幂等。读超时说明客户端没等到结果，但服务端可能已经执行成功。非幂等写操作不能盲目重试。

### 连接超时为什么相对适合重试

连接超时通常发生在请求未成功发送到服务端之前，但也不能绝对保证。实际仍要结合库语义和业务幂等。

### 退避为什么要加随机抖动

避免大量客户端按相同间隔同时重试，造成周期性流量尖峰。

### MQ 消费重试也需要幂等吗

需要。消息通常至少投递一次，消费者必须能处理重复消息。

## 易错点

- 认为所有 5xx 都可以无限重试。
- 忽略读超时后服务端可能已成功。
- 多层重试叠加导致放大。
- 没有整体超时预算。
- 幂等键用随机 UUID 每次生成，导致重试无法命中同一记录。

## 自检清单

- 能否列出适合和不适合重试的错误。
- 能否解释指数退避和抖动。
- 能否说明幂等键如何设计。
- 能否识别读超时重试风险。
- 能否设计有超时预算的重试策略。
