# TCP 可靠传输

## 面试定位

TCP 可靠传输是 TCP 题目的主干。面试官通常会追问：TCP 为什么可靠、可靠是不是绝对不丢、序号和 ACK 怎么工作、丢包怎么发现、滑动窗口解决什么、流量控制和拥塞控制有什么区别。

回答时建议用一句话开头：

TCP 在不可靠的 IP 之上，通过序号、确认、重传、滑动窗口、校验和、流量控制、拥塞控制等机制，实现面向连接的可靠有序字节流。

## 核心机制

TCP 可靠性由多种机制共同保证：

| 机制 | 作用 |
| --- | --- |
| 序号 | 标识字节流位置，支持排序和去重 |
| 确认应答 | 接收方告诉发送方已经收到哪些字节 |
| 超时重传 | ACK 未按时到达时重发 |
| 快速重传 | 收到重复 ACK 后提前判断丢包 |
| 校验和 | 发现传输过程中的比特错误 |
| 滑动窗口 | 支持批量发送，提高吞吐 |
| 流量控制 | 避免发送方压垮接收方 |
| 拥塞控制 | 避免发送方压垮网络 |
| SACK | 精确告知已收到的非连续数据块 |

## TCP 是可靠字节流

TCP 不保证“消息”可靠，它保证字节流可靠有序。

例如应用连续写两次：

```java
out.write("hello".getBytes());
out.write("world".getBytes());
```

对端可能一次读到 `helloworld`，也可能分多次读到。这是正常的，因为 TCP 没有消息边界。

自定义协议必须定义边界：

- 固定长度头。
- 长度字段。
- 分隔符。
- HTTP 的 Content-Length 或 chunked。

## 序号和确认号

TCP 对每个字节编号。假设发送方发送 100 字节，起始序号为 1000：

```text
seq = 1000, len = 100
```

接收方如果连续收到这些字节，会回复：

```text
ack = 1100
```

ACK 表示期望收到的下一个字节序号，具有累计确认语义。

累计确认例子：

```text
收到 1000-1099 -> ack 1100
收到 1100-1199 -> ack 1200
```

如果 1100-1199 丢了，但 1200-1299 到了，普通 ACK 仍会回 `ack = 1100`，表示 1100 之前连续字节已收到。

## 重传

TCP 发现丢包的主要方式：

- 超时重传：发送后超过 RTO 没收到 ACK。
- 快速重传：收到多个重复 ACK。
- SACK 辅助：接收方告诉发送方哪些区间已经收到。

重传保证最终能补齐丢失字节，但代价是延迟升高和吞吐下降。

## 有序交付

IP 层可能乱序。TCP 接收方会缓存乱序到达的数据，等缺口补齐后再按序交给应用。

例如：

```text
已收到 1000-1099
收到 1200-1299，但 1100-1199 未到
```

TCP 不能把 1200-1299 直接交给应用，因为字节流必须有序。等 1100-1199 到达后，才一起向上交付。

## 去重

重传可能导致接收方收到重复数据。TCP 根据序号判断重复段，丢弃重复内容，同时继续发送 ACK。

这也是 TCP 可靠性的一个重要部分：应用不会读到重复字节。

## 校验和

TCP Header 中有校验和，覆盖 TCP 首部、数据和伪首部。接收方校验失败会丢弃该段。

注意：

- 校验和只能发现错误，不负责纠错。
- 被丢弃的数据需要依赖重传恢复。

## 流量控制和拥塞控制

两者都限制发送速度，但目标不同：

| 机制 | 保护对象 | 核心变量 | 反馈来源 |
| --- | --- | --- | --- |
| 流量控制 | 接收方 | 接收窗口 rwnd | 接收方通告窗口 |
| 拥塞控制 | 网络 | 拥塞窗口 cwnd | 丢包、延迟、ACK |

发送方实际可发送窗口通常受两者共同约束：

```text
send_window = min(rwnd, cwnd)
```

## 可靠不等于业务一定成功

TCP 可靠只说明字节尽力可靠传到对端应用的 Socket 缓冲区，不代表：

- HTTP 请求一定成功。
- 服务端一定处理成功。
- 数据库事务一定提交。
- 客户端一定收到业务响应。

典型场景：客户端读超时后重试，但服务端可能已经处理成功。这就是重试必须考虑幂等的原因。

## 抓包和命令示例

观察序号和 ACK：

```bash
sudo tcpdump -i any host 203.0.113.20 and tcp port 8080 -nn -vv
```

保存后用 Wireshark 分析：

```bash
sudo tcpdump -i any host 203.0.113.20 and port 8080 -w /tmp/tcp.pcap
```

查看 TCP 连接细节：

```bash
ss -ti dst 203.0.113.20
```

模拟丢包：

```bash
sudo tc qdisc add dev eth0 root netem loss 5%
sudo tc qdisc del dev eth0 root
```

## 线上排查

TCP 可靠性相关问题通常表现为：

- 请求耗时抖动。
- P99 延迟升高。
- 连接没有断，但吞吐下降。
- 抓包看到重传、重复 ACK、窗口变小。
- `ss -ti` 中 rto、retrans、cwnd 异常。

排查方向：

- 网络是否丢包。
- 链路 RTT 是否升高。
- 接收端是否读得太慢。
- 发送端是否写入过快。
- 网卡或宿主机是否有丢包。
- 容器网络、代理、LB 是否引入重传。

## 常见追问

### TCP 会不会丢数据

应用视角下，正常连接中 TCP 会通过重传尽力保证数据可靠到达。如果连接最终断开，未确认的数据可能传输失败，应用需要通过错误和重试处理。

### ACK 是对包确认还是对字节确认

对字节序号确认。ACK 表示期望收到的下一个字节序号。

### TCP 能否保证数据不重复

能在字节流层面去重。重复到达的 TCP 段会被识别并丢弃，不会重复交给应用。

### TCP 能否保证消息边界

不能。TCP 只保证字节流有序可靠，消息边界由应用协议定义。

## 易错点

- 把 TCP 可靠理解成请求一定成功。
- 把 ACK 理解成确认“包编号”，而不是字节序号。
- 把流量控制和拥塞控制混为一谈。
- 认为 TCP 没有粘包问题。准确说 TCP 是字节流，应用层需要处理边界。
- 只看应用超时，不看底层重传和窗口。
