# 拥塞控制

## 面试定位

拥塞控制是 TCP 中最容易和流量控制混淆的题。面试常问：拥塞控制保护什么、慢启动和拥塞避免怎么工作、快重传和快恢复是什么、丢包为什么会降低窗口、线上吞吐下降如何定位。

核心回答：

TCP 拥塞控制通过动态调整拥塞窗口 cwnd 控制发送速率，避免过多数据注入网络导致链路拥塞和大面积丢包。

## 拥塞控制保护网络

流量控制看接收方能不能接住，拥塞控制看网络能不能承载。

发送方实际可发送数据量：

```text
send_window = min(rwnd, cwnd)
```

其中：

- `rwnd`：接收方通告窗口。
- `cwnd`：发送方根据网络状况维护的拥塞窗口。

## 慢启动

连接刚开始时，发送方不知道网络容量，不能一上来就高速发送。慢启动从较小 cwnd 开始，随 ACK 增长。

简化理解：

```text
cwnd = 1 MSS
每经过一个 RTT，cwnd 近似翻倍
1 -> 2 -> 4 -> 8 -> 16
```

慢启动不是速度慢，而是从小窗口开始指数增长。

## 拥塞避免

当 cwnd 增长到慢启动阈值 `ssthresh` 后，进入拥塞避免，窗口线性增长。

```text
cwnd 达到 ssthresh
  -> 从指数增长改为线性增长
```

目的是更平滑地探测网络容量，避免增长过快造成拥塞。

## 拥塞发生

TCP 通常通过丢包或显著延迟判断拥塞。

常见信号：

- 超时重传。
- 多个重复 ACK 触发快速重传。
- ECN 显式拥塞通知，取决于网络支持。

不同信号的处理不同：

- 超时重传通常认为拥塞严重，cwnd 大幅下降。
- 快速重传通常认为只是局部丢包，进入快速恢复。

## 快速重传和快速恢复

当发送方收到多个重复 ACK，说明后续数据到达了，但中间某段丢了。发送方不等 RTO，直接重传缺失段。

快速恢复会降低 cwnd，但不会像超时那样完全重新开始，尽量保持一定吞吐。

经典流程简化：

```text
收到 3 个重复 ACK
  -> ssthresh = cwnd / 2
  -> 重传丢失段
  -> cwnd 降低并进入快速恢复
  -> 收到新 ACK 后进入拥塞避免
```

## 常见拥塞控制算法

不同系统可能使用不同算法：

- Reno：经典算法，面试常见。
- CUBIC：Linux 常用算法，适合高带宽长 RTT 链路。
- BBR：基于带宽和 RTT 估算瓶颈，目标是降低排队延迟并提升吞吐。

查看 Linux 当前算法：

```bash
sysctl net.ipv4.tcp_congestion_control
sysctl net.ipv4.tcp_available_congestion_control
```

面试时不需要深入推导 BBR，但可以说明不同算法对丢包、带宽、RTT 的响应策略不同。

## 拥塞窗口和吞吐

吞吐近似受窗口和 RTT 影响：

```text
throughput ≈ window / RTT
```

如果 RTT 高、cwnd 小，吞吐会明显下降。跨地域调用、大文件传输、HTTP/2 单连接多流都容易受影响。

## 和应用限流的区别

拥塞控制是 TCP 传输层机制，自动根据网络反馈调节。

应用限流是业务或网关策略，例如：

- QPS 限制。
- 并发限制。
- 令牌桶。
- 熔断降级。

两者目标不同，但都可能表现为吞吐受限或延迟上升。

## 命令示例

查看连接拥塞信息：

```bash
ss -ti dst 203.0.113.20
```

可能看到：

```text
cubic wscale:7,7 rto:204 rtt:20.5/3.1 cwnd:10 ssthresh:20 retrans:0/2
```

查看系统拥塞控制算法：

```bash
sysctl net.ipv4.tcp_congestion_control
```

抓包分析重复 ACK 和重传：

```bash
sudo tcpdump -i any host 203.0.113.20 and tcp port 8080 -w /tmp/congestion.pcap
```

Wireshark 过滤：

```text
tcp.analysis.duplicate_ack
tcp.analysis.fast_retransmission
tcp.analysis.retransmission
```

## 线上排查

### 吞吐突然下降

检查：

- 是否有丢包和重传。
- RTT 是否升高。
- cwnd 是否降低。
- 是否某条链路或某个机房异常。
- 是否服务端或客户端网卡丢包。

### P99 延迟升高

拥塞可能导致：

- 排队延迟增加。
- RTO 触发长尾。
- HTTP/2 连接内多个请求同时变慢。

排查要结合：

- 应用耗时拆分。
- 网络 RTT。
- 重传率。
- 网关和 LB 指标。

### 跨机房调用慢

跨机房通常 RTT 更高，同样窗口下吞吐更低。可以考虑：

- 减少跨机房同步调用。
- 就近访问。
- 批量压缩。
- 合理并发。
- 连接池和 HTTP/2 并发流配置。

## 常见追问

### 慢启动为什么叫慢启动

因为它不是一开始就按最大速率发送，而是从小窗口开始探测网络容量。虽然增长是指数级，但初始发送量保守。

### 丢包一定代表拥塞吗

不一定。丢包可能来自链路错误、设备问题、策略丢弃。但 TCP 传统拥塞控制常把丢包当拥塞信号处理。

### 拥塞控制和流量控制哪个优先生效

发送方实际受 `min(rwnd, cwnd)` 限制，谁小谁成为瓶颈。

### BBR 和 CUBIC 区别

CUBIC 主要基于丢包反馈调节窗口，BBR 试图估算瓶颈带宽和最小 RTT。实际使用取决于内核、网络和业务场景。

## 易错点

- 把拥塞控制说成防止接收方缓冲区满。
- 只背慢启动、拥塞避免，不知道 cwnd 和 ssthresh。
- 认为丢包后 TCP 只重传，不会影响发送速率。
- 只看 QPS 降低，不看 RTT、重传和 cwnd。
- 认为换成 HTTP/2 就不会受 TCP 拥塞影响。

## 自检清单

- 能否说明拥塞控制保护网络。
- 能否解释慢启动、拥塞避免、快重传、快恢复。
- 能否区分 rwnd 和 cwnd。
- 能否用 `ss -ti` 观察 cwnd、rtt、retrans。
- 能否把吞吐下降和重传、RTT、拥塞窗口联系起来。
