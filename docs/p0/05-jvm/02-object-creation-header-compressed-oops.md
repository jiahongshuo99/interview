# 对象创建过程、对象头、指针压缩

## 面试定位

这类题考察你是否理解 Java 对象从 `new` 指令到堆上实例的全过程，以及对象在内存中的布局。高质量回答要能串起类加载检查、内存分配、并发分配安全、对象初始化、对象头、锁状态、GC 年龄、指针压缩和内存对齐。

建议回答顺序：

1. 遇到 `new` 指令先做类加载检查。
2. 在堆上为对象分配内存，方式可能是指针碰撞或空闲列表。
3. 通过 CAS 或 TLAB 保证并发分配安全。
4. 将分配到的内存初始化为零值。
5. 设置对象头。
6. 执行构造方法，即字节码层面的 `<init>`。

## 对象创建过程

### 1. 类加载检查

JVM 执行 `new` 指令时，会先检查常量池中是否能定位到目标类的符号引用，并检查该类是否已经被加载、解析和初始化。如果没有，会先触发类加载流程。

注意：

- 创建对象前，类必须完成必要的加载流程。
- 类初始化不等于对象初始化。
- 父类构造方法会在子类构造方法之前执行，但这属于对象初始化阶段。

### 2. 分配内存

类加载完成后，对象所需内存大小基本可以确定。JVM 在堆中为对象分配一块内存。

常见分配方式：

- 指针碰撞：堆内存规整时，用一个指针作为已用和空闲内存分界线，分配时移动指针即可。
- 空闲列表：堆内存不规整时，JVM 维护可用内存块列表，分配时从列表中找到合适空间。

具体使用哪种方式，取决于垃圾收集器是否带有压缩整理能力。

### 3. 并发分配安全

堆是线程共享区域，多线程同时创建对象时需要保证分配安全。

常见机制：

- CAS 重试：更新分配指针时使用原子操作。
- TLAB：Thread Local Allocation Buffer，线程本地分配缓冲区。每个线程先在 Eden 中获得一小块私有区域，大多数对象在 TLAB 内快速分配，减少竞争。

相关参数：

```bash
-XX:+UseTLAB
-XX:+PrintTLAB
```

面试表达：对象仍在堆上，只是分配动作优先在 Eden 中划给线程的 TLAB 里完成，TLAB 不是独立于堆的新区域。

### 4. 零值初始化

内存分配完成后，JVM 会将对象内存空间初始化为零值，不包括对象头。这样 Java 对象字段即使没有显式赋值，也可以直接使用默认值。

例如：

```java
class User {
    int age;       // 默认 0
    boolean vip;   // 默认 false
    Object ext;    // 默认 null
}
```

### 5. 设置对象头

JVM 会设置对象头信息，例如对象所属类的元数据指针、哈希码、GC 分代年龄、锁状态等。对象头是理解锁升级、对象大小和 GC 的关键。

### 6. 执行 `<init>`

最后执行构造方法，完成业务字段赋值和构造逻辑。字节码层面构造方法对应 `<init>`，类初始化方法对应 `<clinit>`。

## 对象内存布局

HotSpot 中普通对象内存布局通常包括：

1. 对象头。
2. 实例数据。
3. 对齐填充。

数组对象还会在对象头中额外包含数组长度。

```text
普通对象:
+----------------+
| Mark Word      |
+----------------+
| Klass Pointer  |
+----------------+
| Instance Data  |
+----------------+
| Padding        |
+----------------+

数组对象:
+----------------+
| Mark Word      |
+----------------+
| Klass Pointer  |
+----------------+
| Array Length   |
+----------------+
| Array Elements |
+----------------+
| Padding        |
+----------------+
```

## 对象头

对象头主要由 Mark Word 和 Klass Pointer 组成。数组对象还会包含数组长度。

### Mark Word

Mark Word 存储对象运行时数据，具体内容会随对象状态变化复用。

常见信息：

- 对象哈希码。
- GC 分代年龄。
- 锁状态标志。
- 偏向锁线程 ID，旧版本中涉及。
- 轻量级锁指针。
- 重量级锁 Monitor 指针。

锁相关状态通常包括：

- 无锁。
- 偏向锁：JDK 15 之后偏向锁被废弃，面试中应说明版本差异。
- 轻量级锁。
- 重量级锁。
- GC 标记。

### Klass Pointer

Klass Pointer 指向对象所属类的元数据。JVM 通过它知道对象是哪个类的实例、字段布局、方法表等。

如果开启指针压缩，Klass Pointer 通常可以被压缩为 4 字节，否则在 64 位 JVM 中通常是 8 字节。

### 数组长度

数组对象需要额外存储数组长度，通常为 4 字节。普通对象没有这个字段。

## 实例数据

实例数据是真正保存业务字段的部分，包括从父类继承来的字段和子类定义的字段。HotSpot 会根据字段类型、大小、对齐要求进行布局优化，不一定完全按照源码声明顺序排列。

字段大小通常可以粗略理解为：

- `boolean`、`byte`：1 字节。
- `char`、`short`：2 字节。
- `int`、`float`：4 字节。
- `long`、`double`：8 字节。
- 引用类型：开启压缩普通对象指针时常见为 4 字节，否则为 8 字节。

## 对齐填充

HotSpot 对象大小通常按 8 字节对齐。对象头和实例数据加起来不是 8 的倍数时，会填充额外字节。

为什么需要对齐：

- 提升 CPU 访问效率。
- 简化对象地址计算。
- 满足平台内存访问要求。

可以用 JOL 观察对象布局：

```java
System.out.println(org.openjdk.jol.info.ClassLayout.parseClass(User.class).toPrintable());
```

## 指针压缩

在 64 位 JVM 中，对象引用原本需要 8 字节。为了降低内存占用、提升缓存命中率，HotSpot 支持压缩普通对象指针和压缩类指针。

常见参数：

```bash
-XX:+UseCompressedOops         # 压缩普通对象指针
-XX:+UseCompressedClassPointers
-XX:ObjectAlignmentInBytes=8
```

### Compressed Oops

Oops 是 Ordinary Object Pointers，即普通对象指针。开启压缩后，堆中对象引用可以用 32 位偏移量表示，再通过基址和对象对齐换算成真实地址。

简化理解：

```text
真实地址 = heap_base + (compressed_oop << alignment_shift)
```

如果对象按 8 字节对齐，32 位偏移量可以表示约 32GB 的寻址范围。因此在常见配置下，堆小于约 32GB 时更适合开启指针压缩。

### 为什么堆超过某个阈值后压缩效果可能下降？

如果堆太大，32 位压缩引用无法覆盖寻址范围，JVM 可能需要关闭压缩指针，导致引用字段从 4 字节变为 8 字节。这样实际可用对象容量未必随着 `-Xmx` 增长线性提升，还可能降低 CPU cache 效率。

### 指针压缩的收益

- 降低引用字段占用。
- 降低对象头中 Klass Pointer 占用。
- 提升 CPU cache 命中率。
- 降低 GC 扫描引用的内存带宽压力。

### 指针压缩的代价

- 每次访问引用时需要解码。
- 大堆场景可能无法使用或收益变弱。
- 与对象对齐参数有关，调参需要结合实际测试。

## 分配路径与逃逸分析

虽然 Java 语义上对象通过 `new` 创建，但 JIT 可能基于逃逸分析做优化。

常见优化：

- 栈上分配：对象没有逃出方法作用域时，可能不在堆上真实分配。
- 标量替换：对象字段被拆成独立变量，甚至对象本身不再创建。
- 锁消除：对象不会被多线程共享时，移除不必要的同步。

面试中要注意措辞：绝大多数对象概念上分配在堆上，但 JIT 优化后可能消除实际堆分配。

## 排查和调优方法

对象创建和布局相关问题通常体现在内存占用、GC 频率和分配速率上。

常用思路：

1. 用 GC 日志看 Young GC 是否频繁、Eden 是否快速被打满。
2. 用 `jstat -gc` 观察 Eden、Survivor、Old 使用变化。
3. 用 JFR 或 async-profiler allocation profiling 看热点分配点。
4. 用 heap dump 分析对象数量、对象大小、引用链。
5. 用 JOL 分析单对象布局，评估字段类型和引用压缩影响。
6. 对高频小对象考虑复用、批量处理、减少装箱、减少临时集合。

常用命令：

```bash
jcmd <pid> GC.class_histogram
jmap -histo:live <pid>
jcmd <pid> JFR.start name=alloc settings=profile duration=120s filename=alloc.jfr
jstat -gc <pid> 1000 10
```

## 常见追问

### 对象创建一定会执行类初始化吗？

如果目标类还没有初始化，`new` 会触发类初始化。但如果类已经初始化过，不会重复执行 `<clinit>`。对象构造方法 `<init>` 每次创建对象都会执行。

### 对象一定分配在堆上吗？

语言语义上对象是引用类型，主要在堆上分配；但 JIT 经过逃逸分析后，可能进行标量替换或栈上分配优化，使对象不发生真实堆分配。

### TLAB 是不是线程私有内存？

TLAB 对线程私有，但它是堆中 Eden 的一部分。它解决的是多线程分配对象时的竞争问题，不改变对象属于堆的事实。

### 对象头里一定有哈希码吗？

Mark Word 可以存放哈希码，但并不是对象创建后马上一定计算哈希码。通常调用 `hashCode()` 或相关逻辑后才会生成并写入。锁状态也会影响 Mark Word 的复用。

### 为什么建议堆不要盲目设置超过 32GB？

一个重要原因是可能影响压缩普通对象指针。引用变大后，对象整体占用上升，GC 扫描和 CPU cache 压力增加。实际阈值与 JVM、对象对齐和参数有关，需要以启动日志和压测为准。

## 易错点

- 把对象创建过程简化成“堆上分配加构造方法”，忽略类加载检查、零值初始化和对象头设置。
- 认为 TLAB 是堆外内存。
- 认为对象字段一定按照源码顺序排列。
- 认为对象头大小在所有 JVM、所有配置下固定。
- 忽略数组对象有数组长度。
- 认为指针压缩只影响堆大小，不影响对象大小和缓存效率。
- 认为所有 `new` 都必然产生堆对象，忽略 JIT 逃逸分析。

## 项目表达

可以这样表达一次优化：

> 我们在压测中发现 Young GC 很频繁，通过 JFR allocation profiling 看到热点集中在请求组装阶段，大量创建临时 `List` 和包装对象。优化时减少中间集合、避免不必要装箱，并把部分对象构建改成批量复用。优化后分配速率下降，Eden 压力降低，Young GC 次数和 P99 延迟都有改善。

涉及对象大小时可以补充：

> 对于高基数缓存对象，我们用 JOL 看过对象布局，发现多个包装类型和引用字段带来明显额外开销。调整字段类型、合并状态字段，并确认压缩指针开启后，单对象内存占用下降，缓存总内存更可控。
