# GC Roots、可达性分析

## 面试定位

GC Roots 和可达性分析是理解垃圾回收的基础。面试官通常会从“JVM 怎么判断对象是否可回收”开始，继续追问引用计数为什么不行、哪些对象能作为 GC Roots、四种引用区别、对象 finalize 是否会让对象复活、为什么内存泄漏本质上是对象仍然可达。

高质量回答：

1. HotSpot 主要通过可达性分析判断对象是否存活。
2. 从一组 GC Roots 出发，沿引用链遍历，能到达的对象是存活对象。
3. 不可达对象并不一定立刻死亡，还可能经历 `finalize` 等过程，但现代 Java 不建议依赖它。
4. 内存泄漏不是对象不可达，而是无用对象仍然被 GC Roots 间接引用。

## 为什么不用引用计数

引用计数的思路是：每个对象维护一个计数器，被引用时加一，引用失效时减一，计数为 0 就可以回收。

优点：

- 实现直观。
- 回收判断局部化，不一定需要全堆遍历。

缺点：

- 难以解决循环引用。
- 每次引用赋值都要维护计数，有额外开销。
- 多线程下计数更新也需要同步或原子保证。

循环引用例子：

```java
class Node {
    Node next;
}

Node a = new Node();
Node b = new Node();
a.next = b;
b.next = a;
a = null;
b = null;
```

`a` 和 `b` 彼此引用，但外部已经无法访问。引用计数可能认为它们计数不为 0；可达性分析可以从 GC Roots 出发发现它们不可达，从而回收。

## 可达性分析

可达性分析从 GC Roots 作为起点，向下搜索对象引用图。搜索经过的路径称为引用链。

```text
GC Roots
  |
  v
对象 A -> 对象 B -> 对象 C

对象 X -> 对象 Y
```

如果 `对象 X` 和 `对象 Y` 没有任何 GC Roots 引用链可达，它们就是可回收候选对象。

核心点：

- “不可达”是从 GC Roots 出发不可达，不是没有任何对象引用。
- 即使对象互相引用，只要没有根可达，也可以被回收。
- 对象可达不代表业务上还需要，只代表 GC 不能回收。

## 哪些对象可以作为 GC Roots

常见 GC Roots：

- 虚拟机栈中引用的对象，例如局部变量、方法参数、临时变量。
- 本地方法栈中 JNI 引用的对象。
- 方法区中类静态属性引用的对象。
- 方法区中常量引用的对象。
- 被同步锁持有的对象，例如正在被 `synchronized` 锁住的对象。
- JVM 内部引用，例如基本类型对应的 `Class` 对象、系统类加载器、异常对象等。
- 活跃线程对象。

面试表达时可以按“栈、静态、常量、JNI、锁、JVM 内部”来记。

## OopMap 与安全点

可达性分析需要知道哪些位置保存了对象引用。JVM 不会在 GC 时盲目扫描所有内存，而是依赖 OopMap 等结构记录对象引用位置。

OopMap：

- 记录栈和寄存器中哪些位置是对象引用。
- JIT 编译时会在特定位置生成这些元数据。
- GC 时可以快速准确找到引用。

安全点 Safepoint：

- 线程只有运行到安全点附近，JVM 才能安全地进行某些全局操作，例如 STW GC。
- 常见安全点位置包括方法调用、循环跳转、异常跳转等。

安全区域 Safe Region：

- 线程处于阻塞或睡眠状态时，无法主动跑到安全点。
- 如果线程声明自己处于安全区域，JVM 可以认为这段时间引用关系不会变化，GC 可以继续。

面试中提到这些可以体现你理解 GC 不只是“遍历引用”，还涉及线程协作和运行时元数据。

## 引用类型

Java 引用强度从强到弱：

1. 强引用。
2. 软引用。
3. 弱引用。
4. 虚引用。

### 强引用

最普通的引用，只要强引用仍然存在，对象就不会被回收。

```java
Object obj = new Object();
```

常见内存泄漏大多是无用对象仍被强引用持有。

### 软引用

软引用描述有用但非必需的对象。内存不足时，软引用对象可能被回收。

```java
SoftReference<byte[]> ref = new SoftReference<>(new byte[1024]);
```

适合缓存，但现代项目中通常更推荐使用成熟缓存框架，因为软引用回收策略不适合作为稳定缓存容量控制手段。

### 弱引用

弱引用对象只能存活到下一次 GC。只要没有强引用，GC 发生时弱引用对象就会被回收。

常见场景：

- `WeakHashMap`。
- 避免监听器、元数据缓存对 key 形成强持有。

### 虚引用

虚引用不能通过 `get()` 获取对象，主要用于跟踪对象被回收的状态，通常与 `ReferenceQueue` 配合。

常见场景：

- 直接内存释放跟踪。
- 对象回收通知。

## 引用队列

软引用、弱引用、虚引用可以和 `ReferenceQueue` 配合。对象被回收或进入相应状态后，引用对象会进入队列，业务代码可以轮询队列做清理。

示例方向：

```java
ReferenceQueue<Object> queue = new ReferenceQueue<>();
WeakReference<Object> ref = new WeakReference<>(obj, queue);
```

这类机制常用于缓存清理、资源释放辅助，但不能替代明确的资源生命周期管理。

## finalize 与对象复活

对象不可达后，如果覆盖了 `finalize()`，可能会被放入 F-Queue，由低优先级 Finalizer 线程执行 `finalize()`。对象可以在 `finalize()` 中把自己重新赋给某个 GC Roots 可达引用，实现“复活”。

但要注意：

- `finalize()` 最多只会被 JVM 自动调用一次。
- 执行时机不确定。
- 可能拖慢回收。
- 容易导致资源泄漏。
- 已经被废弃，不推荐使用。

现代 Java 更推荐：

- `try-with-resources`。
- `Cleaner`。
- 显式 `close()`。
- 框架生命周期回调。

## 内存泄漏与可达性

Java 内存泄漏的核心不是对象不可达，而是对象业务上已经没用，但仍然从 GC Roots 可达。

常见泄漏来源：

- 静态集合缓存无限增长。
- `ThreadLocal` 在线程池中未 `remove`。
- 监听器、回调、观察者未注销。
- 定时任务、线程、队列持有上下文对象。
- 自定义类加载器被静态变量或线程上下文类加载器引用。
- 大对象被日志上下文、异常栈、异步任务引用。

判断思路：

1. 对象是否持续增长？
2. 对象是否业务上已经不需要？
3. 从 heap dump 看 GC Roots 引用链。
4. 找到最近的强引用持有者。

## 排查方法

### Heap Dump 分析

常用命令：

```bash
jmap -dump:format=b,file=heap.hprof <pid>
jcmd <pid> GC.heap_dump heap.hprof
```

分析关注：

- Dominator Tree。
- Retained Size。
- GC Roots Path。
- 类实例数量。
- 大集合内部元素。
- ThreadLocalMap。
- ClassLoader 引用链。

### 在线观察

```bash
jcmd <pid> GC.class_histogram
jmap -histo:live <pid>
jstat -gcutil <pid> 1000 10
```

`jmap -histo:live` 会触发 Full GC，生产环境慎用。

### Arthas

```bash
dashboard
memory
heapdump /tmp/heap.hprof
vmtool --action getInstances --className com.demo.User --limit 10
```

注意：线上使用 Arthas 要控制命令成本，避免对大堆或高并发服务造成明显影响。

## 常见追问

### 可达对象一定不会被回收吗？

强可达对象不会被回收。软引用、弱引用、虚引用有特殊规则。对象是否可回收还要结合引用强度和 GC 阶段。

### 不可达对象一定马上被回收吗？

不一定。GC 发生时才会处理，不同收集器和阶段策略也不同；如果涉及 `finalize`，还可能经历二次标记。但业务上不能依赖回收时机。

### 静态变量一定是 GC Roots 吗？

静态变量引用的对象通常可以作为从方法区类元数据出发的根链路的一部分。但如果类和类加载器都可以卸载，相关静态引用也会消失。

### ThreadLocal 为什么容易泄漏？

`ThreadLocalMap` 的 key 是弱引用，但 value 是强引用。在线程池中线程长期存活，如果没有 `remove`，value 可能一直被线程持有，导致泄漏。

### `System.gc()` 会不会立即回收不可达对象？

不会保证立即执行，也不会保证一定回收。它只是给 JVM 一个建议，具体是否执行取决于 JVM 参数和实现。

## 易错点

- 把“没有引用”当作可回收条件，忽略循环引用和 GC Roots。
- 认为 Java 没有内存泄漏。
- 认为弱引用对象只有内存不足才回收，实际弱引用通常下一次 GC 就会被回收。
- 依赖 `finalize` 做资源释放。
- 只看 Shallow Size，不看 Retained Size 和引用链。
- 在线上随意执行 `jmap -histo:live` 或 Full GC 相关命令。

## 项目表达

可以这样表达：

> 我们排查过一个内存泄漏，现象是 Full GC 后老年代仍持续上升。导出 heap dump 后看 Dominator Tree，发现大量请求上下文对象被线程池工作线程的 `ThreadLocalMap` 持有。沿 GC Roots 路径看到 value 仍是强引用，而 key 已经是弱引用。最终在请求结束 finally 中统一 `remove`，并加了线程池场景下的上下文清理拦截器。

缓存场景：

> 某个本地缓存使用静态 Map，业务上数据已经过期，但 Map 没有淘汰策略，导致对象始终从静态变量可达。修复时改为 Caffeine，设置最大容量、过期时间和监控指标，从根因上避免无界强引用。
