# 核心链路

## 面试定位

核心链路用来证明你理解系统运行时的真实过程。系统架构讲的是全貌，核心链路讲的是一次请求、一次任务、一次发放、一次 Agent 执行到底怎么走。面试官会通过核心链路追问你的代码细节、事务边界、幂等设计、异常处理和性能瓶颈。

对 Java 后端面试来说，核心链路是最容易体现技术深度的部分。对 AI Agent 项目来说，核心链路要说明用户输入如何变成任务计划，任务计划如何调用工具，工具结果如何被验证和沉淀。

## 核心目标

- 讲清楚主流程，而不是只讲模块名。
- 讲清楚关键状态如何变化。
- 讲清楚同步、异步、事务、幂等、重试和补偿边界。
- 让面试官相信你能从入口追到落库、消息、下游和监控。
- 为技术难点和故障复盘提供具体场景。

## 准备要点

### 主链路入口

先明确链路从哪里开始。

常见入口：

- 用户点击、提交、领取、查询。
- 运营后台配置、发布、灰度、回滚。
- 定时任务扫描。
- MQ 消息触发。
- 第三方系统 RPC 调用。
- AI Agent 用户输入任务。
- CI、文档、代码仓库或测试系统事件触发。

入口要说明：

- 调用方是谁。
- 协议是什么，HTTP、RPC、MQ、定时任务还是工具调用。
- 是否需要鉴权、限流、参数校验。
- 请求是否有 traceId、幂等键和业务上下文。

### 状态流转

核心链路必须准备状态。

示例：

```text
初始化 -> 校验中 -> 处理中 -> 成功 -> 失败可重试 -> 失败终态
```

或：

```text
待发布 -> 灰度中 -> 已发布 -> 已回滚
```

AI Agent 示例：

```text
已创建 -> 规划中 -> 等待工具执行 -> 等待用户确认 -> 执行中 -> 已完成 -> 已失败 -> 可恢复
```

状态流转要说明：

- 谁推进状态。
- 状态是否允许回退。
- 哪些状态可重试。
- 哪些状态是终态。
- 状态更新是否在事务内。
- 异常状态如何扫描和补偿。

### 事务边界

准备回答：

- 哪些操作必须在一个本地事务中。
- 哪些操作不能放在事务中，例如远程 RPC、耗时下游、发送外部通知。
- MQ 消息和数据库状态如何保证最终一致。
- 失败时是回滚、重试、补偿还是进入失败状态。

常见表达：

```text
核心状态落库和幂等记录在同一个本地事务里完成，确保同一个业务请求只会生成一条主记录。下游调用不放在这个事务里，而是通过消息异步触发。消费者执行前会重新读取状态，只有状态满足条件才推进，避免重复消息造成重复处理。
```

### 幂等点

核心链路中要主动说明幂等。

常见幂等来源：

- 用户重复点击。
- 前端超时重试。
- 网关重试。
- RPC 调用方重试。
- MQ 重投递。
- 定时任务重复扫描。
- Agent 工具重复执行。

常见手段：

- 业务幂等键。
- 唯一索引。
- 状态机条件更新。
- 去重表。
- 分布式锁。
- 乐观锁版本号。
- 工具执行 actionId。

### 异常路径

面试官很少只听正常路径。至少准备三类异常：

- 参数或资格校验失败。
- 下游超时、限流、返回业务失败。
- 本地状态和外部状态不一致。
- MQ 消费失败或消息堆积。
- 缓存穿透、缓存失效、配置未命中。
- Agent 工具执行失败、权限不足、模型输出不合法、用户取消确认。

## 展开框架

### 六步讲链路

1. 请求进入：入口、身份、协议、幂等键。
2. 前置校验：参数、权限、资格、配置、风控。
3. 核心处理：规则计算、状态创建或推进、事务落库。
4. 下游交互：RPC、MQ、缓存、数据库、第三方服务。
5. 异常处理：重试、补偿、降级、失败状态。
6. 结果观测：返回结果、日志、指标、告警、数据同步。

### Java 后端链路模板

```text
用户请求进入网关后，先完成鉴权、限流和参数校验。
Controller 只做协议适配，把请求交给应用服务。
应用服务生成或读取业务幂等键，查询主记录状态。
如果是新请求，就在本地事务中创建主记录、写入幂等表并初始化状态。
核心规则由领域服务计算，必要的配置从缓存读取，缓存未命中回源 DB。
对耗时下游调用，事务提交后发送 MQ。
消费者读取消息后重新校验状态，调用下游系统，按结果推进状态。
失败进入可重试状态，由消费者重试或补偿任务扫描处理。
整条链路打 traceId，监控接口耗时、错误率、消息堆积和业务失败率。
```

### AI Agent 链路模板

```text
用户提交任务后，入口层记录任务和身份信息。
编排层先识别任务类型，加载相关上下文，例如代码、文档、历史记录和工具元数据。
模型只负责生成下一步计划或工具调用意图。
工具执行层在真正执行前做 schema 校验、权限校验和风险判断。
高风险工具需要用户确认，低风险工具可以自动执行。
执行结果写入任务状态和审计日志，再反馈给模型决定下一步。
如果工具失败，任务进入可恢复状态，用户可以重试、修改参数或终止。
最终结果经过校验和摘要，输出给用户，并沉淀评估指标。
```

## 常见追问

### 一次请求从入口到落库怎么走

回答要有顺序。

示例：

```text
请求先到网关，网关做身份和基础限流。进入服务后 Controller 做参数校验和协议转换。应用服务根据用户、场景和业务单号生成幂等键，先查是否已有处理记录。如果没有，就在事务里创建主记录和幂等记录，状态置为处理中。事务提交后发送 MQ，由消费者完成下游履约，最后根据结果把状态推进到成功或失败可重试。
```

### 事务在哪里开启

回答要避免把所有操作都塞进事务。

示例：

```text
事务只覆盖本地核心状态更新，包括主记录、幂等记录和必要的流水。远程 RPC 和 MQ 消费不放在同一个事务里。原因是远程调用不可控，放在事务里会拉长锁时间并放大故障。我们通过状态机和幂等消费保证最终一致。
```

### MQ 消息发出后消费者失败怎么办

回答要包含幂等、重试和补偿。

示例：

```text
消费者失败分两类。临时失败，例如超时和限流，会走 MQ 重试或延迟重试；业务失败则根据错误码决定是否终态失败。每次消费前都会读取主记录状态，只有处于待处理或失败可重试状态才执行，避免重复消费。对于超过重试次数的记录，会进入异常状态，由补偿任务扫描并告警。
```

### 如何防止重复发放或重复处理

回答要讲多层防线。

示例：

```text
第一层是入口业务幂等键，同一个用户、场景和业务单号只能创建一条记录。第二层是数据库唯一索引，防止并发请求同时插入。第三层是状态机条件更新，消费者只有在合法状态下才能推进。第四层是下游调用携带外部幂等号，即使出现重试，下游也能识别同一业务动作。
```

### Agent 工具调用链路如何保证可控

回答重点：

- 模型只产生意图。
- 后端执行层负责校验和审计。

示例：

```text
用户任务进入后，模型会生成工具调用意图，但不会直接执行。后端会先校验工具名、参数 schema、权限和风险等级。比如只读查询可以自动执行，写文件、提交代码、发布或删除动作需要人工确认。每次工具执行都有 actionId 和审计日志，失败后任务状态会保留现场，允许用户重试或改参数。
```

## 优秀回答结构

### 正常路径

```text
正常情况下，这条链路分成入口校验、幂等落库、业务计算、异步履约和结果回写五步。
```

### 异常路径

```text
异常主要有三类：入口校验失败直接返回；下游临时失败进入重试；状态不一致进入补偿和告警。
```

### 关键保证

```text
核心保证包括本地事务保证主记录一致、唯一索引保证幂等、状态机保证重复消息不重复推进、监控和补偿保证异常可发现可恢复。
```

### 可复用模板

```text
这条核心链路从【入口】开始。
第一步做【鉴权/参数/资格】校验。
第二步用【幂等键/业务单号】识别请求，核心状态在【事务】中落库。
第三步由【领域服务】完成【规则计算/状态推进】。
第四步对【耗时或不稳定动作】通过【MQ/任务】异步处理。
第五步消费者根据【状态机】调用下游并回写结果。
异常情况下通过【重试/补偿/告警】处理，整体通过【日志/指标/traceId】观测。
```

## 常见误区

- 只讲正常流程，不讲异常流程。
- 只讲接口调用，不讲状态变化。
- 把事务范围说得过大，包含远程调用和耗时操作。
- 说用了 MQ，但说不清重复消费和失败处理。
- 说用了缓存，但说不清缓存未命中和缓存不一致。
- 说做了幂等，但没有唯一键、状态条件或下游幂等号。
- Agent 链路只讲 prompt，不讲任务状态、工具执行和审计。
- 不知道日志和指标如何定位链路中的问题。

## 自检清单

- 能否按时间顺序讲完一次请求。
- 能否说清楚主状态有哪些，以及谁推进状态。
- 能否说明事务边界在哪里。
- 能否说明幂等键如何生成和校验。
- 能否说明 MQ 消费失败后如何处理。
- 能否说明缓存、数据库和下游系统的读写顺序。
- 能否讲清楚至少三个异常路径。
- 能否指出链路中最容易出问题的节点。
- 如果是 AI Agent 项目，是否能讲清楚计划、工具、执行、确认、审计和评估的完整路径。
