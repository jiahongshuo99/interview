# 故障与复盘

## 面试定位

故障与复盘用来证明你经历过真实线上系统，也能从问题中沉淀工程改进。面试官问故障不是为了听事故猎奇，而是判断你是否具备稳定性意识、定位能力、止损能力、复盘能力和系统性改进能力。

对 Java 后端项目，故障复盘通常涉及超时、限流、数据库慢查询、缓存问题、MQ 堆积、重复处理、配置错误和数据不一致。对 AI Agent 项目，故障复盘还可能涉及工具误调用、权限绕过、模型幻觉、上下文错误、自动化执行失败和成本失控。

## 核心目标

- 准备一个真实或可脱敏的故障案例。
- 讲清楚影响、时间线、根因、止损、修复和预防。
- 不甩锅，不只说下游问题或用户问题。
- 展示你从故障中改进了监控、流程、架构或代码。
- 把复盘和项目能力连接起来。

## 准备要点

### 复盘基本要素

| 要素 | 需要回答的问题 |
| --- | --- |
| 背景 | 故障发生在哪个业务场景 |
| 影响 | 影响了多少用户、请求、任务或数据 |
| 时间线 | 什么时候发现，什么时候止损，什么时候恢复 |
| 根因 | 直接原因和深层原因分别是什么 |
| 止损 | 如何快速恢复服务或降低影响 |
| 修复 | 代码、配置、数据或架构如何修 |
| 预防 | 后续加了什么监控、校验、灰度、预案 |
| 反思 | 当时哪些机制缺失 |

### 可讲故障类型

#### 接口超时

常见根因：

- 下游服务延迟升高。
- 线程池耗尽。
- 数据库慢查询。
- 同步链路过长。
- 连接池配置不合理。

可沉淀：

- 超时隔离。
- 异步化。
- 线程池监控。
- 慢 SQL 优化。
- 下游熔断和降级。

#### MQ 堆积

常见根因：

- 消费者处理变慢。
- 下游限流。
- 单条消息异常导致反复重试。
- 分区不均。
- 消费线程不足。

可沉淀：

- 消费失败分类。
- 死信队列。
- 限速重试。
- 积压告警。
- 批量消费和并发调整。

#### 重复处理

常见根因：

- 入口幂等缺失。
- MQ 重投递未做幂等。
- 定时补偿重复扫描。
- 下游超时结果不确定。

可沉淀：

- 业务幂等键。
- 唯一索引。
- 状态机条件更新。
- 外部幂等号。
- 对账任务。

#### 配置错误

常见根因：

- 配置缺少校验。
- 灰度范围设置错误。
- 配置发布无审批。
- 缓存未刷新。

可沉淀：

- 配置 schema 校验。
- 预发布校验。
- 灰度和回滚。
- 配置 diff。
- 操作审计。

#### Agent 自动化事故

常见根因：

- 模型生成了错误工具参数。
- 工具权限过大。
- 缺少高风险确认。
- 上下文过期或检索错误。
- 输出没有经过后端校验。

可沉淀：

- 工具白名单。
- 风险分级。
- 人工确认。
- 沙箱执行。
- actionId 审计。
- 回滚和撤销能力。

## 展开框架

### 标准复盘结构

```text
故障背景：在哪条链路发生。
影响范围：影响了什么用户、请求或数据。
发现方式：告警、用户反馈、运营反馈还是巡检发现。
时间线：发现、定位、止损、恢复、修复。
直接原因：导致故障的直接触发点。
根本原因：系统设计、监控、流程上的缺口。
止损动作：如何先恢复或降低影响。
长期改进：代码、架构、监控、流程如何改。
复盘结论：这次故障带来的工程经验。
```

### 示例一：下游超时导致入口接口抖动

```text
故障发生在核心履约链路。某次活动高峰期间，下游资产系统延迟升高，导致我们入口接口的 P99 和超时率明显上升。最初是监控告警发现接口超时，随后用户侧也出现处理中时间变长。

直接原因是入口链路同步等待下游返回；根本原因是我们把不稳定下游放在了用户强同步路径里，缺少隔离。止损上，我们先临时降低入口并发并开启降级，把部分请求转成处理中状态。长期修复是把履约调用改为 MQ 异步，入口只完成校验和状态落库，消费者限速调用下游，并补充 MQ 堆积、下游耗时和失败状态告警。
```

### 示例二：重复消息导致重复处理风险

```text
一次故障中，部分 MQ 消息因为消费者超时被重复投递。虽然业务结果没有大面积异常，但暴露出消费者幂等不完整的问题。直接原因是消费者处理成功后确认消息失败，消息被重新投递；根本原因是消费逻辑依赖消息只处理一次，没有在业务状态层做条件判断。

止损上，我们先暂停相关消费者，确认受影响业务单并做对账。长期修复是消费前读取主记录状态，只允许待处理状态推进；下游调用携带外部幂等号；数据库增加唯一约束；并补充重复消费测试用例和异常指标。
```

### 示例三：Agent 工具误调用

```text
Agent 场景中，一次问题是模型根据过期上下文生成了错误工具参数，导致工具执行失败并反复重试。影响范围主要是部分任务失败，没有产生不可逆写操作。

直接原因是工具参数缺少强校验，模型输出不合法时系统仍尝试执行；根本原因是我们过度依赖 prompt 约束，没有把校验放在后端工具层。止损上先关闭该工具的自动执行，改为人工确认。长期修复是定义严格 schema、参数白名单、失败重试上限和风险分级，高风险工具必须用户确认。
```

## 常见追问

### 故障是怎么发现的

回答重点：

- 优先说监控发现。
- 如果是用户反馈，要说明后续补了监控。

示例：

```text
这次最早是接口超时率告警发现的，随后我们通过 traceId 看到耗时集中在下游调用。用户反馈比告警晚一些。复盘后我们补了更细粒度的业务状态告警，因为只看接口错误率还不够，异步链路里的失败和积压也需要提前发现。
```

### 当时怎么止损

回答重点：

- 先恢复服务，再完整修复。

示例：

```text
止损分两步。第一步先降低影响面，通过限流、降级和暂停部分非核心消费者减轻下游压力。第二步对已经进入异常状态的记录做对账和补偿。代码修复没有直接在线上热改，而是先用配置和流量控制稳定住，再走正常发布修复。
```

### 根因是什么

回答重点：

- 区分直接原因和根本原因。

示例：

```text
直接原因是下游超时，但根本原因是我们把下游调用放在了用户同步链路里，缺少故障隔离和超时降级。只说下游慢是不完整的，因为系统设计应该预期下游会波动。
```

### 复盘后做了什么改进

回答重点：

- 改进要具体。

示例：

```text
复盘后主要做了四件事：第一，把非强实时下游调用异步化；第二，增加消费者失败分类和补偿任务；第三，补充接口、MQ、下游和业务状态四类监控；第四，在上线 checklist 里增加压测、灰度和回滚预案检查。
```

## 优秀回答结构

1. 不回避：承认故障和影响。
2. 有边界：说明影响范围，不夸大也不缩小。
3. 有时间线：发现、定位、止损、恢复、修复。
4. 有根因：直接原因和系统性原因。
5. 有动作：短期止损和长期修复。
6. 有沉淀：监控、流程、架构、测试和预案。
7. 有反思：如果重来会怎么做。

可复用模板：

```text
我经历过一次【故障类型】。
它发生在【业务链路】，影响是【影响范围】。
我们通过【发现方式】发现，定位时通过【日志/指标/traceId】确认问题集中在【节点】。
直接原因是【直接原因】，根本原因是【设计/流程/监控缺口】。
当时先通过【止损动作】恢复，之后通过【长期修复】解决。
复盘后沉淀了【监控/测试/灰度/预案】，这也是我后续做类似项目时会优先考虑的点。
```

## 常见误区

- 不愿承认故障，只说系统一直很稳定。
- 把故障完全甩给下游、产品或用户。
- 只讲修了 bug，不讲为什么监控没提前发现。
- 只讲恢复了，不讲长期改进。
- 没有影响范围和时间线。
- 根因分析停在直接原因，没有系统性原因。
- 复盘动作很虚，例如「加强监控」「加强测试」，但没有具体指标和规则。
- AI Agent 故障只说模型不稳定，没有讲工具层、权限层和审计层改造。

## 自检清单

- 是否准备了一个可脱敏的故障案例。
- 是否能说明影响范围和时间线。
- 是否能区分直接原因和根本原因。
- 是否能说明当时如何止损。
- 是否能说明后续代码、架构、监控或流程改进。
- 是否能讲清楚这次故障带来的个人成长。
- 是否避免甩锅。
- Java 后端主线是否体现了稳定性治理。
- 如果是 AI Agent 项目，是否包含工具权限、执行审计和风险分级。
