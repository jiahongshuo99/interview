# 可扩展性

## 面试定位

可扩展性用来证明你的方案不是只为当前需求临时拼出来，而是能适应业务增长、场景变化和团队协作。面试官关注的是你能否识别未来变化点，并用合理的抽象、边界和演进路径支撑变化。

对 Java 后端项目，可扩展性通常包括流量扩展、数据扩展、业务规则扩展、接入方扩展、服务拆分、配置化、多租户和运维治理。对 AI Agent 项目，可扩展性还包括工具扩展、任务模板扩展、模型扩展、上下文扩展、权限扩展和评估体系扩展。

## 核心目标

- 说明系统当前支持哪些变化。
- 说明未来规模扩大或业务变化时，瓶颈在哪里。
- 说明扩展点是有意识设计的，不是过度设计。
- 展示你能区分当前阶段和未来演进。
- 把可扩展性和 Java 后端工程能力连接起来。

## 准备要点

### 扩展维度

| 维度 | 常见问题 | 准备方向 |
| --- | --- | --- |
| 流量扩展 | QPS 增长、峰值更高 | 水平扩容、缓存、限流、异步削峰 |
| 数据扩展 | 表数据量增长 | 分库分表、归档、索引、冷热分离 |
| 业务扩展 | 新策略、新活动、新规则 | 配置化、策略模式、规则引擎 |
| 接入扩展 | 更多调用方接入 | API 契约、鉴权、限流、版本兼容 |
| 组织扩展 | 多团队协作 | 模块边界、领域服务、文档和规范 |
| 稳定性扩展 | 故障面扩大 | 隔离、熔断、降级、灰度、回滚 |
| Agent 扩展 | 更多工具和任务 | 工具注册、权限模型、任务编排、评估 |

### 变化点识别

可扩展性不是把所有东西都抽象。先识别真正可能变化的点。

常见变化点：

- 活动类型会增加。
- 权益类型会增加。
- 规则维度会增加。
- 下游系统会增加。
- 接入方会增加。
- 数据量会持续增长。
- 流量峰值会周期性变化。
- Agent 工具和任务类型会增加。
- 模型供应商或模型版本会变化。

不常变化的点不要过度抽象，否则会增加复杂度。

### 当前方案和未来方案

面试中要体现阶段感：

- 当前阶段：用简单、稳定、可控的方案解决主要问题。
- 下一阶段：当规模或复杂度达到阈值时再演进。
- 演进条件：明确什么指标触发升级。

示例：

```text
当前规则数量有限，用策略模式加配置模型足够；如果后续规则数量增长到运营需要频繁组合，并且研发变更成为瓶颈，可以演进到规则引擎或 DSL。
```

## 展开框架

### 可扩展性表达结构

```text
当前系统主要面临【变化点】。
我们在设计上通过【扩展点】支持这些变化。
对于当前规模，这个方案的复杂度可控。
如果未来【规模或复杂度】增长，瓶颈会出现在【瓶颈】。
对应的演进方向是【演进方案】。
```

### Java 后端可扩展性设计

#### 流量扩展

可讲点：

- 服务无状态化，支持水平扩容。
- 热点配置和读请求走缓存。
- 写链路通过 MQ 削峰。
- 下游调用限速和隔离。
- 高峰前压测和容量评估。

示例：

```text
入口服务尽量保持无状态，用户会话和业务状态都落在外部存储中，因此可以通过增加实例水平扩容。读多的配置和基础数据走缓存，写链路中非强实时部分通过 MQ 削峰。真正的瓶颈主要在数据库写入和下游履约能力，所以后续扩容要重点关注分库分表、消费者并发和下游限流。
```

#### 数据扩展

可讲点：

- 主表和流水表分离。
- 热数据和历史数据分离。
- 索引按查询场景设计。
- 大表归档。
- 分库分表预留分片键。

示例：

```text
数据上我们把当前状态和操作流水分开。主表保存当前业务状态，流水表保存每次状态变更和下游调用结果。查询高频的是主表，流水主要用于审计和排查。随着数据增长，可以按业务时间做归档，或者按用户、业务单号等维度分片。
```

#### 业务规则扩展

可讲点：

- 策略接口。
- 配置模型。
- 规则版本。
- 灰度发布。
- 配置校验。

示例：

```text
业务规则变化比较频繁，所以我们没有把所有规则写死在主流程里，而是把人群、时间、权益、渠道等维度抽成配置和策略接口。主链路只负责加载规则和执行策略，新增规则时尽量新增策略实现和配置，不改主流程。
```

#### 接入方扩展

可讲点：

- API 契约稳定。
- 版本号。
- 调用方身份。
- 限流配额。
- 错误码规范。
- 接入文档。

示例：

```text
接入方扩展主要靠稳定接口契约。我们会区分内部字段和对外字段，接口增加字段保持向后兼容，关键行为通过版本或场景码区分。每个接入方有身份和限流配置，避免新接入方影响老链路。
```

### AI Agent 可扩展性设计

#### 工具扩展

可讲点：

- 工具注册中心。
- 统一 schema。
- 权限声明。
- 风险等级。
- 执行适配器。
- 结果标准化。

示例：

```text
Agent 工具扩展不能靠在 prompt 里手写工具说明。更好的方式是工具注册机制，每个工具声明名称、用途、参数 schema、权限要求、风险等级和结果格式。编排层只依赖统一协议，新增工具时不需要改主流程。
```

#### 任务类型扩展

可讲点：

- 任务模板。
- 状态机复用。
- 步骤编排。
- 中断和恢复。
- 任务评估。

示例：

```text
不同 Agent 任务虽然业务不同，但都有创建、规划、执行、观察、确认、完成或失败这些状态。我们把任务状态机和工具执行能力抽成通用层，具体任务通过模板定义步骤和约束。这样从代码检索扩展到文档生成、测试触发或发布检查时，不需要重写底层任务系统。
```

#### 模型扩展

可讲点：

- 模型适配层。
- 能力路由。
- 成本控制。
- 降级模型。
- 输出格式校验。

示例：

```text
模型层通过适配器隔离供应商差异，上层编排只关心统一的推理接口。不同任务可以按成本和能力路由到不同模型，例如简单分类用低成本模型，复杂代码分析用能力更强的模型。无论模型如何切换，工具调用输出都必须经过统一 schema 校验。
```

## 常见追问

### 如果流量增长十倍怎么办

回答重点：

- 不要只说加机器。
- 讲清楚瓶颈和演进顺序。

示例：

```text
十倍流量下，入口服务可以先水平扩容，因为它是无状态的。接下来要看缓存命中率、数据库写入和 MQ 消费能力。如果读压力上升，优先优化缓存和热点隔离；如果写压力上升，要评估分库分表和批处理；如果 MQ 堆积，要扩消费者、优化下游调用并做限速。单纯加入口机器不能解决数据库和下游瓶颈。
```

### 如果新增一种业务规则怎么办

回答重点：

- 说明扩展点在哪里。

示例：

```text
如果新增规则维度，理想情况下是在规则配置和策略实现里扩展，不改主链路。主链路只负责加载规则上下文、执行策略接口和处理结果。新增规则还要配套配置校验、灰度和回滚，避免配置错误影响线上。
```

### 为什么没有一开始就做成规则引擎

回答重点：

- 说明避免过度设计。

示例：

```text
一开始规则数量有限，主要变化点也比较清楚，用策略模式和配置模型足够。如果直接引入规则引擎，会增加学习成本、调试成本和运行时风险。我们保留了策略接口和规则版本字段，后续规则复杂到需要运营自由组合时，再演进到规则引擎更合适。
```

### Agent 新增工具会不会影响老任务

回答重点：

- 工具协议、权限和评估隔离。

示例：

```text
新增工具不会直接影响老任务，因为工具通过注册机制接入，每个工具有独立 schema、权限和风险等级。老任务使用的工具集合是白名单，不会自动暴露新工具。新工具上线前要经过离线评估和灰度，确认工具调用成功率和失败边界。
```

## 优秀回答结构

1. 先说明当前系统为哪些变化做了准备。
2. 再说明哪些变化暂时没有过度设计。
3. 接着讲当前扩展点，例如策略、配置、接口、工具注册、任务模板。
4. 然后讲未来瓶颈，例如数据库、缓存、MQ、下游、模型成本。
5. 最后讲演进条件和演进路径。

可复用模板：

```text
这个系统的可扩展性主要考虑【流量/数据/规则/接入方】几个维度。
当前我们通过【无状态服务/缓存/MQ/策略接口/配置模型】支持主要变化。
没有一开始做得过重，是因为当前阶段【规模和复杂度】还没到那个程度。
如果未来【指标或业务复杂度】增长，瓶颈会在【瓶颈】。
到时候可以按【演进路径】升级，例如【分库分表/规则引擎/工具注册/模型路由】。
```

## 常见误区

- 把可扩展性等同于微服务拆分。
- 一上来就说分库分表，但当前数据量根本不需要。
- 所有规则都配置化，导致配置复杂且难以校验。
- 只讲流量扩展，不讲业务规则和接入方扩展。
- 只讲未来方案，不讲当前为什么这样做。
- Agent 项目只说可以接更多模型，不讲工具、权限、任务和评估扩展。
- 不知道系统真正瓶颈在哪里。

## 自检清单

- 是否识别了项目中最可能变化的点。
- 是否能说明当前方案支持哪些扩展。
- 是否能说明哪些地方没有过度设计，以及为什么。
- 是否能说清楚流量增长时的瓶颈顺序。
- 是否能说清楚数据增长时的归档、索引或分片策略。
- 是否能说清楚业务规则新增时如何扩展。
- 是否能说清楚接入方新增时如何保证兼容和隔离。
- 如果是 AI Agent 项目，是否包含工具注册、任务模板、模型适配、权限和评估扩展。
