# 方案权衡

## 面试定位

方案权衡用来证明你不是只会套技术方案，而是能在业务目标、系统约束、交付成本和长期维护之间做判断。面试官常用方案权衡来判断候选人的工程成熟度，尤其是中高级 Java 后端面试。

好的权衡表达不是「我们用了某个方案」，而是「我们评估了哪些方案，各自优缺点是什么，为什么在当时选择这个方案，代价是什么，未来什么条件下会调整」。

## 核心目标

- 说明技术选择背后的约束和判断。
- 展示你能看到方案收益，也能看到方案代价。
- 避免把所有选择都包装成唯一正确答案。
- 把 Java 后端常见技术选择讲成工程决策。
- 把 AI Agent 项目的模型选择、工具边界和自动化程度讲清楚。

## 准备要点

### 权衡维度

常见维度：

- 一致性：强一致、最终一致、补偿一致。
- 性能：平均延迟、P99、吞吐、峰值承载。
- 可用性：下游故障时主链路是否可用。
- 复杂度：代码复杂度、运维复杂度、排查复杂度。
- 成本：机器成本、存储成本、开发成本、模型 token 成本。
- 可维护性：模块边界、配置化程度、测试难度。
- 可扩展性：新增场景、新增接入方、新增策略是否容易。
- 交付风险：上线周期、迁移风险、回滚难度。
- 安全性：权限、审计、数据安全、操作风险。

### 权衡表达原则

- 先讲约束，再讲选择。
- 先承认备选方案合理，再说明为什么不选。
- 方案要和业务阶段匹配，不要脱离规模追求复杂架构。
- 主动讲代价和补偿措施。
- 面试中不要说「没有缺点」。

### 常见 Java 后端权衡题

| 权衡 | 适合讨论的问题 |
| --- | --- |
| 同步 vs 异步 | 用户是否必须立即得到最终结果，下游是否稳定 |
| 强一致 vs 最终一致 | 是否涉及资金、库存、权益、状态展示 |
| 缓存 vs 直接查库 | 热点读、数据新鲜度、缓存一致性 |
| 分布式锁 vs 数据库唯一约束 | 幂等、并发、可靠性、性能 |
| 单体内聚 vs 微服务拆分 | 团队规模、发布频率、领域边界 |
| 配置化 vs 硬编码 | 策略变化频率、校验复杂度、运营能力 |
| 规则引擎 vs 代码策略 | 规则复杂度、可读性、调试难度 |
| 定时补偿 vs 实时补偿 | 延迟要求、系统压力、实现复杂度 |
| DB 查询 vs ES 搜索 | 数据量、查询维度、实时性、成本 |
| 本地缓存 vs Redis | 访问延迟、一致性、容量、更新频率 |

### 常见 AI Agent 权衡题

| 权衡 | 适合讨论的问题 |
| --- | --- |
| 自动执行 vs 人工确认 | 效率和安全风险 |
| Prompt 规则 vs 后端校验 | 模型灵活性和可靠性 |
| 长上下文 vs 检索增强 | 完整性、成本和噪音 |
| 通用 Agent vs 场景 Agent | 泛化能力和成功率 |
| 模型推理 vs 规则流程 | 灵活性、可解释性、稳定性 |
| 一次性回答 vs 多步任务状态 | 简单性和可恢复性 |
| 在线调用大模型 vs 离线预处理 | 实时性、成本、稳定性 |
| 工具开放 vs 工具白名单 | 能力边界和安全控制 |

## 展开框架

### 标准权衡结构

```text
当时有两个主要方案。
方案 A 的优点是【优点】，缺点是【缺点】。
方案 B 的优点是【优点】，缺点是【缺点】。
结合当时的业务约束【约束】，我们选择了【方案】。
这个方案的代价是【代价】，所以我们补了【治理手段】。
如果未来【条件变化】，可以演进到【未来方案】。
```

### 示例一：同步和异步

```text
核心链路里我们评估过全同步和异步化两种方案。全同步的优点是用户可以立即拿到最终结果，状态简单；缺点是下游延迟会直接影响入口接口，活动高峰时容易放大故障。异步化的优点是能削峰、隔离下游波动，入口稳定性更好；缺点是状态会变成最终一致，需要用户看到处理中状态，也需要补偿和监控。

结合业务上允许短时间处理中、但不能接受入口大量超时的约束，我们选择入口同步受理加 MQ 异步履约。为了弥补最终一致带来的复杂度，我们补了状态机、幂等消费、失败重试和补偿任务。
```

### 示例二：缓存和直接查库

```text
配置读取一开始可以直接查 DB，优点是数据一致性简单，开发成本低；缺点是高频请求下 DB 压力较大，且配置变化频率远低于读取频率。使用缓存可以降低数据库压力和接口延迟，但会引入缓存一致性和刷新失败问题。

最后我们选择缓存配置数据，因为它读多写少，且短时间延迟生效可以接受。数据库仍然作为事实源，缓存设置 TTL，并在配置发布时主动刷新。缓存异常时可以回源 DB，保证不会因为缓存问题影响核心流程。
```

### 示例三：分布式锁和唯一索引

```text
对于幂等问题，分布式锁可以在入口减少并发冲突，但锁超时、锁释放失败和网络异常都会带来边界问题。数据库唯一索引的性能成本更高一些，但它是最终写入层面的强约束，更适合作为兜底。

因此我们没有只依赖 Redis 锁，而是用业务幂等键加数据库唯一索引保证最终唯一。对于高并发热点场景，可以在入口增加短期锁或本地限流降低冲突，但正确性仍然以数据库约束为准。
```

### 示例四：Agent 自动执行和人工确认

```text
Agent 工具调用也有自动执行和人工确认的权衡。自动执行效率最高，但模型可能选错工具或传错参数，写文件、提交代码、发布等动作风险较高。全部人工确认又会降低自动化价值。

所以我们按风险分级处理。只读查询、代码搜索、日志检索可以自动执行；写操作、删除操作、发布操作必须人工确认；所有工具调用都经过 schema 校验和权限校验，并记录审计日志。这样在效率和安全之间取得平衡。
```

## 常见追问

### 为什么不用更简单的方案

回答重点：

- 承认简单方案有价值。
- 说明它在哪个约束下不够。

示例：

```text
如果只是低频内部操作，直接同步调用确实更简单。但这个链路面向高峰用户请求，下游延迟波动会直接影响入口可用性。我们选择异步不是为了复杂化，而是因为稳定性约束比状态简单性更重要。
```

### 为什么不用更先进的方案

回答重点：

- 先进不等于适合。
- 讲业务阶段和团队成本。

示例：

```text
当时也可以引入完整规则引擎，但项目第一阶段规则数量有限，而且变化主要集中在几个维度。如果一开始引入复杂规则引擎，会增加配置校验、调试和运维成本。所以我们先用策略模式加配置模型解决主要问题，同时保留后续规则引擎化的扩展点。
```

### 你的方案有什么缺点

回答重点：

- 主动讲真实缺点。
- 说明治理手段。

示例：

```text
缺点是链路状态变多，排查复杂度提升。为了解决这个问题，我们给每个业务请求生成 traceId，状态表里保留关键阶段和错误码，消息消费和下游调用都打点，同时补了异常状态查询和补偿任务。
```

### 如果业务规模扩大十倍怎么办

回答重点：

- 当前方案是否还能扛。
- 哪些部分需要演进。

示例：

```text
如果扩大十倍，入口服务可以水平扩容，缓存和 MQ 也能承接一部分压力。首先要关注的是数据库写入和热点幂等键冲突，可能需要分库分表或按业务维度拆分写入。其次是消费者吞吐和下游限流，需要做分组消费、限速和批处理。当前方案的状态机和异步边界可以复用，但存储和消费能力要扩展。
```

## 优秀回答结构

1. 问题背景：当时为什么需要选择。
2. 方案列表：至少两个可行方案。
3. 维度比较：性能、一致性、复杂度、成本、风险。
4. 选择理由：结合业务阶段和技术约束。
5. 代价承认：这个方案牺牲了什么。
6. 补偿措施：如何治理这个代价。
7. 演进条件：未来什么情况下会换方案。

可复用模板：

```text
这个点当时主要在【方案 A】和【方案 B】之间权衡。
方案 A 更适合【场景】，优势是【优势】，问题是【问题】。
方案 B 更适合【场景】，优势是【优势】，问题是【问题】。
结合我们当时【业务规模/一致性要求/交付周期/团队成本】，最终选择了【方案】。
它的代价是【代价】，所以我们补了【监控/补偿/灰度/兜底】。
如果未来【条件变化】，我会考虑演进到【新方案】。
```

## 常见误区

- 只说自己选的方案好，不说其他方案。
- 把权衡讲成技术名词堆砌。
- 说「为了高并发所以用了 MQ」，但不讲业务是否允许异步。
- 说「为了性能用了缓存」，但不讲一致性和失效策略。
- 说「为了安全加人工确认」，但不讲自动化价值如何保留。
- 不承认方案代价。
- 未来演进只说「可以优化」，没有具体触发条件。

## 自检清单

- 是否为每个核心技术选择准备了至少两个备选方案。
- 是否能从性能、一致性、复杂度、成本和风险维度比较。
- 是否能解释为什么当前阶段选择这个方案。
- 是否能主动说明方案代价。
- 是否有补偿措施，例如监控、补偿、灰度、回滚。
- 是否能说明未来演进条件。
- Java 后端主线是否明确。
- AI Agent 项目是否讲清楚自动化效率和执行安全之间的权衡。
