# 个人角色

## 面试定位

个人角色用来回答「这件事到底是不是你做的」。面试官会通过这一部分判断你的真实贡献、责任边界、技术深度和协作能力。好的个人角色表达不是简单说「我负责后端开发」，而是讲清楚你在项目中承担了哪些决策、设计、编码、联调、上线、治理和复盘工作。

对 Java 后端候选人来说，个人角色必须能落到具体模块和工程动作，例如接口设计、领域建模、数据库设计、缓存策略、消息链路、幂等控制、性能优化、监控告警、灰度上线。对 AI Agent 项目，则要讲清楚你负责的是 Agent 编排、工具接入、上下文管理、权限控制、评估体系还是平台工程。

## 核心目标

- 证明你不是旁观者，而是项目中有明确产出的 owner。
- 准确区分「我主导」「我负责」「我参与」「我配合」。
- 让面试官能沿着你的职责继续追问技术细节。
- 避免过度包装，避免把团队成果全部说成个人成果。
- 展示工程完整度，包括设计、实现、上线、运维和复盘。

## 准备要点

### 角色级别

建议提前把自己的角色分成四档。

| 表达 | 含义 | 可承受追问强度 |
| --- | --- | --- |
| 我主导 | 你负责方案方向、核心决策和推进结果 | 必须能讲清楚权衡、风险和复盘 |
| 我负责 | 你是某个模块或链路 owner | 必须能讲清楚实现细节和边界 |
| 我参与 | 你贡献了部分实现、联调或优化 | 要说明具体贡献，不要泛化 |
| 我配合 | 你协助测试、上线、排查或数据验证 | 不适合包装成核心项目 |

面试中可以诚实使用这些词。真实的表达比夸大更容易经得住追问。

### 责任范围

至少准备以下维度：

- 业务范围：负责哪类业务能力，例如活动配置、权益发放、任务状态、Agent 工具编排。
- 系统范围：负责哪个服务、模块、接口、任务、消息消费者或管理后台。
- 技术范围：负责哪些技术点，例如缓存、MQ、事务、幂等、监控、灰度。
- 协作范围：和哪些角色协作，例如产品、前端、测试、算法、运维、下游系统。
- 生命周期：参与了需求评审、技术方案、开发、联调、上线、监控、复盘中的哪些环节。

### 可验证产出

个人角色最好能绑定产出物。

常见产出：

- 技术方案文档。
- 接口定义和 API 契约。
- 数据库表结构和索引设计。
- 核心代码模块。
- 消息 topic、消费者和补偿任务。
- 监控看板和告警规则。
- 压测报告、性能优化记录。
- 上线方案、灰度计划和回滚预案。
- AI Agent 的工具 schema、执行日志、评估集、任务模板。

### Java 后端能力映射

个人角色要尽量挂到 Java 后端主线。

可表达的能力点：

- 领域建模：把业务概念抽象成实体、状态、策略和服务。
- 接口设计：定义请求参数、响应结构、错误码、兼容策略。
- 事务和一致性：处理本地事务、分布式事务、最终一致性和补偿。
- 幂等和并发：处理重复请求、重复消息、库存扣减、重复发放。
- 性能优化：缓存、批处理、异步化、索引、线程池、连接池。
- 可观测性：日志、指标、traceId、告警、排查工具。
- 稳定性：限流、降级、熔断、隔离、灰度、回滚。
- 工程质量：代码结构、测试覆盖、配置治理、发布流程。

### AI Agent 能力映射

AI Agent 项目也要尽量回到后端工程能力，而不是只讲模型。

可表达的能力点：

- Agent 编排：任务拆解、步骤控制、状态流转、失败重试。
- 工具接入：工具 schema、参数校验、权限控制、执行沙箱。
- 上下文管理：检索、摘要、记忆、窗口控制、历史压缩。
- 安全边界：敏感操作确认、审计日志、输出校验。
- 评估体系：任务成功率、人工采纳率、工具调用成功率、幻觉率。
- 平台工程：多租户、任务队列、并发调度、监控告警、成本控制。

## 展开框架

### ODEI 框架

可以用 ODEI 讲个人角色：

- Owner：我负责哪一块。
- Decision：我做了哪些关键决策。
- Execution：我落地了哪些实现。
- Impact：这些工作带来了什么结果。

示例：

```text
我在项目中主要负责核心履约链路的后端设计和实现。具体包括接口契约设计、订单状态机建模、幂等表设计、MQ 消费和失败补偿任务。我做的关键决策是把非强实时的通知和统计链路异步化，把核心写链路收敛在一个事务边界内，并通过幂等键防止重复发放。上线后，核心接口 P99 和重复处理问题都有明显改善。
```

### RACI 框架

如果项目多人协作，建议用 RACI 梳理。

| 维度 | 说明 | 面试表达 |
| --- | --- | --- |
| Responsible | 实际执行者 | 我负责实现核心接口和消息消费者 |
| Accountable | 最终 owner | 我对履约链路稳定性和上线结果负责 |
| Consulted | 被咨询者 | 下游资产系统参与了接口契约确认 |
| Informed | 被同步者 | 运营、客服和测试关注上线节奏和数据结果 |

面试里不需要直接说 RACI，但你要能清楚表达谁负责什么。

### 一句话角色表达

```text
我在这个项目中是【模块/链路】的后端 owner，主要负责【设计/实现/联调/上线】。关键工作包括【工作一】、【工作二】、【工作三】，并且对【指标/稳定性/交付结果】负责。
```

## 常见追问

### 你具体做了什么

回答不要停在「写接口」。

较弱回答：

```text
我主要负责后端开发，写了一些接口，也做了联调。
```

较好回答：

```text
我具体负责三块。第一是核心接口契约，包括参数校验、错误码和版本兼容；第二是履约状态机和幂等逻辑，保证重复请求和重复消息不会重复处理；第三是上线后的监控和补偿任务，包括失败率、消息堆积和异常状态扫描。
```

### 哪些方案是你决定的

回答重点：

- 讲一个或两个关键决策。
- 说明备选方案和选择原因。
- 不要说所有决策都是自己拍板。

示例：

```text
我主导的一个关键决策是把核心发放链路拆成同步校验和异步履约两段。同步部分只做参数、资格和幂等校验，异步部分通过 MQ 完成下游调用和补偿。这样做的原因是下游资产系统在高峰期延迟波动较大，如果全链路同步会影响入口可用性。代价是状态会变成最终一致，所以我们补了状态机、重试任务和用户可查询的处理中状态。
```

### 你和其他人怎么分工

回答重点：

- 主动说明边界，体现协作成熟度。
- 不要贬低其他人。
- 不要把团队工作全揽到自己身上。

示例：

```text
这个项目里产品负责业务规则和运营流程定义，前端负责管理台和用户端入口，测试负责主链路和异常场景覆盖。我这边主要负责后端领域模型、接口、数据库、MQ 和监控。和下游资产系统的接口契约是我和对方后端一起确认的，上线节奏由我和测试、运维一起推进。
```

### 你最有技术含量的贡献是什么

回答重点：

- 选最能体现 Java 后端能力的点。
- 讲清楚问题、方案和效果。

示例：

```text
我认为最有技术含量的是幂等和状态补偿设计。这个链路会受到用户重复点击、前端重试、网关超时重试和 MQ 重投递影响，如果只靠接口层去重不够。所以我用业务幂等键加唯一索引保证同一业务动作只落一条主记录，再用状态机区分初始化、处理中、成功和失败可重试状态。异步消费者根据状态推进，下游失败时不直接回滚主记录，而是进入可补偿状态，由定时任务按重试策略处理。
```

### AI Agent 项目里你是不是只调了模型

回答重点：

- 强调后端工程部分。
- 说明模型只是系统的一环。

示例：

```text
不是。模型调用只是 Agent 链路中的一个节点。我负责的重点是任务编排和工具执行层，包括把用户意图拆成可执行步骤、定义工具 schema、做参数校验和权限控制、记录每一步执行日志，以及在失败时给出可恢复状态。这样 Agent 才不是一次性问答，而是一个可观测、可审计、可重试的后端任务系统。
```

## 优秀回答结构

### 标准结构

1. 先定位角色：我是哪个模块或链路的 owner。
2. 再列关键工作：设计、实现、联调、上线、监控。
3. 然后讲关键决策：为什么这么做，有什么权衡。
4. 接着给证据：代码模块、接口、数据表、指标、问题复盘。
5. 最后承接追问：可以继续展开架构、核心链路或技术难点。

### 可复用模板

```text
我在项目里主要负责【模块/链路】，角色是【主导/负责/参与】。
具体工作有三块：
第一，【需求/设计类工作】；
第二，【核心实现类工作】；
第三，【上线/监控/治理类工作】。
其中我主导的关键决策是【决策】，原因是【约束和权衡】。
最终这部分支撑了【结果】，也可以从【技术点】继续展开。
```

## 常见误区

- 只说「我们做了」，不说「我做了」。
- 只说「我负责开发」，不说负责哪个模块、哪条链路。
- 夸大个人职责，结果被追问细节时答不上来。
- 把业务沟通、代码实现、上线运维割裂开讲，看不出完整 owner 意识。
- 说了很多技术名词，但没有具体产出物。
- 只讲顺利过程，不讲遇到的问题和自己的处理。
- AI Agent 项目只讲 prompt，不讲工具、任务、权限、评估和稳定性。

## 自检清单

- 能否一句话说清楚自己在项目中的角色。
- 能否列出至少三个具体产出物。
- 能否区分自己主导、负责、参与和配合的内容。
- 能否讲清楚一个自己做过的关键技术决策。
- 能否说明和产品、前端、测试、下游系统的分工。
- 能否在追问下讲出代码层面的实现细节。
- 能否把个人贡献和项目指标关联起来。
- 是否避免把团队成果全部说成个人成果。
- 如果是 AI Agent 项目，是否能说明自己做的是系统工程而不只是模型调用。
